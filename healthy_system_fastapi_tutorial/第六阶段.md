# Healthy System FastAPI 第六阶段详细教程

## 1. 这一阶段到底要解决什么

第六阶段要开始为“主动服务”打地基。

很多项目一开始就喜欢说:

1. 主动关怀
2. 智能提醒
3. 智能介入

但如果没有:

1. 触发规则
2. 主动行为日志

这些功能其实都是空的。

所以第六阶段先不急着做“真的主动发消息”，而是先把两个底层模块做好:

1. `TriggerRule`
2. `ActiveLog`

一句话理解这一阶段:

先把“为什么要触发”和“触发后记录了什么”这两件事建清楚。

---

## 2. 第六阶段完成后的验收标准

做到下面这些，就算第六阶段合格:

1. 规则可配置
2. 可手动触发规则检查
3. 日志可记录一次触发过程
4. `trigger_rule` 能存配置条件
5. `active_log` 能记录一次检查结果
6. 用户只能查看自己的主动日志

---

## 3. 为什么先做规则和日志，而不是直接做主动服务

因为如果你只是做“主动发送”，但没有触发规则和日志，后面你会立刻遇到这些问题:

1. 为什么触发了
2. 为什么没触发
3. 是不是重复触发了
4. 哪条规则最有效
5. 这次触发到底对谁触发的
6. 触发时后端到底做了什么

如果没有日志，你根本查不清楚。

所以第六阶段最合理的顺序是:

1. 先定义规则
2. 先能手动检查规则
3. 每次检查都留下日志
4. 后面再做真正主动发送动作

这才是一个可调试、可追踪、可迭代的工程顺序。

---

## 4. 第六阶段默认建立在前面几个阶段已经完成的前提上

第六阶段不是独立存在的。

它默认你前面至少已经有:

1. 第二阶段的 `User / Auth / Profile`
2. 第三阶段的 `Record`
3. 第四阶段的 `Conversation / Message`
4. 第五阶段的聊天主链路

为什么:

因为触发规则通常要基于已有数据来判断，比如:

1. 最近几天有没有记录健康数据
2. 最近聊天里有没有焦虑关键词
3. 最近有没有继续活跃

所以规则引擎一定是依赖现有业务数据的。

---

## 5. 先说明你当前项目里的主动相关草稿文件

你现在仓库里已经有这些草稿文件:

1. `backend/app/db/models/proactive_window.py`
2. `backend/app/db/models/proactive_log.py`
3. `backend/app/schemas/proactive.py`

这些文件说明:

你项目已经开始往“主动提醒”方向预留结构了。

但第六阶段我们建议你先正式统一成新的命名和结构:

1. `TriggerRule`
2. `ActiveLog`

### 5.1 这一阶段的处理原则

1. 正式版规则模型放到 `app/models/trigger_rule.py`
2. 正式版日志模型放到 `app/models/active_log.py`
3. `app/db/models/proactive_*` 先不要继续作为正式主链路使用
4. `app/schemas/proactive.py` 这一阶段先不要动
5. 新的 schema、service、route 全部按第六阶段正式结构新建

### 5.2 为什么这里不继续直接用 `proactive_window`

因为 `proactive_window` 更像是:

“什么时候允许主动触达”

它偏向的是“时间窗口策略”。

而第六阶段当前要做的是:

“什么条件下触发主动介入”

这两个概念不是一回事。

所以第六阶段更适合先建:

1. `TriggerRule`
2. `ActiveLog`

后面你如果要加“时间窗口限制”，再把 `proactive_window` 接回来也不迟。

## 6. 第六阶段推荐的数据设计

## 6.1 TriggerRule 表

推荐字段:

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | int | 主键 |
| `name` | str | 规则名称 |
| `trigger_type` | str | 触发类型 |
| `enabled` | bool | 是否启用 |
| `condition_json` | json | 规则条件 |
| `priority` | int | 优先级 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

### `condition_json` 存什么

它的作用是:

让规则的条件可配置，而不是写死在代码里。

比如:

```json
{
  "days_without_record": 3,
  "record_type": "blood_pressure"
}
```

或者:

```json
{
  "keywords": ["焦虑", "睡不着", "害怕"],
  "recent_limit": 10
}
```

### 这里为什么推荐用 `JSON`

因为规则条件天然就是灵活结构。

如果你现在把所有条件都硬拆成固定字段，很快就会遇到:

1. 新增一种规则就得改表
2. 删除一种规则字段就很尴尬
3. 规则类型越多，表越臃肿

所以第六阶段最合适的做法就是:

规则元信息固定，具体条件放进 `condition_json`。

## 6.2 ActiveLog 表

推荐字段:

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | int | 主键 |
| `user_id` | int | 触发目标用户 |
| `trigger_rule_id` | int | 来源规则 |
| `action_type` | str | 执行动作类型 |
| `status` | str | 状态 |
| `request_payload` | json | 本次检查输入 |
| `response_payload` | json | 本次检查输出 |
| `created_at` | datetime | 创建时间 |

### 为什么这里先做日志，而不是先做主动发送

因为第六阶段的重点不是“把消息推给用户”，而是:

先把“这次检查发生了什么”记录清楚。

比如你手动检查一条规则时，日志至少应该能回答:

1. 检查的是哪条规则
2. 检查的是哪个用户
3. 当时传入了什么条件
4. 最终有没有命中
5. 如果命中，理由是什么
6. 如果失败，报错是什么

这就是日志模块的价值。

## 7. 第六阶段建议支持的最小触发类型

这一阶段不要贪多。

建议你先只做下面两种触发类型:

1. `days_without_record`
2. `recent_chat_keyword`

这已经足够把规则引擎和日志链路练通。

## 7.1 `days_without_record`

意思是:

用户在最近若干天内，没有某类健康记录。

例如:

```json
{
  "days_without_record": 3,
  "record_type": "blood_pressure"
}
```

表示:

如果用户 3 天内没有血压记录，就算触发。

## 7.2 `recent_chat_keyword`

意思是:

检查用户最近若干条聊天消息里，是否出现某些关键词。

例如:

```json
{
  "keywords": ["焦虑", "害怕", "睡不着"],
  "recent_limit": 10
}
```

表示:

如果最近 10 条用户消息里出现这些关键词之一，就触发。

### 为什么第六阶段不建议先做血压阈值判断

因为你第三阶段里 `Record.value` 先被设计成了字符串。

比如血压可能长这样:

`138/91`

这时候如果你马上做复杂数值阈值解析，会把第六阶段搞得太复杂。

所以第六阶段先挑:

1. 一个基于“有没有数据”的规则
2. 一个基于“聊天关键词”的规则

这两个最容易讲清楚，也最适合先练通。

所以这一版直接把“规则 CRUD + 规则评估 + 手动检查写日志”统一放进 `trigger_rule_service.py`。

为什么这里没有立刻提供“主动发送消息”接口:

因为第六阶段的目标还不是主动执行动作，而是:

先把规则和日志地基打稳。

## 10. 开发顺序建议

这个顺序不要乱:

1. 先建 `TriggerRule` 模型
2. 再建 `ActiveLog` 模型
3. 再补 `app/models/__init__.py`
4. 再补 `app/db/base.py`
5. 再写 `TriggerRule` schema
6. 再写 `ActiveLog` schema
7. 再写 `ActiveLogService`
8. 再写 `TriggerRuleService`
9. 再写 `trigger_rules.py`
10. 再写 `active_logs.py`
11. 再补总路由
12. 最后做 Alembic migration 和 Swagger 验收

原因很简单:

1. 没有模型，规则和日志就没有表
2. 没有 schema，输入输出边界不清楚
3. 没有 service，路由层会堆满业务逻辑
4. 没有 base 导入，Alembic 会生成空 migration
5. 规则评估先收敛在 `trigger_rule_service.py`，第七阶段才能直接升级成复用版

## 12. 新建 `TriggerRule` ORM 模型

### 12.1 新建 `app/models/trigger_rule.py`

完整代码如下:

```python
from __future__ import annotations

from datetime import UTC, datetime
from typing import Any

from sqlalchemy import JSON, Boolean, DateTime, Integer, String
from sqlalchemy.orm import Mapped, mapped_column

from app.db.session import Base


class TriggerRule(Base):
    __tablename__ = "trigger_rules"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, index=True)
    name: Mapped[str] = mapped_column(String(100), nullable=False)
    trigger_type: Mapped[str] = mapped_column(String(50), nullable=False, index=True)
    enabled: Mapped[bool] = mapped_column(Boolean, nullable=False, default=True, index=True)
    condition_json: Mapped[dict[str, Any]] = mapped_column(JSON, nullable=False)
    priority: Mapped[int] = mapped_column(Integer, nullable=False, default=100, index=True)
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        default=lambda: datetime.now(UTC),
        nullable=False,
    )
    updated_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        default=lambda: datetime.now(UTC),
        onupdate=lambda: datetime.now(UTC),
        nullable=False,
    )
```

### 12.2 这份模型代码在做什么

1. `name` 负责描述这条规则是什么
2. `trigger_type` 负责标记规则类型
3. `enabled` 表示规则是否启用
4. `condition_json` 用来存具体条件
5. `priority` 用来表达优先级
6. `created_at` 和 `updated_at` 用来追踪时间

### 12.3 为什么 `condition_json` 不拆成固定字段

因为不同规则的条件结构天然不同。

比如:

1. `days_without_record` 需要 `days_without_record` 和 `record_type`
2. `recent_chat_keyword` 需要 `keywords` 和 `recent_limit`

如果你现在就把这些都拆成固定字段，后面规则一多，表会很乱。

所以这类“规则系统”最适合:

规则元信息固定，具体条件用 JSON 存。

## 13. 新建 `ActiveLog` ORM 模型

### 13.1 新建 `app/models/active_log.py`

完整代码如下:

```python
from __future__ import annotations

from datetime import UTC, datetime
from typing import Any

from sqlalchemy import JSON, DateTime, ForeignKey, Integer, String
from sqlalchemy.orm import Mapped, mapped_column

from app.db.session import Base


class ActiveLog(Base):
    __tablename__ = "active_logs"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, index=True)
    user_id: Mapped[int] = mapped_column(
        ForeignKey("users.id", ondelete="CASCADE"),
        nullable=False,
        index=True,
    )
    trigger_rule_id: Mapped[int] = mapped_column(
        ForeignKey("trigger_rules.id"),
        nullable=False,
        index=True,
    )
    action_type: Mapped[str] = mapped_column(String(30), nullable=False, default="rule_check")
    status: Mapped[str] = mapped_column(String(20), nullable=False, default="pending", index=True)
    request_payload: Mapped[dict[str, Any] | None] = mapped_column(JSON, nullable=True)
    response_payload: Mapped[dict[str, Any] | None] = mapped_column(JSON, nullable=True)
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        default=lambda: datetime.now(UTC),
        nullable=False,
        index=True,
    )
```

### 13.2 这份日志模型代码在做什么

1. `user_id` 表示这次检查是针对哪个用户
2. `trigger_rule_id` 表示这次检查来自哪条规则
3. `action_type` 表示动作类型
4. `status` 表示这次检查是命中、未命中还是失败
5. `request_payload` 表示这次检查的输入
6. `response_payload` 表示这次检查的输出

### 13.3 为什么这里 `action_type` 先固定成 `rule_check`

因为第六阶段还没有真正做主动执行动作。

所以当前阶段最主要的行为其实是:

“手动做一次规则检查”

这正适合先统一记成:

`rule_check`

后面如果你真的开始主动发送消息了，再扩展成:

1. `send_message`
2. `create_task`
3. `push_notification`

都可以。

如果你已经按前面阶段一路做过来，建议这里直接替换成下面这份累积版代码。


如果你前面阶段还没有真正落地，就把对应没做过的模型先删掉，保留你当前已有的部分。

### 14.2 修改 `app/db/base.py`

同样建议直接替换成下面这份累积版代码。

完整代码如下:

```python
from app.db.session import Base
from app.models.active_log import ActiveLog  # noqa: F401
from app.models.conversation import Conversation  # noqa: F401
from app.models.message import Message  # noqa: F401
from app.models.profile import Profile  # noqa: F401
from app.models.record import Record  # noqa: F401
from app.models.trigger_rule import TriggerRule  # noqa: F401

    "Base",
    "User",
    "Profile",
    "Record",
    "Conversation",
    "Message",
    "TriggerRule",
    "ActiveLog",
]
```

### 14.3 这一步为什么必须做

因为 Alembic 自动生成 migration 时，最终是通过:

`Base.metadata`

来发现项目里的 ORM 模型。

如果你忘了把新模型接进来，后果就是:

1. migration 是空的
2. 新表根本不会被创建
3. 后面接口一跑就报表不存在

## 15. 写 `TriggerRule` 的 Schema

schema 的作用是:

1. 定义创建规则时前端能传什么
2. 定义更新规则时前端能传什么
3. 定义读取规则时返回什么
4. 定义手动触发检查后返回什么

### 15.1 新建 `app/schemas/trigger_rule.py`

完整代码如下:

```python
from datetime import datetime
from typing import Any

from pydantic import BaseModel, ConfigDict, Field


class TriggerRuleBase(BaseModel):
    name: str = Field(..., min_length=1, max_length=100)
    trigger_type: str = Field(..., min_length=1, max_length=50)
    enabled: bool = True
    condition_json: dict[str, Any] = Field(default_factory=dict)
    priority: int = Field(default=100, ge=0, le=9999)


class TriggerRuleCreate(TriggerRuleBase):
    pass


class TriggerRuleUpdate(BaseModel):
    name: str | None = Field(default=None, min_length=1, max_length=100)
    trigger_type: str | None = Field(default=None, min_length=1, max_length=50)
    enabled: bool | None = None
    condition_json: dict[str, Any] | None = None
    priority: int | None = Field(default=None, ge=0, le=9999)


class TriggerRuleRead(TriggerRuleBase):
    id: int
    created_at: datetime
    updated_at: datetime

    model_config = ConfigDict(from_attributes=True)


class TriggerRuleListData(BaseModel):
    items: list[TriggerRuleRead]


class TriggerRuleCheckData(BaseModel):
    log_id: int
    rule_id: int
    user_id: int
    action_type: str
    status: str
    triggered: bool
    reason: str
    response_payload: dict[str, Any] | None = None
    created_at: datetime
```

### 15.2 这份规则 schema 在做什么

#### `TriggerRuleCreate`

作用:

创建一条规则。

#### `TriggerRuleUpdate`

作用:

更新规则。

这里字段都允许不传，是为了支持局部更新。

#### `TriggerRuleRead`

作用:

把数据库里的规则记录返回给前端。

#### `TriggerRuleCheckData`

作用:

表示“手动检查一次规则”之后的返回结果。

为什么这里直接返回检查结果，而不是只返回日志 id:

因为你在 Swagger 演示时，最想马上看到的是:

1. 这次有没有命中
2. 为什么命中
3. 日志 id 是多少

所以这个返回结构对调试最友好。

## 16. 写 `ActiveLog` 的 Schema

### 16.1 新建 `app/schemas/active_log.py`

```python
class ActiveLogRead(BaseModel):

    items: list[ActiveLogRead]
```

### 16.2 为什么日志 schema 只做读取，不做创建

因为第六阶段的日志不应该由前端自己创建。

日志的正确创建方式应该是:

后端在规则检查过程中自动写入。

也就是说:

1. 前端可以看日志
2. 但前端不应该手动伪造日志

这就是为什么这里没有 `ActiveLogCreate`。

## 18. 写 `TriggerRuleService`

这一层是第六阶段真正的核心。

它要同时负责三件事:

1. 规则的增删改查
2. 规则评估
3. 手动检查时写主动日志

### 18.1 新建 `app/services/trigger_rule_service.py`

完整代码如下:

```python
from __future__ import annotations

from datetime import UTC, datetime, timedelta
from typing import Any

from sqlalchemy import select
from sqlalchemy.orm import Session

from app.core.exception import BusinessException
from app.models.conversation import Conversation
from app.models.message import Message
from app.models.record import Record
from app.models.trigger_rule import TriggerRule
from app.schemas.trigger_rule import TriggerRuleCreate, TriggerRuleUpdate
from app.services.active_log_service import active_log_service


class TriggerRuleService:
    def create(
        self,
        db: Session,
        payload: TriggerRuleCreate,
    ) -> TriggerRule:
        rule = TriggerRule(
            name=payload.name.strip(),
            trigger_type=payload.trigger_type.strip(),
            enabled=payload.enabled,
            condition_json=payload.condition_json,
            priority=payload.priority,
        )
        db.add(rule)
        db.commit()
        db.refresh(rule)
        return rule

    def list_all(self, db: Session) -> list[TriggerRule]:
        stmt = select(TriggerRule).order_by(
            TriggerRule.priority.asc(),
            TriggerRule.id.asc(),
        )
        return list(db.scalars(stmt).all())

    def get_or_raise(self, db: Session, rule_id: int) -> TriggerRule:
        rule = db.get(TriggerRule, rule_id)
        if not rule:
            raise BusinessException(
                code=40462,
                message="trigger rule not found",
                status_code=404,
            )
        return rule

    def update(
        self,
        db: Session,
        rule_id: int,
        payload: TriggerRuleUpdate,
    ) -> TriggerRule:
        rule = self.get_or_raise(db, rule_id)
        update_data = payload.model_dump(exclude_unset=True)

        if "name" in update_data and update_data["name"] is not None:
            update_data["name"] = update_data["name"].strip()

        if (
            "trigger_type" in update_data
            and update_data["trigger_type"] is not None
        ):
            update_data["trigger_type"] = update_data["trigger_type"].strip()

        for field, value in update_data.items():
            setattr(rule, field, value)

        db.add(rule)
        db.commit()
        db.refresh(rule)
        return rule

    def delete(self, db: Session, rule_id: int) -> None:
        rule = self.get_or_raise(db, rule_id)
        db.delete(rule)
        db.commit()

    def evaluate_for_user(
        self,
        db: Session,
        user_id: int,
        rule: TriggerRule,
    ) -> dict[str, Any]:
        if not rule.enabled:
            return {
                "triggered": False,
                "reason": "rule is disabled",
                "trigger_type": rule.trigger_type,
            }

        if rule.trigger_type == "days_without_record":
            return self._evaluate_days_without_record(db, user_id, rule)

        if rule.trigger_type == "recent_chat_keyword":
            return self._evaluate_recent_chat_keyword(db, user_id, rule)

        raise BusinessException(
            code=40063,
            message=f"unsupported trigger_type: {rule.trigger_type}",
            status_code=400,
        )

    def check_for_user(
        self,
        db: Session,
        rule_id: int,
        user_id: int,
    ) -> dict[str, Any]:
        rule = self.get_or_raise(db, rule_id)
        request_payload = {
            "rule_id": rule.id,
            "trigger_type": rule.trigger_type,
            "condition_json": rule.condition_json,
            "checked_at": datetime.now(UTC).isoformat(),
        }

        try:
            result = self.evaluate_for_user(db, user_id, rule)
            status = "triggered" if result["triggered"] else "not_triggered"
            log = active_log_service.create_log(
                db=db,
                user_id=user_id,
                trigger_rule_id=rule.id,
                action_type="rule_check",
                status=status,
                request_payload=request_payload,
                response_payload=result,
            )
            return {
                "log_id": log.id,
                "rule_id": rule.id,
                "user_id": user_id,
                "action_type": "rule_check",
                "status": status,
                "triggered": result["triggered"],
                "reason": result["reason"],
                "response_payload": result,
                "created_at": log.created_at,
            }
        except BusinessException as exc:
            active_log_service.create_log(
                db=db,
                user_id=user_id,
                trigger_rule_id=rule.id,
                action_type="rule_check",
                status="failed",
                request_payload=request_payload,
                response_payload={"error": exc.message},
            )
            raise

    def _evaluate_days_without_record(
        self,
        db: Session,
        user_id: int,
        rule: TriggerRule,
    ) -> dict[str, Any]:
        condition = rule.condition_json or {}
        days_without_record = int(condition.get("days_without_record", 0))
        record_type = str(condition.get("record_type", "")).strip()

        if days_without_record <= 0:
            raise BusinessException(
                code=40064,
                message="days_without_record must be greater than 0",
                status_code=400,
            )

        if not record_type:
            raise BusinessException(
                code=40065,
                message="record_type is required for days_without_record",
                status_code=400,
            )

        stmt = (
            select(Record)
            .where(
                Record.user_id == user_id,
                Record.record_type == record_type,
            )
            .order_by(Record.record_time.desc(), Record.id.desc())
            .limit(1)
        )
        last_record = db.scalar(stmt)

        cutoff_time = datetime.now(UTC) - timedelta(days=days_without_record)
        triggered = last_record is None or last_record.record_time < cutoff_time

        if last_record is None:
            reason = f"user has no {record_type} record yet"
        elif triggered:
            reason = f"last {record_type} record is older than {days_without_record} days"
        else:
            reason = f"user still has recent {record_type} record"

        return {
            "triggered": triggered,
            "reason": reason,
            "trigger_type": rule.trigger_type,
            "record_type": record_type,
            "days_without_record": days_without_record,
            "last_record_time": (
                last_record.record_time.isoformat() if last_record else None
            ),
        }

    def _evaluate_recent_chat_keyword(
        self,
        db: Session,
        user_id: int,
        rule: TriggerRule,
    ) -> dict[str, Any]:
        condition = rule.condition_json or {}
        raw_keywords = condition.get("keywords", [])
        keywords = [str(item).strip() for item in raw_keywords if str(item).strip()]
        recent_limit = int(condition.get("recent_limit", 10))

        if not keywords:
            raise BusinessException(
                code=40066,
                message="keywords cannot be empty",
                status_code=400,
            )

        if recent_limit <= 0:
            raise BusinessException(
                code=40067,
                message="recent_limit must be greater than 0",
                status_code=400,
            )

        stmt = (
            select(Message)
            .join(Conversation, Message.conversation_id == Conversation.id)
            .where(
                Conversation.user_id == user_id,
                Message.role == "user",
            )
            .order_by(Message.created_at.desc(), Message.id.desc())
            .limit(recent_limit)
        )
        recent_messages = list(db.scalars(stmt).all())

        matched_items: list[dict[str, Any]] = []
        for message in recent_messages:
            lowered_content = message.content.lower()
            for keyword in keywords:
                if keyword.lower() in lowered_content:
                    matched_items.append(
                        {
                            "keyword": keyword,
                            "message_id": message.id,
                        }
                    )

        matched_keywords = list(
            dict.fromkeys(item["keyword"] for item in matched_items)
        )
        matched_message_ids = list(
            dict.fromkeys(item["message_id"] for item in matched_items)
        )
        triggered = bool(matched_items)

        if triggered:
            reason = f"recent chat contains keywords: {', '.join(matched_keywords)}"
        else:
            reason = "recent chat does not contain configured keywords"

        return {
            "triggered": triggered,
            "reason": reason,
            "trigger_type": rule.trigger_type,
            "recent_limit": recent_limit,
            "matched_keywords": matched_keywords,
            "matched_message_ids": matched_message_ids,
        }


trigger_rule_service = TriggerRuleService()
```

### 18.2 这份 `TriggerRuleService` 最关键的点

#### 1. 第六阶段就先把规则评估抽成 `evaluate_for_user`

这是为了后面第七阶段直接复用。

因为第七阶段真正需要的不是“再写一遍规则判断”，而是:

复用第六阶段已经成立的评估能力。

#### 2. `check_for_user` 本质上是:

`evaluate_for_user + create_log`

也就是说:

1. 规则判断负责回答“命中没命中”
2. 日志负责回答“这次检查发生了什么”

这样职责就清楚了。

#### 3. 为什么这里不再单独拆 `trigger_engine_service.py`

因为第六阶段的规则类型还很少。

现在只支持:

1. `days_without_record`
2. `recent_chat_keyword`

这时候再额外拆一层 engine，会让教程复杂度先上去，但对当前阶段收益不大。

先把评估能力收敛在 `trigger_rule_service.py`，等后面规则类型明显增多，再拆独立引擎也不迟。

#### 4. 即使规则检查失败，也要写日志

这点很重要。

比如:

1. 配置缺字段
2. `trigger_type` 不支持
3. 条件值非法

这些都应该留下 `failed` 日志。

因为第六阶段的价值之一，就是把规则检查过程变成可追踪对象。

## 24. 第六阶段的 Swagger 验收顺序

下面给你一个推荐的 Swagger 验收顺序。

这一段的目标不是把接口机械地点一遍，而是用最少步骤把第六阶段的 6 条验收目标全部证明出来。

## 24.1 先创建一条容易验收的规则

推荐先用 `days_without_record` 做第一条规则。

调用:

`POST /api/v1/trigger-rules`

请求体示例:

```json
{
  "name": "三天未记录血压",
  "trigger_type": "days_without_record",
  "enabled": true,
  "condition_json": {
    "days_without_record": 3,
    "record_type": "blood_pressure"
  },
  "priority": 10
}
```

如果你的当前测试账号最近 3 天里已经有 `blood_pressure` 记录，也没关系。

你可以改成:

1. 调大 `days_without_record`
2. 或者换成一个当前账号还没有写过的 `record_type`

这一步验的是:

1. 规则可创建
2. `condition_json` 可正常落库
3. 规则配置不是写死在代码里

记住返回里的 `rule_id`。

## 24.2 查看规则列表和规则详情

先调用:

`GET /api/v1/trigger-rules`

你应该能在列表里看到刚刚创建的规则。

然后调用:

`GET /api/v1/trigger-rules/{rule_id}`

你应该重点确认:

1. `name` 正确
2. `trigger_type` 正确
3. `enabled` 正确
4. `condition_json` 正确
5. `priority` 正确

这一步验的是:

规则不只是“创建接口返回成功”，而是真的可以被读取和管理。

## 24.3 再更新一次规则

调用:

`PUT /api/v1/trigger-rules/{rule_id}`

请求体示例:

```json
{
  "name": "三天未记录血压-高优先级",
  "enabled": true,
  "condition_json": {
    "days_without_record": 3,
    "record_type": "blood_pressure"
  },
  "priority": 5
}
```

更新后再查一次详情。

你应该能看到:

1. 名称已变化
2. 优先级已变化
3. `condition_json` 仍然保持正确结构

这一步验的是:

规则不仅能创建，还能被持续维护。

## 24.4 手动执行一次当前用户规则检查

调用:

`POST /api/v1/trigger-rules/{rule_id}/check/me`

这是第六阶段最核心的 Swagger 验收接口。

你应该重点看返回里有没有这些字段:

1. `log_id`
2. `rule_id`
3. `user_id`
4. `action_type`
5. `status`
6. `triggered`
7. `reason`
8. `created_at`

如果当前用户满足条件，比如最近 3 天没有对应记录，你应该至少看到:

1. `triggered = true`
2. `reason` 能解释为什么命中
3. `log_id` 已生成

如果当前用户不满足条件，你也应该至少看到:

1. `triggered = false`
2. `reason` 能解释为什么未命中
3. `log_id` 依然已生成

这里非常关键:

无论命中还是未命中，第六阶段都应该留下检查日志。

因为第六阶段验的不是“有没有真的主动发消息”，而是:

规则判断链路和日志链路是不是通了。

## 24.5 查看主动日志列表

调用:

`GET /api/v1/active-logs`

你应该能看到刚刚那次检查生成的日志。

建议重点确认:

1. `user_id` 是当前登录用户
2. `trigger_rule_id` 等于刚才的 `rule_id`
3. `action_type` 是你当前实现里的规则检查动作类型
4. `status` 和刚才手动检查接口返回一致
5. `created_at` 已写入

如果你的实现把规则检查动作统一写成:

`rule_check`

那么这里就应该看到 `action_type = "rule_check"`。

## 24.6 查看单条主动日志详情

调用:

`GET /api/v1/active-logs/{log_id}`

这一步要重点看日志详情是不是足够解释这次检查。

至少应该能看清:

1. 这是哪条规则触发的
2. 这是给哪个用户检查的
3. 当时的输入条件是什么
4. 最终结果是什么
5. 为什么命中或为什么没命中

如果你的实现正确，通常会在:

1. `request_payload`
2. `response_payload`

里看到本次检查的输入和输出。

这一步验的是:

`active_log` 不是一个空壳表，而是真的能承载“检查结果留痕”。

## 24.7 用第二个账号验证日志隔离

这是很多人最容易漏掉的一步，但它正好对应第六阶段最后一条验收标准。

你应该再准备一个账号，然后:

1. 用第二个账号登录
2. 在 Swagger 里重新 `Authorize`
3. 调用 `GET /api/v1/active-logs`
4. 再尝试调用第一个账号产生的 `GET /api/v1/active-logs/{log_id}`

你应该看到的正确结果是:

1. 第二个账号的日志列表里，不会出现第一个账号的日志
2. 第二个账号访问第一个账号的日志详情时，会返回 `404` 或 `403`

如果这里还能读到别人的日志，第六阶段就不能算通过。

## 24.8 如果你想把演示做得更完整

可以再补一条:

`recent_chat_keyword`

规则，再手动检查一次。

比如:

```json
{
  "name": "最近聊天出现焦虑关键词",
  "trigger_type": "recent_chat_keyword",
  "enabled": true,
  "condition_json": {
    "keywords": ["焦虑", "害怕", "睡不着"],
    "recent_limit": 10
  },
  "priority": 20
}
```

这样你就能证明:

第六阶段不是只支持一种硬编码判断，而是真的开始具备“规则引擎雏形”了。

## 25. 第六阶段补完整后，你应该已经具备什么

如果你按现在这份教程完整做完，你应该已经有了:

1. `TriggerRule`
2. `ActiveLog`
3. `app/services/active_log_service.py`
4. `app/services/trigger_rule_service.py`
5. `trigger_rule_service.evaluate_for_user()`
6. `POST /api/v1/trigger-rules/{rule_id}/check/me`
7. `GET /api/v1/active-logs`
8. 可执行的 Alembic migration

这几个点正是后面第七阶段和第八阶段要复用的底层能力。

也就是说:

1. 第七阶段可以直接在这个基础上接 `ProactiveWindow` 和 `ProactiveMessage`
2. 第七阶段可以直接复用 `trigger_rule_service.evaluate_for_user()`
3. 第八阶段的 Scheduler 可以建立在第七阶段主动服务之上继续往前推

到这里，第六阶段才算真正闭环。
