02. Flask 到 FastAPI 的思维迁移

1. 不要把 FastAPI 当成“Flask 换皮”

很多会 Flask 的人学 FastAPI，第一反应是：

  • Flask 里怎么写，我在 FastAPI 里也这么写
  • 只要会 @app.get() / @app.post() 就算学会了

这会导致一个问题：
你写出来的还是 Flask 风格的“小脚本后端”，只是外面套了 FastAPI。

FastAPI 真正强的地方不是“语法更短”，而是：

  • 类型提示更自然
  • schema 更规范
  • 文档自动生成
  • 适合项目化拆分

2. Flask 与 FastAPI 的对应关系

Flask 

from flask import Flask, request

app = Flask(__name__)

@app.route("/chat", methods=["POST"])
def chat():
    data = request.get_json()
    msg = data["message"]
    return {"reply": f"你说的是：{msg}"}

FastAPI 

from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class ChatRequest(BaseModel):
    message: str

@app.post("/chat")
def chat(data: ChatRequest):
    return {"reply": f"你说的是：{data.message}"}

你要注意的关键点：

  • FastAPI 喜欢先定义数据模型
  • 参数解析更依赖类型系统
  • 返回值更容易约束成统一格式

3. 你的思维要从“拿数据”变成“定义数据”

Flask 常见思维

  • 从 request 里掏参数
  • 自己判断字段存不存在
  • 代码里到处写 if not ...

FastAPI 推荐思维

  • 先定义请求和响应长什么样
  • 用 schema 明确字段类型和约束
  • 让框架自动校验

比如：

from pydantic import BaseModel, Field

class ProfileCreate(BaseModel):
    name: str = Field(..., min_length=1, max_length=50)
    age: int = Field(..., ge=0, le=120)
    gender: str

这样做的好处是：

  • 参数错误会自动报 422
  • Swagger 文档自动反映字段信息
  • 代码更容易读懂

4. 你的接口函数不再是“把所有事做完”

Flask 初学者很容易写出：

@app.route("/xxx")
def xxx():
    # 读参数
    # 校验参数
    # 查数据库
    # 调第三方接口
    # 拼接返回值

FastAPI 项目里你应该逐步形成这种分工：

路由层

只负责：

  • 接收请求
  • 调 service
  • 返回结果

service 层

只负责：

  • 业务规则
  • 调用数据库 / 模型 / 外部服务

schema 层

只负责：

  • 数据结构定义

这才是你从 Flask 迁移时最该完成的能力升级。

5. 什么叫“面向业务而不是面向 HTTP”

比如聊天功能。

错误写法是把“调用模型”的逻辑直接写在路由里：

@app.post("/api/chat")
def chat(data: ChatRequest):
    response = dashscope.Generation.call(...)
    return {"reply": response["output"]["text"]}

更好的写法是：

@app.post("/api/chat")
def chat(data: ChatRequest):
    reply = chat_service.reply(data.message)
    return success(data={"reply": reply})

此时：

  • 接口只是一个 HTTP 入口
  • 真正的业务是“根据消息生成健康助手回复”

这就是面向业务的思维。

6. FastAPI 里最值得你重视的 5 个点

6.1 Pydantic

这是 FastAPI 的核心体验之一。
你以后所有复杂请求都应该优先写 schema。

6.2 APIRouter

不要所有接口都塞在一个文件里。

6.3 response_model

让响应结构更清晰，也能自动生成文档。

6.4 依赖注入（Dependency）

第一阶段你可以先少用，但要知道以后会用于：

  • 鉴权
  • 数据库 session
  • 公共参数
  • 权限控制

6.5 自动文档

/docs 是你联调时的神兵利器。

7. 你作为 Flask 用户最容易犯的 7 个错误

错误 1：继续把所有逻辑写在 main.py

这是最常见的问题。

错误 2：过度依赖 GET

聊天、创建、提交数据都应该优先考虑 POST。

错误 3：不用 schema

直接拿 dict 写起来快，但很快会乱。

错误 4：不写服务层

所有复杂逻辑都写在路由里，后面会爆炸。

错误 5：忽略统一响应格式

前后端联调时会很痛苦。

错误 6：配置散落在多个文件

以后换环境会很麻烦。

错误 7：为了“高级”而过早引入异步

你现在先把结构写对，比先用 async def 更重要。

8. 关于 def 和 async def

很多人学 FastAPI 会纠结：
“是不是必须全都写成异步？”

答案：不是。

第一阶段建议

如果你：

  • 调普通 Python 函数
  • 调同步 SDK
  • 还没接异步数据库

那么先写普通的 def 就行。

@router.post("/api/chat")
def chat(data: ChatRequest):
    ...

什么时候考虑 async

以后如果你：

  • 接异步数据库
  • 接异步 HTTP 客户端
  • 整个调用链都是异步

再考虑 async def。

9. 一句话总结迁移思路

从 Flask 迁移到 FastAPI，最重要的不是学会几个装饰器，而是完成下面这个转变：

从“脚本式写接口”转向“项目式写后端”。
