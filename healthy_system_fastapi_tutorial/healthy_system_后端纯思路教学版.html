healthy_system 后端纯思路教学版

只讲设计思路 不直接讲实现细节 适合和完整代码版对照

这份教程解决的不是“代码怎么写”，而是“为什么要这么设计”。如果你只会照抄代码，后面一变需求你就不会改；但如果你理解了这份思路版，再回去看完整代码版，你会知道每个文件到底在承担什么职责。

0. 这份教程的定位

你现在已经有一个“完整代码版”教程，那一份适合解决下面的问题：

  1. 这个文件具体写什么。
  2. 接口应该怎么组织。
  3. 某个模块最小可跑版本怎么搭出来。

但只看代码版会有一个明显问题：你容易把后端理解成很多文件的拼接，而不是一个有逻辑的系统。

所以这一份“纯思路教学版”要解决的是另外几个问题：

  1. 为什么你的系统中心一定是老人档案，而不是聊天接口。
  2. 为什么比赛项目不能只做 CRUD，也不能一上来做太多炫技功能。
  3. 为什么要分成 route、service、repository 这几层。
  4. 为什么聊天、主动触发、WebSocket、调度器之间必须按顺序搭建。

你可以把这份文档当成“后端脑图版”。它不替你写代码，但它会告诉你每一行代码背后的判断依据。

1. 你到底在做什么系统

你做的不是一个普通的健康数据录入系统，也不是一个单纯的大模型问答机器人。

你真正要做的是一个以老人健康数据为底座、具备连续对话能力、并且能主动关怀的健康陪护后端。

这里面有三个关键词：

关键词含义对后端的要求
健康数据不是空聊天，必须有老人档案和健康记录要有稳定的数据表和查询逻辑
连续对话不是一问一答结束，而是有会话上下文要保存消息历史，能取上下文
主动关怀系统不能永远等用户来问要有触发规则、消息生成、推送和日志
如果你把这三个关键词理解透，你后面就不会乱。任何一个模块，只要你不知道该不该做，就问自己一句：它是在支撑健康数据、连续对话，还是主动关怀？

2. 比赛版后端和普通 CRUD 的区别

普通学生项目常见的路线是：建几张表，写几个增删改查接口，再加一个看起来像智能功能的接口。这种项目的问题是，技术元素虽然齐，但没有主线。

比赛版项目则完全不同。老师看重的不只是“功能数量”，更看重你是不是把一个明确场景的主链路闭环做出来了。

普通 CRUD 思路

表很多，接口很多，但每个功能相互孤立。老人档案和聊天没有关系，聊天和健康记录没有关系，提醒功能也只是单独存在。

比赛闭环思路

老人档案为基础，健康记录持续沉淀，聊天利用这些数据进行上下文回复，主动触发再基于这些数据做判断，然后把生成的主动消息实时推送出去。

这两种思路的根本差异不在于“会不会写代码”，而在于你有没有建立一条清晰主线。

对你的项目来说，真正的主线不是“我用了 FastAPI、SQLite、WebSocket、LLM”，而是“我让系统围绕老人健康场景形成了一条可解释、可演示、可扩展的服务链路”。

3. 核心业务对象怎么想

如果你一开始业务建模没想清楚，后面所有代码都会变得很别扭。所谓“业务建模”，本质上就是在回答这个问题：

系统里真正稳定、真正重要、真正长期存在的数据对象，到底有哪些？

3.1 为什么 profile 是中心

在你的项目里，profile 不是一个普通用户表，而是所有业务的锚点。

原因很简单：无论是健康记录、聊天记录、主动提醒，最后都必须能回答“这是针对哪位老人”。

所以 profile 承担的是三种作用：

  1. 身份作用：确定系统服务对象是谁。
  2. 画像作用：保存年龄、慢病、作息等长期信息。
  3. 关联作用：让其它表都挂到一个主实体上。

也就是说，你的后端并不是围绕“登录用户”建模，而是围绕“被服务的老人”建模。

只要这一点想清楚，你就能解释为什么很多接口都要带 profile_id。

3.2 为什么 record 先做通用表

很多人做健康项目时，第一反应是把血压、血糖、睡眠、运动、服药分别拆成很多表。这样理论上很规范，但对你当前阶段并不划算。

比赛版第一阶段更重要的是快速形成健康数据底座，而不是做特别精细的医疗信息系统。

因此把 record 设计成通用健康记录表，本质上是在做一个权衡：

方案优点缺点
很多专业子表结构清晰、后期统计方便早期开发成本高，迁移和联调都慢
一个通用记录表成型快、扩展快、适合比赛演示字段约束没那么强，需要靠业务层规范

所以你现在不是不知道更“正规”的表怎么做，而是故意选了一个更适合现阶段的方案。

通用记录表不是偷懒，而是工程上的阶段性最优解。

3.3 为什么 conversation 和 message 要拆开

很多初学者写聊天接口时，只有一个请求进来，后端直接返回一句话。这样虽然表面上“能聊天”，但这个系统其实没有记忆。

真正的聊天系统至少要区分两个层面：

  1. conversation：表示一次会话本身。
  2. message：表示会话中的一条条消息。

你可以把它理解成：

  • conversation 是一个容器。
  • message 是容器里面的内容。

这样拆开的原因不是为了“表更多”，而是因为业务本来就是两层。

如果你不拆，会立刻遇到这几个问题：

  1. 一位老人可以有多次聊天，你怎么区分不同会话？
  2. 你怎么查询某次会话的历史消息？
  3. 主动消息插到哪里？是聊天的一部分还是另一套系统？

拆开之后，普通聊天消息和主动关怀消息都能进入同一套消息体系，只是 message_type 不同。

3.4 为什么还要主动窗口和主动日志

这里是比赛版和普通版拉开差距的关键。

如果你只是“在某个时刻直接发一条主动消息”，老师很可能会问你两个问题：

  1. 为什么是这个时候发？
  2. 你怎么证明系统真的有主动判断逻辑？

所以你需要两张额外的表：

表真正作用
proactive_window表达“系统允许在什么时间段打扰用户”
proactive_log记录“系统为什么触发、触发了什么、结果是什么”

这两张表最重要的价值不是代码层面的，而是可解释性。

你不是在做一个黑盒系统，而是在做一个能解释“为什么发出这条关怀消息”的系统。

4. 分层不是形式，是在解决复杂度

很多人知道“要分层”，但不知道为什么要分层。结果是文件虽然拆了，逻辑仍然乱。

你必须先明白一件事：

分层的目的不是好看，而是把不同类型的问题分开处理。

4.1 Route 层到底负责什么

route 层只做三件事：

  1. 接收请求。
  2. 调用 service。
  3. 返回统一响应。

它不应该负责复杂判断，不应该直接写数据库操作，也不应该拼一大堆业务流程。

为什么？因为路由的本质是“入口”。入口应该轻，才能稳定。

如果你把业务逻辑写到路由里，短期看起来快，长期一定会出现两个问题：

  1. 后面新增第二个入口时，逻辑没法复用。
  2. 测试和调试时，你很难看清问题出在哪一层。

4.2 Service 层为什么是大脑

service 层是真正处理业务规则的地方，它关心的是“这件事应该怎么做”。

比如聊天这个动作，service 会关心：

  1. 老人档案存不存在。
  2. 会话要不要自动创建。
  3. 消息先存再调模型，还是先调模型再存。
  4. 历史上下文怎么拿。

这些问题都不是路由层应该回答的，也不是 repo 层应该回答的。

所以你可以把 service 理解为：

service 决定“业务怎么走”，repo 决定“数据怎么拿”。

4.3 Repository 层为什么值得单独抽

有些小项目确实可以不抽 repo，service 里直接写数据库查询。但你的项目后面会越来越复杂，尤其是聊天、触发规则、日志查询会不断增加。

这时 repo 的价值就很明显：

  1. 把数据库查询细节收起来。
  2. 让 service 看起来更接近业务语义。
  3. 后面改数据库实现时，影响范围更小。

比如 “查询某位老人最新一条血压记录” 这种事情，本质上是数据获取逻辑，应该归 repo。

4.4 Schema 层为什么不能省

很多新手嫌 schema 麻烦，觉得直接用字典不就行了。这样做一开始省事，但很快会出问题。

schema 至少在做三件有价值的事：

  1. 约束输入，避免接口参数混乱。
  2. 定义输出，保持前后端联调稳定。
  3. 让接口文档可读，便于测试和答辩展示。

你要把 schema 理解成“接口合同”。没有合同，团队协作和后续维护都很痛苦。

5. 五条关键业务链路怎么理解

你的后端不是很多独立模块，而是由几条链路串起来的。理解链路，比理解单个文件更重要。

5.1 档案链路

这条链路是最基础的：

前端提交档案信息 -> 后端校验 -> 持久化保存 -> 返回老人对象

这条链路的思维重点不在技术，而在业务中心确定。

只有老人档案稳定存在，后面的记录、对话、主动关怀才有归属对象。否则你的整个系统就是漂浮的。

5.2 记录链路

健康记录链路的本质是：让系统拥有“最近发生了什么”的事实依据。

聊天为什么能给出健康场景建议？因为它能看到记录。

主动触发为什么能判断异常？因为它能看到记录。

所以 record 链路并不是辅助模块，它是在给系统喂事实数据。

5.3 聊天链路

聊天链路要分成两层理解。

表面动作

用户发一句话，系统回一句话。

真正链路

确认老人身份 -> 确认会话 -> 保存用户消息 -> 组装上下文 -> 调模型 -> 保存助手消息 -> 返回结果。

也就是说，聊天的核心不是“调大模型”，而是“把调模型放进一条完整的业务链路里”。

这一点特别关键，因为很多人误以为接上模型 API 就等于做了智能后端。其实不是。真正的难点是让模型和业务数据发生关系。

5.4 主动触发链路

主动触发链路可以拆成四个问题：

  1. 该不该触发？
  2. 为什么触发？
  3. 触发后说什么？
  4. 触发结果怎么记录？

这四个问题如果混在一起写，你的代码会很快失控。所以一定要拆开：

  • trigger_service 负责“该不该触发”。
  • proactive_service 负责“触发后做什么”。
  • proactive_log 负责“留下证据”。

你会发现，这其实是在把一个复杂动作拆成可解释的多个环节。

5.5 推送链路

很多项目做到主动消息生成这一步就停了，但那样还不能叫“主动交互”。因为如果消息只存在数据库里，用户根本感知不到系统主动性。

所以最终链路必须接到推送：

系统判定应该主动关怀 -> 生成消息 -> 保存消息 -> 通过 WebSocket 推给前端

这条链路的价值不是技术炫耀，而是把“系统主动性”变成用户可见行为。

6. 几个关键抽象为什么要提前做

你这个项目里有几个抽象，初学者常常会觉得“是不是太早了”。其实这些抽象都不是多余设计，而是在提前避免混乱。

抽象解决的问题为什么早做更好
llm_service统一模型调用入口以后更换模型供应商不会影响业务层
memory_service统一组装上下文聊天和主动生成都可能复用
trigger_service统一判断是否主动触发避免把规则写散在各处
proactive_service统一执行主动消息流程手动触发和定时触发都能复用
ws manager统一管理连接和推送后面一个用户多端连接也好扩展
scheduler统一周期任务入口主动检查逻辑不用散在 main 里

判断一个抽象是否值得做，有一个很好用的标准：

只要同一类逻辑未来有两个以上调用入口，就应该考虑抽出来。

比如主动触发既可以手动测试触发，也可以定时任务触发，所以它天然值得抽成服务。

7. 为什么开发顺序必须这样排

你现在最需要避免的一件事，就是按照“哪个看起来酷就先写哪个”的顺序开发。

正确顺序应该是：

  1. 先把 profile 做好。
  2. 再把 record 做好。
  3. 再补 conversation 和 message。
  4. 再让聊天链路落地。
  5. 再做主动窗口和主动日志。
  6. 再做触发服务和主动服务。
  7. 最后才做 WebSocket 和 scheduler。

这个顺序背后其实是一条非常清楚的依赖关系：

后做的模块依赖什么
聊天依赖 profile、conversation、message
主动触发依赖 profile、record、message
WebSocket 推送依赖主动消息已经能生成
调度器依赖主动服务已经稳定可复用

如果你逆着依赖顺序写，会不断返工。比如还没做好主动服务就上调度器，最后调度器里全是临时逻辑，后面一定重写。

8. 你最容易犯的思维错误

错误 1：把“接上模型”当成智能系统的核心

模型只是能力来源，不是系统本身。系统真正的价值来自它能不能结合老人数据、聊天历史和主动规则形成闭环。

错误 2：把“表设计很细”当成架构成熟

第一版最重要的是主链路跑通，而不是把每种记录都拆成很多表。

错误 3：把所有逻辑写进路由

这样短期快，长期必乱。路由应该越轻越好。

错误 4：主动触发只做“发消息”，不做“说明原因”

没有日志和原因记录，答辩时你解释不清触发逻辑。

错误 5：过早追求自动化

手动触发接口还没稳，就先做调度器和完整部署，这会显著增加调试成本。

你可以记住一句特别实用的话：

先把系统做对，再把系统做炫。

9. 两个版本怎么对照学习

你现在手里应该有两份教程：

  1. 完整代码版：告诉你文件里具体写什么。
  2. 纯思路教学版：告诉你为什么要这么写。

最有效的学习方法不是先把一份看完再看另一份，而是按模块对照：

学习顺序先看什么再看什么目的
Profile思路版里 profile 为什么是中心代码版里 profile 的 model/schema/service/route理解“主实体”如何落地
Record思路版里为什么先做通用记录表代码版里 record 的增查逻辑理解“阶段性最优解”
Chat思路版里聊天的完整业务链路代码版里 chat_service、message、conversation理解“会话式系统”而不是单接口
Proactive思路版里主动触发四个问题代码版里 trigger_service、proactive_service理解“规则判断”和“执行流程”的分离
Push思路版里为什么需要推送代码版里 websocket 和 scheduler理解主动交互闭环

你学习时最好始终问自己两个问题：

  1. 这个模块在系统里解决的是什么问题？
  2. 代码版里哪些文件是在实现这个问题的解决方案？

如果你能把这两个问题答出来，你就不是在抄代码，而是在理解系统。

10. 学完这两个版本你应该达到什么程度

学完之后，你不应该只是“会写一个当前版本”。你至少应该具备下面这些能力：

  1. 知道为什么业务主实体是 profile。
  2. 知道为什么健康记录第一版应该做成通用表。
  3. 知道为什么聊天一定要有会话和消息持久化。
  4. 知道为什么主动功能一定要拆成“判断”和“执行”两部分。
  5. 知道为什么路由不应该承载业务复杂度。
  6. 知道为什么 WebSocket 和 scheduler 一定是靠后的阶段性能力。
  7. 面对新需求时，能判断它应该落在哪一层、哪一类表、哪一条链路上。
你最终学会的应该不是“这一套代码”，而是“以后遇到类似项目时，也能自己搭出一套合理后端”的思考方式。

如果你还要继续，我下一步最适合给你补的是一份“对照学习导航版”，把这份思路版的每个章节，精确对应到完整代码版中的相关章节和文件路径，让你复习时更快来回切换。
