Healthy System FastAPI 第七阶段详细教程

1. 这一阶段到底要解决什么

第七阶段正式把系统从：

“会判断要不要主动”

推进到：

“会真正生成一条主动内容”

也就是说，第六阶段你已经有了：

  1. TriggerRule
  2. ActiveLog
  3. trigger_rule_service.evaluate_for_user()

第七阶段要做的，不是重写一套判断逻辑，而是在这个基础上再补两层：

  1. ProactiveWindow
  2. ProactiveMessage

一句话理解第七阶段：

先让系统在“合适时间”里，根据“已命中的规则”，生成一条真正可展示的主动消息。

2. 第七阶段完成后的验收标准

做到下面这些，就算第七阶段合格：

  1. 系统可根据规则生成主动消息
  2. 系统能限制主动打扰时间
  3. 系统能限制同一天主动频率
  4. 主动消息可以单独查询
  5. 主动消息可以标记为已展示
  6. 主动执行过程可以继续写入 active_log

3. 为什么第七阶段不是重写规则，而是复用第六阶段

因为第六阶段已经把最核心的一层做好了：

trigger_rule_service.evaluate_for_user()

它已经能回答：

  1. 这条规则有没有命中
  2. 为什么命中
  3. 为什么没命中

第七阶段真正新增的判断，不是“规则判断”，而是另外两层：

  1. 当前是不是允许主动打扰
  2. 今天是不是已经主动太多次了

所以第七阶段最合理的主链路应该是：

  1. 先复用第六阶段规则评估
  2. 再检查主动窗口
  3. 再检查频率限制
  4. 最后生成主动消息
  5. 全程继续写主动日志

这样前后阶段才是衔接的。

4. 这一阶段默认建立在前面哪些东西已经完成的前提上

第七阶段默认你前面至少已经有：

  1. 第二阶段的 User / Auth / Profile
  2. 第三阶段的 Record
  3. 第四阶段的 Conversation / Message
  4. 第五阶段的聊天主链路
  5. 第六阶段的 TriggerRule / ActiveLog

为什么：

因为主动服务不是孤立模块。

它依赖：

  1. 用户身份，决定消息归谁
  2. 规则评估，决定为什么触发
  3. 日志系统，决定能不能追踪
  4. 现有消息体系，决定后面怎么接前端展示或 WebSocket

5. 先说明你当前项目里的主动相关草稿

你仓库里现在已经有这些和主动能力相关的草稿文件：

  1. backend/app/db/models/proactive_window.py
  2. backend/app/db/models/proactive_log.py
  3. backend/app/schemas/proactive.py

另外，从仓库里残留的编译文件也能看出来，你项目原本已经朝下面这套结构走过：

  1. ProactiveWindow
  2. ProactiveMessage
  3. ProactiveService
  4. proactive.py 路由

所以第七阶段最自然的正式落地方向，不是继续扩 app/db/models/proactive_* 草稿，而是统一回你当前正式工程结构：

  1. ORM 模型放 app/models/
  2. schema 放 app/schemas/
  3. service 放 app/services/
  4. route 放 app/api/routes/

6. 为什么这一阶段我建议用 ProactiveMessage 表，而不是直接写进 message

你的开发计划里给了两个最小版本：

  1. 在 conversation 里插入一条系统主动消息
  2. 在主动消息表中生成待展示内容

这两个方案都能成立。

但基于你当前项目的阶段衔接，我更建议第七阶段先选第二种：

先做 ProactiveMessage

理由很简单：

  1. 这和第六阶段文档结尾提到的 ProactiveWindow + ProactiveMessage 能直接对上
  2. 这样前端可以单独拉取“待展示的主动消息”，演示更清楚
  3. 还可以顺手支持 pending / displayed 状态，比直接塞聊天记录更好控
  4. 第八阶段接 WebSocket 时，也更容易把“待展示主动消息”实时推给前端

这不代表以后不能写进 messages 表。

只是第七阶段先把“主动生成 + 主动窗口 + 主动频率 + 主动日志”这条链路做稳。

7. 第七阶段推荐的数据设计

7.1 proactive_windows

这张表的作用不是存规则，而是存：

“当前用户允许被主动打扰的边界”

建议字段：

字段 类型 说明
id int 主键
user_id int 一个用户一条窗口配置
enabled bool 是否启用主动触达
quiet_hours_start str 安静时间起点
quiet_hours_end str 安静时间终点
max_trigger_per_day int 每天最多主动几次
created_at datetime 创建时间
updated_at datetime 更新时间

这里要特别注意：

quiet_hours_start 和 quiet_hours_end 表示的是“不要打扰”的时间段。

比如默认：

  1. quiet_hours_start = "22:00"
  2. quiet_hours_end = "08:00"

它表示：

晚上 22:00 到第二天 08:00 之间不要主动触达。

换句话说，真正允许主动触达的时间，就是：

08:00 - 22:00

这正好对应你开发计划里的阶段目标。

7.2 proactive_messages

这张表的作用是：

真正把主动生成的内容先存下来，交给前端展示。

建议字段：

字段 类型 说明
id int 主键
user_id int 这条主动消息属于谁
trigger_rule_id int | null 来源规则
trigger_type str 触发类型
title str 主动消息标题
content str 主动消息正文
status str pending / displayed
created_at datetime 生成时间
displayed_at datetime | null 展示时间

为什么这里要单独有 status：

因为第七阶段不只是“生成消息”，还要回答：

  1. 这条主动消息有没有被前端展示
  2. 这条主动消息是不是还处于待展示状态
  3. 后面要不要通过 WebSocket 推送

如果没有这个状态，前端联调会比较乱。

8. 这一阶段还会复用哪些旧模块

第七阶段你不需要重写下面这些文件：

12. 新建 ProactiveWindow ORM 模型

12.1 新建 app/models/proactive_window.py

完整代码如下：

from __future__ import annotations

from datetime import UTC, datetime

from sqlalchemy import Boolean, DateTime, ForeignKey, Integer, String
from sqlalchemy.orm import Mapped, mapped_column

from app.db.session import Base


class ProactiveWindow(Base):
    __tablename__ = "proactive_windows"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, index=True)
    user_id: Mapped[int] = mapped_column(
        ForeignKey("users.id", ondelete="CASCADE"),
        unique=True,
        nullable=False,
        index=True,
    )
    enabled: Mapped[bool] = mapped_column(Boolean, nullable=False, default=True)
    quiet_hours_start: Mapped[str] = mapped_column(
        String(5),
        nullable=False,
        default="22:00",
    )
    quiet_hours_end: Mapped[str] = mapped_column(
        String(5),
        nullable=False,
        default="08:00",
    )
    max_trigger_per_day: Mapped[int] = mapped_column(
        Integer,
        nullable=False,
        default=1,
    )
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        default=lambda: datetime.now(UTC),
        nullable=False,
    )
    updated_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        default=lambda: datetime.now(UTC),
        onupdate=lambda: datetime.now(UTC),
        nullable=False,
    )

12.2 这份模型代码在做什么

  1. 一个用户只允许有一条主动窗口配置，所以 user_id 做了唯一限制
  2. enabled 决定该用户整体是否允许主动触达
  3. quiet_hours_start 和 quiet_hours_end 定义安静时段
  4. max_trigger_per_day 定义每天最多主动几次

12.3 为什么这里改成 user_id，而不是草稿里的 profile_id

因为你当前正式后端主链路已经基本统一成：

  1. 会话按 user_id
  2. 记录按 user_id
  3. 主动日志也按 user_id

所以第七阶段继续统一成 user_id 会更顺。

否则你会在主动模块里突然切回 profile_id，前后阶段会不一致。

13. 新建 ProactiveMessage ORM 模型

13.1 新建 app/models/proactive_message.py

完整代码如下：

from __future__ import annotations

from datetime import UTC, datetime

from sqlalchemy import DateTime, ForeignKey, Integer, String, Text
from sqlalchemy.orm import Mapped, mapped_column

from app.db.session import Base


class ProactiveMessage(Base):
    __tablename__ = "proactive_messages"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, index=True)
    user_id: Mapped[int] = mapped_column(
        ForeignKey("users.id", ondelete="CASCADE"),
        nullable=False,
        index=True,
    )
    trigger_rule_id: Mapped[int | None] = mapped_column(
        ForeignKey("trigger_rules.id", ondelete="SET NULL"),
        nullable=True,
        index=True,
    )
    trigger_type: Mapped[str] = mapped_column(String(50), nullable=False)
    title: Mapped[str] = mapped_column(String(100), nullable=False)
    content: Mapped[str] = mapped_column(Text, nullable=False)
    status: Mapped[str] = mapped_column(
        String(20),
        nullable=False,
        default="pending",
        index=True,
    )
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        default=lambda: datetime.now(UTC),
        nullable=False,
        index=True,
    )
    displayed_at: Mapped[datetime | None] = mapped_column(
        DateTime(timezone=True),
        nullable=True,
    )

13.2 这份模型代码在做什么

  1. user_id 说明这条主动消息属于谁
  2. trigger_rule_id 表示这条消息来源于哪条规则
  3. trigger_type 方便后面做分类展示
  4. title + content 是最终给前端展示的内容
  5. status 用来区分待展示还是已展示
  6. displayed_at 用来记前端真正看到的时间

13.3 为什么 trigger_rule_id 允许为空

因为主动消息是历史数据。

如果以后某条规则被删除了，你不应该把历史主动消息一起删没。

所以这里更合理的做法是：

  1. 规则删了
  2. trigger_rule_id 置空
  3. 主动消息历史继续保留

这就是这里用 ondelete="SET NULL" 的原因。

from datetime import datetime

from pydantic import BaseModel, ConfigDict, Field

TIME_PATTERN = r"^\d{2}:\d{2}$"


class ProactiveWindowUpdate(BaseModel):

    quiet_hours_end: str = Field(default="08:00", pattern=TIME_PATTERN)

class ProactiveWindowRead(ProactiveWindowUpdate):
    id: int
    user_id: int
    created_at: datetime
    updated_at: datetime

    model_config = ConfigDict(from_attributes=True)


class ProactiveMessageRead(BaseModel):
    id: int
    user_id: int
    trigger_rule_id: int | None = None
    trigger_type: str
    title: str
    content: str
    status: str
    created_at: datetime
    displayed_at: datetime | None = None

    model_config = ConfigDict(from_attributes=True)


class ProactiveMessageListData(BaseModel):
    items: list[ProactiveMessageRead]


class ProactiveExecuteData(BaseModel):

    rule_id: int
    triggered: bool
    message_created: bool
    status: str
    reason: str
    proactive_message: ProactiveMessageRead | None = None

15.2 这份 schema 在做什么

它主要定义了三类输入输出：

  1. 主动窗口更新
  2. 主动消息读取
  3. 主动执行结果返回

15.3 为什么 ProactiveWindowRead 继承 ProactiveWindowUpdate

因为这两个结构几乎一样：

  1. enabled
  2. quiet_hours_start
  3. quiet_hours_end
  4. max_trigger_per_day

读取时只是额外多了：

  1. id
  2. user_id
  3. created_at
  4. updated_at

这种场景很适合继承，代码更简洁。

15.4 为什么时间字段这里只校验 HH:MM

第七阶段先做最小可演示版本。

所以时间字符串先约束成：

HH:MM

就足够了。

后面如果你要做用户时区、星期几、节假日、场景白名单，再继续扩也不迟。

16. 写 ProactiveWindowService

这一层负责三件事：

  1. 给用户拿默认主动窗口
  2. 更新主动窗口配置
  3. 判断当前时刻是否允许主动触达

16.1 新建 app/services/proactive_window_service.py

完整代码如下：

from __future__ import annotations

from datetime import datetime, time

from sqlalchemy import select
from sqlalchemy.orm import Session

from app.core.exception import BusinessException
from app.models.proactive_window import ProactiveWindow
from app.schemas.proactive import ProactiveWindowUpdate


class ProactiveWindowService:
    def get_or_create_for_user(
        self,
        db: Session,
        user_id: int,
    ) -> ProactiveWindow:
        window = self._get_by_user_id(db, user_id)
        if window:
            return window

        window = ProactiveWindow(
            user_id=user_id,
            enabled=True,
            quiet_hours_start="22:00",
            quiet_hours_end="08:00",
            max_trigger_per_day=1,
        )
        db.add(window)
        db.commit()
        db.refresh(window)
        return window

    def update_for_user(
        self,
        db: Session,
        user_id: int,
        payload: ProactiveWindowUpdate,
    ) -> ProactiveWindow:
        self._validate_time_range(
            payload.quiet_hours_start,
            payload.quiet_hours_end,
        )

        window = self.get_or_create_for_user(db, user_id)
        window.enabled = payload.enabled
        window.quiet_hours_start = payload.quiet_hours_start
        window.quiet_hours_end = payload.quiet_hours_end
        window.max_trigger_per_day = payload.max_trigger_per_day

        db.add(window)
        db.commit()
        db.refresh(window)
        return window

    def allow_trigger_now(
        self,
        window: ProactiveWindow,
        now: datetime | None = None,
    ) -> tuple[bool, str]:
        if not window.enabled:
            return False, "proactive window is disabled"

        current_time = (now or datetime.now()).time().replace(
            second=0,
            microsecond=0,
        )
        quiet_start = self._parse_time_text(window.quiet_hours_start)
        quiet_end = self._parse_time_text(window.quiet_hours_end)

        if self._is_in_quiet_hours(current_time, quiet_start, quiet_end):
            return False, "current time is inside quiet hours"

        return True, "allowed"

    def _get_by_user_id(
        self,
        db: Session,
        user_id: int,
    ) -> ProactiveWindow | None:
        stmt = select(ProactiveWindow).where(ProactiveWindow.user_id == user_id)
        return db.scalar(stmt)

    def _validate_time_range(
        self,
        start_text: str,
        end_text: str,
    ) -> None:
        self._parse_time_text(start_text)
        self._parse_time_text(end_text)

        if start_text == end_text:
            raise BusinessException(
                code=40071,
                message="quiet_hours_start cannot be equal to quiet_hours_end",
                status_code=400,
            )

    def _parse_time_text(self, value: str) -> time:
        try:
            return datetime.strptime(value, "%H:%M").time()
        except ValueError as exc:
            raise BusinessException(
                code=40072,
                message="time format must be HH:MM",
                status_code=400,
            ) from exc

    def _is_in_quiet_hours(
        self,
        current_time: time,
        quiet_start: time,
        quiet_end: time,
    ) -> bool:
        if quiet_start < quiet_end:
            return quiet_start <= current_time < quiet_end

        return current_time >= quiet_start or current_time < quiet_end


proactive_window_service = ProactiveWindowService()

16.2 这份窗口服务的关键点

1. 没配置也能先跑起来

get_or_create_for_user() 会自动给当前用户补一份默认窗口配置。

默认值就是：

  1. 启用主动
  2. 安静时段 22:00 - 08:00
  3. 每天最多一次

这非常适合比赛演示，因为你不需要先手工插库。

2. 安静时段支持跨天

这一点很重要。

比如：

  1. 22:00 -> 08:00

它不是普通的同一天区间，而是跨天区间。

所以 _is_in_quiet_hours() 要同时兼容两种情况：

  1. 12:00 -> 14:00 这种不跨天
  2. 22:00 -> 08:00 这种跨天

3. 这里先按服务端本地时间判断

你会看到这里用的是：

datetime.now()

不是用户时区。

这是第七阶段有意为之的简化。

因为你当前项目还没有“用户时区配置”这一层。

比赛版先按服务部署机器的本地时间判断，完全够用。

后面如果你真的要做更完整的产品，再加：

  1. 用户时区字段
  2. 本地化 quiet hours 计算

完整代码如下：

from __future__ import annotations

from datetime import UTC, datetime

from sqlalchemy import select
from sqlalchemy.orm import Session

from app.core.exception import BusinessException
from app.models.proactive_message import ProactiveMessage
from app.models.trigger_rule import TriggerRule


class ProactiveMessageService:
    def create_message(
        self,
        db: Session,
        user_id: int,
        trigger_rule: TriggerRule,
        title: str,
        content: str,
    ) -> ProactiveMessage:
        message = ProactiveMessage(
            user_id=user_id,
            trigger_rule_id=trigger_rule.id,
            trigger_type=trigger_rule.trigger_type,
            title=title,
            content=content,
            status="pending",
        )
        db.add(message)
        db.commit()
        db.refresh(message)
        return message

    def list_for_user(
        self,
        db: Session,
        user_id: int,
    ) -> list[ProactiveMessage]:
        stmt = (
            select(ProactiveMessage)
            .where(ProactiveMessage.user_id == user_id)
            .order_by(
                ProactiveMessage.created_at.desc(),
                ProactiveMessage.id.desc(),
            )
        )
        return list(db.scalars(stmt).all())

    def get_or_raise(
        self,
        db: Session,
        user_id: int,
        message_id: int,
    ) -> ProactiveMessage:
        stmt = select(ProactiveMessage).where(
            ProactiveMessage.id == message_id,
            ProactiveMessage.user_id == user_id,
        )
        message = db.scalar(stmt)
        if not message:
            raise BusinessException(
                code=40471,
                message="proactive message not found",
                status_code=404,
            )
        return message

    def mark_displayed(
        self,
        db: Session,
        user_id: int,
        message_id: int,
    ) -> ProactiveMessage:
        message = self.get_or_raise(db, user_id, message_id)

        if message.status != "displayed":
            message.status = "displayed"
            message.displayed_at = datetime.now(UTC)
            db.add(message)
            db.commit()
            db.refresh(message)

        return message


proactive_message_service = ProactiveMessageService()

1. 消息列表必须按用户隔离

这和主动日志一样，必须限制成：

2. mark_displayed() 要做成幂等

完整代码如下：

from __future__ import annotations

from datetime import UTC, datetime
from typing import Any

from sqlalchemy import func, select
from sqlalchemy.orm import Session

from app.models.active_log import ActiveLog
from app.models.trigger_rule import TriggerRule
from app.services.active_log_service import active_log_service
from app.services.proactive_message_service import proactive_message_service
from app.services.proactive_window_service import proactive_window_service
from app.services.trigger_rule_service import trigger_rule_service


class ProactiveService:
    def execute_rule_for_user(
        self,
        db: Session,
        user_id: int,
        rule_id: int,
    ) -> dict[str, Any]:
        rule = trigger_rule_service.get_or_raise(db, rule_id)
        request_payload = {
            "rule_id": rule.id,
            "trigger_type": rule.trigger_type,
            "condition_json": rule.condition_json,
            "executed_at": datetime.now(UTC).isoformat(),
        }

        try:
            evaluation = trigger_rule_service.evaluate_for_user(db, user_id, rule)

            if not evaluation["triggered"]:
                log = active_log_service.create_log(
                    db=db,
                    user_id=user_id,
                    trigger_rule_id=rule.id,
                    action_type="generate_proactive_message",
                    status="skipped",
                    request_payload=request_payload,
                    response_payload=evaluation,
                )
                return {
                    "log_id": log.id,
                    "rule_id": rule.id,
                    "triggered": False,
                    "message_created": False,
                    "status": "skipped",
                    "reason": evaluation["reason"],
                    "proactive_message": None,
                }

            window = proactive_window_service.get_or_create_for_user(db, user_id)
            allowed, window_reason = proactive_window_service.allow_trigger_now(window)

            if not allowed:
                response_payload = {
                    **evaluation,
                    "window_reason": window_reason,
                    "quiet_hours_start": window.quiet_hours_start,
                    "quiet_hours_end": window.quiet_hours_end,
                }
                log = active_log_service.create_log(
                    db=db,
                    user_id=user_id,
                    trigger_rule_id=rule.id,
                    action_type="generate_proactive_message",
                    status="blocked",
                    request_payload=request_payload,
                    response_payload=response_payload,
                )
                return {
                    "log_id": log.id,
                    "rule_id": rule.id,
                    "triggered": True,
                    "message_created": False,
                    "status": "blocked",
                    "reason": window_reason,
                    "proactive_message": None,
                }

            today_count = self.count_today_created(db, user_id)
            if today_count >= window.max_trigger_per_day:
                response_payload = {
                    **evaluation,
                    "today_count": today_count,
                    "max_trigger_per_day": window.max_trigger_per_day,
                }
                log = active_log_service.create_log(
                    db=db,
                    user_id=user_id,
                    trigger_rule_id=rule.id,
                    action_type="generate_proactive_message",
                    status="rate_limited",
                    request_payload=request_payload,
                    response_payload=response_payload,
                )
                return {
                    "log_id": log.id,
                    "rule_id": rule.id,
                    "triggered": True,
                    "message_created": False,
                    "status": "rate_limited",
                    "reason": "max trigger per day exceeded",
                    "proactive_message": None,
                }

            title, content = self.build_message_content(rule, evaluation)
            proactive_message = proactive_message_service.create_message(
                db=db,
                user_id=user_id,
                trigger_rule=rule,
                title=title,
                content=content,
            )

            response_payload = {
                **evaluation,
                "proactive_message_id": proactive_message.id,
                "title": proactive_message.title,
                "status": proactive_message.status,
            }
            log = active_log_service.create_log(
                db=db,
                user_id=user_id,
                trigger_rule_id=rule.id,
                action_type="generate_proactive_message",
                status="created",
                request_payload=request_payload,
                response_payload=response_payload,
            )
            return {
                "log_id": log.id,
                "rule_id": rule.id,
                "triggered": True,
                "message_created": True,
                "status": "created",
                "reason": evaluation["reason"],
                "proactive_message": proactive_message,
            }
        except Exception as exc:
            active_log_service.create_log(
                db=db,
                user_id=user_id,
                trigger_rule_id=rule.id,
                action_type="generate_proactive_message",
                status="failed",
                request_payload=request_payload,
                response_payload={"error": str(exc)},
            )
            raise

    def count_today_created(
        self,
        db: Session,
        user_id: int,
    ) -> int:
        day_start = datetime.now(UTC).replace(
            hour=0,
            minute=0,
            second=0,
            microsecond=0,
        )

        stmt = (
            select(func.count())
            .select_from(ActiveLog)
            .where(
                ActiveLog.user_id == user_id,
                ActiveLog.action_type == "generate_proactive_message",
                ActiveLog.status == "created",
                ActiveLog.created_at >= day_start,
            )
        )
        return db.scalar(stmt) or 0

    def build_message_content(
        self,
        rule: TriggerRule,
        evaluation: dict[str, Any],
    ) -> tuple[str, str]:
        if rule.trigger_type == "days_without_record":
            return self._build_days_without_record_message(evaluation)

        if rule.trigger_type == "recent_chat_keyword":
            return self._build_recent_chat_keyword_message(evaluation)

        return (
            "主动关怀提醒",
            "系统检测到你可能需要一次健康关注，你可以打开应用查看建议。",
        )

    def _build_days_without_record_message(
        self,
        evaluation: dict[str, Any],
    ) -> tuple[str, str]:
        record_type = str(evaluation.get("record_type", "健康数据"))
        days_without_record = int(evaluation.get("days_without_record", 1))
        record_label = self._record_type_label(record_type)

        title = "健康记录提醒"
        content = (
            f"我注意到你已经 {days_without_record} 天没有记录{record_label}了。"
            "如果今天方便，可以补记一条；如果最近状态有波动，也建议尽快测一次。"
        )
        return title, content

    def _build_recent_chat_keyword_message(
        self,
        evaluation: dict[str, Any],
    ) -> tuple[str, str]:
        keywords = evaluation.get("matched_keywords") or []
        keyword_text = "、".join(str(item) for item in keywords) or "当前困扰"

        title = "健康关怀提示"
        content = (
            f"我注意到你最近的表达里出现了“{keyword_text}”相关内容。"
            "如果你愿意，可以继续说说现在最困扰你的点，我会先帮你整理成几个可执行的小建议。"
        )
        return title, content

    def _record_type_label(self, record_type: str) -> str:
        mapping = {
            "blood_pressure": "血压",
            "blood_glucose": "血糖",
            "sleep": "睡眠",
            "medication": "用药",
            "mood": "情绪",
            "checkin": "健康打卡",
        }
        return mapping.get(record_type, record_type)


proactive_service = ProactiveService()

18.2 这份服务最核心的设计是什么

核心就是把一次主动执行拆成 4 个结果分支：

  1. skipped
  2. blocked
  3. rate_limited
  4. created

再加一个异常时的：

  1. failed

1. skipped

规则根本没命中。

比如：

用户最近明明就有血压记录。

那这一轮主动执行就应该到此结束，但仍然要写日志。

2. blocked

规则命中了，但当前在安静时段，或者窗口被关闭。

比如：

  1. 命中了“3 天未记录血压”
  2. 但现在是凌晨 1 点

这时不能硬发消息，而是要记成：

blocked

3. rate_limited

规则命中了，也不在安静时段，但今天已经主动够次数了。

这时也不能继续发。

4. created

规则命中了，时间允许，频率也没超。

这时才真正创建 ProactiveMessage。

18.3 为什么频率控制是查 ActiveLog

因为第七阶段要限制的是：

“今天成功执行了多少次主动动作”

而不是：

“消息表里有多少条记录”

最稳的统计口径就是查日志里：

  1. action_type = "generate_proactive_message"
  2. status = "created"

这样以后哪怕你后面把主动消息同步到会话表、WebSocket、通知中心，频率口径也不会乱。

18.4 为什么这里没有直接调大模型生成文案

第七阶段先不要急着把主动文案做成 LLM 版本。

现在最重要的是把：

  1. 规则
  2. 窗口
  3. 频率
  4. 留痕
  5. 持久化

这几层先跑通。

所以这里先用模板式文案是最稳的。

到第八阶段你再决定：

  1. 用 WebSocket 推送
  2. 用 LLM 润色文案
  3. 或者生成更个性化内容

都会简单很多。

    "/messages",
    response_model=ApiResponse[ProactiveMessageListData],

    return success_response(
        data=ProactiveMessageListData(
            items=[ProactiveMessageRead.model_validate(item) for item in messages]
        ),
        message="success",
    )


@router.patch(
    "/messages/{message_id}/displayed",
    response_model=ApiResponse[ProactiveMessageRead],
)

    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_active_user),

        db=db,
        user_id=current_user.id,
        message_id=message_id,

        message="proactive message displayed",
    )

    "/rules/{rule_id}/execute",
    response_model=ApiResponse[ProactiveExecuteData],
)

            proactive_message=(

            ),

    )

22. 这条主动执行链路到底是怎么跑的

这里建议你把整条链路背下来，因为答辩时非常好讲。

22.1 先看主流程

POST /api/v1/proactive/rules/{rule_id}/execute

进入后，后端会按下面顺序处理：

  1. 读取规则
  2. 调用 trigger_rule_service.evaluate_for_user()
  3. 如果规则没命中，写一条 skipped 日志并返回
  4. 如果规则命中，读取或创建用户的 ProactiveWindow
  5. 检查当前时间是否在 quiet hours 内
  6. 如果在 quiet hours 内，写一条 blocked 日志并返回
  7. 再检查今天成功主动了几次
  8. 如果次数超限，写一条 rate_limited 日志并返回
  9. 如果都通过，就生成 title + content
  10. 写入 ProactiveMessage
  11. 写一条 created 日志
  12. 返回主动消息给前端

22.2 这里和第六阶段的职责边界怎么区分

第六阶段负责：

  1. 规则是什么
  2. 规则有没有命中
  3. 为什么命中

第七阶段负责：

  1. 命中了之后允不允许真的主动
  2. 要不要限流
  3. 主动内容怎么落库
  4. 执行结果怎么留痕

这个边界非常清晰。

23. 这一阶段最容易写错的地方

23.1 把 quiet hours 当成 active hours

这是最容易反的。

这里的字段是：

  1. quiet_hours_start
  2. quiet_hours_end

表示的是“不要打扰”的时段。

默认 22:00 -> 08:00 的意思是：

晚上十点到早上八点别主动发。

23.2 频率控制查错表

这里应该查：

active_logs

而不是查 proactive_messages

因为你真正限制的是“成功执行次数”，不是消息记录条数。

23.3 忘了给用户自动创建默认窗口

如果没有 get_or_create_for_user()，你每次演示前都得先手动造一条窗口配置，流程会变得很笨。

23.4 规则没命中时不写日志

很多人会只在成功创建主动消息时写日志。

这是不对的。

因为后面你最常被问到的就是：

“为什么这次没主动？”

所以：

没命中也必须留痕。

23.5 把主动消息和主动日志混为一谈

这两者不是一回事：

  1. ProactiveMessage 是给前端展示的内容
  2. ActiveLog 是给系统审计的留痕

一个面向用户，一个面向系统。

24. Swagger 验收怎么做

这里建议你按下面顺序验收。

24.1 先确保第六阶段已有一条可用规则

比如你已经在第六阶段建过：

{
  "name": "三天未记录血压",
  "trigger_type": "days_without_record",
  "enabled": true,
  "condition_json": {
    "days_without_record": 3,
    "record_type": "blood_pressure"
  },
  "priority": 10
}

或者：

{
  "name": "最近聊天出现焦虑关键词",
  "trigger_type": "recent_chat_keyword",
  "enabled": true,
  "condition_json": {
    "keywords": ["焦虑", "害怕", "睡不着"],
    "recent_limit": 10
  },
  "priority": 20
}

记住这条规则的 rule_id。

24.2 查看默认主动窗口

先调用：

GET /api/v1/proactive/window

第一次调用时，如果数据库里还没有窗口配置，系统会自动创建一条默认数据。

你应该看到类似结果：

  1. enabled = true
  2. quiet_hours_start = "22:00"
  3. quiet_hours_end = "08:00"
  4. max_trigger_per_day = 1

这一步验的是：

窗口配置能自动初始化。

24.3 更新主动窗口

调用：

PUT /api/v1/proactive/window

请求体示例：

{
  "enabled": true,
  "quiet_hours_start": "23:00",
  "quiet_hours_end": "07:00",
  "max_trigger_per_day": 2
}

你应该确认：

  1. 返回内容已变
  2. 再次 GET /proactive/window 时还是这个值

这一步验的是：

主动窗口不是硬编码，而是可配置的。

24.4 手动执行主动规则

调用：

POST /api/v1/proactive/rules/{rule_id}/execute

这是第七阶段最关键的验收接口。

你至少要重点看返回里有没有：

  1. log_id
  2. rule_id
  3. triggered
  4. message_created
  5. status
  6. reason
  7. proactive_message

情况 A：规则没命中

你应该看到：

  1. triggered = false
  2. message_created = false
  3. status = "skipped"
  4. proactive_message = null

情况 B：规则命中，但当前处于安静时段

你应该看到：

  1. triggered = true
  2. message_created = false
  3. status = "blocked"
  4. reason 里说明窗口原因

情况 C：规则命中，但当天已达上限

你应该看到：

  1. triggered = true
  2. message_created = false
  3. status = "rate_limited"

情况 D：规则命中，且真正生成主动消息

你应该看到：

  1. triggered = true
  2. message_created = true
  3. status = "created"
  4. proactive_message 里有完整消息内容

24.5 查看主动消息列表

调用：

GET /api/v1/proactive/messages

如果上一步已经成功创建了主动消息，这里就应该能看到。

你建议重点确认：

  1. user_id 是当前用户
  2. trigger_rule_id 正确
  3. trigger_type 正确
  4. title 和 content 已生成
  5. status = "pending"

24.6 标记主动消息已展示

取上一步拿到的 message_id，调用：

PATCH /api/v1/proactive/messages/{message_id}/displayed

你应该看到：

  1. status 变成 displayed
  2. displayed_at 已写入

这一步验的是：

主动消息不只是“生成了”，还能进入前端展示后的状态流转。

24.7 查看主动日志

再调用：

GET /api/v1/active-logs

你应该能看到本次执行产生的新日志。

建议重点确认：

  1. action_type = "generate_proactive_message"
  2. status 和执行结果一致
  3. request_payload 里有规则信息
  4. response_payload 里能看到为什么被跳过、阻塞、限流或创建

这一步非常关键，因为第七阶段虽然开始生成主动消息了，但它仍然是建立在第六阶段日志体系上的。

25. 做完这一阶段后，你现在真正具备了什么

如果你按这份教程完整补完，第七阶段结束后你已经具备：

  1. ProactiveWindow
  2. ProactiveMessage
  3. proactive_window_service
  4. proactive_message_service
  5. proactive_service.execute_rule_for_user()
  6. GET /api/v1/proactive/window
  7. PUT /api/v1/proactive/window
  8. GET /api/v1/proactive/messages
  9. PATCH /api/v1/proactive/messages/{message_id}/displayed
  10. POST /api/v1/proactive/rules/{rule_id}/execute

更重要的是，你已经把“主动服务”这件事拆成了一个清晰可答辩的闭环：

  1. 规则决定要不要主动
  2. 窗口决定能不能主动
  3. 频率决定要不要克制
  4. 主动消息决定给用户看什么
  5. 主动日志决定系统怎么追踪

这就是比赛里比较亮眼、但又不空的主动能力。

26. 第七阶段和第八阶段怎么衔接

做完这一阶段后，第八阶段就很好接了。

第八阶段你可以直接复用：

  1. proactive_service.execute_rule_for_user()
  2. ProactiveMessage
  3. ProactiveWindow
  4. ActiveLog

然后往上接两层增强：

  1. WebSocket 实时推送
  2. APScheduler 定时执行

也就是说：

第七阶段解决的是：

“主动服务能不能跑起来”

第八阶段解决的是：

“主动服务能不能自动跑、实时到达前端”

到这里，第七阶段才算真正和你现有项目代码、前六阶段教学文件顺利接上。
