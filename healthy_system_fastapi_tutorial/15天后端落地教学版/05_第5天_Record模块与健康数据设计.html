Day 5 / 15健康数据底座

第5天：Record 模块与健康数据设计

今天是整个系统“开始有事实依据”的一天。聊天为什么不是空聊，主动提醒为什么不是乱发，核心都取决于健康记录有没有设计好。

1. 今天要完成哪些文件

  • app/schemas/record.py
  • app/repositories/record_repo.py
  • app/services/record_service.py
  • app/api/routes/records.py
  • main.py 挂上 records_router

2. 今天和昨天最大的不同是什么

昨天的 Profile 是中心主体，今天的 Record 是围绕主体不断增长的动态事实。

也就是说：

  • profile 更像长期画像。
  • record 更像持续事件流。

所以它虽然也能做 CRUD，但你不能只是把它当成“另一张普通表”。

3. 为什么 Record 的输入结构和 Profile 不一样

Profile 大多是固定字段，Record 则有很多变化形态。比如：

血压：{ "systolic": 138, "diastolic": 91 }
睡眠：{ "hours": 6.5, "quality": "一般" }
运动：{ "duration": 30, "type": "walk" }

所以你需要接受一个事实：Record 的业务本质上比 Profile 更“异构”。

这就是为什么你用 record_type + value_json 的组合，而不是给所有数据都硬塞进一套固定列。

4. 代码层每一步在做什么

4.1 Schema：限定最小输入合同

在 schemas/record.py 里，你至少要定义：

  1. 这条记录属于哪个老人。
  2. 这条记录是什么类型。
  3. 具体值以什么形式提交。
  4. 是否允许前端自己传记录时间。

你会发现，Record 的 schema 不是在追求“固定格式”，而是在确保“再灵活也有边界”。

4.2 Repository：按老人查记录、按类型查最新记录

今天 repo 层有一个特别重要的新动作：不仅要能创建，还要能支持“按 profile 查记录”和“按类型查最新一条”。

原因很直接：

  • 记录列表页要按老人查。
  • 聊天上下文要拿最近健康记录。
  • 主动触发要拿最近血压或心率。

这说明 repo 里的查询方法，其实是在为后面功能提前铺路。

4.3 Service：先确认老人存在，再允许写记录

这是今天特别值得理解的一个业务动作：记录不能脱离老人独立创建。

所以在 Service 里，你会先查 Profile 是否存在，再决定要不要创建记录。这个顺序代表的是业务因果，而不是代码习惯。

4.4 Route：接口设计要更贴近使用场景

对于 Record，最基础的接口通常是：

  • POST /records：新增记录。
  • GET /records?profile_id=1：查看某位老人的记录。

注意这里的查询方式已经体现业务导向了：记录列表不是随便查，而是通常围绕某位老人查。

5. 今天你真正搭出来了什么能力

今天完成后，系统会第一次具备“可持续沉淀健康事实”的能力。这个能力会直接被后续两个模块拿去用：

后续模块如何使用 Record
聊天把最近健康数据带入上下文，生成更贴近场景的建议
主动触发根据异常血压、异常心率等数据决定是否主动提醒

6. 今天最容易错在哪

  • 把所有记录内容都设计成固定字段，导致后面扩展困难。
  • 只会新增记录，不会查询“某位老人的最新记录”。
  • 写记录之前没有验证 profile_id 是否存在。
  • 把 Record 当成第二个 Profile，只会做 CRUD，不会考虑它对后续业务的支撑作用。

7. 今天完成标准

  • 能给某位老人添加血压、心率或睡眠等记录。
  • 能查询某位老人的记录列表。
  • 你能说出为什么要支持“最新一条某类型记录”这种查询。
  • 你知道 Record 不只是存数据，而是在给聊天和主动提醒提供事实基础。

8. 明天衔接
