Day 3 / 15业务建模

第3天：数据建模，先把 Profile 和 Record 想明白

今天你做的不是“建两张表”这么简单。你其实是在决定系统以后围绕什么实体运转，以及健康数据要用什么方式被统一接住。

1. 今天要处理哪些文件

  • app/db/models/profile.py
  • app/db/models/record.py
  • 如果你的 main.py 还没引入模型并创建表，今天要顺手补上

2. 为什么先建这两个模型

因为你整个项目后面所有“看起来高级”的能力，本质上都依赖这两类数据：

  1. Profile：系统服务对象是谁。
  2. Record：这个对象最近发生了什么健康相关事件。

没有 profile，你的系统没有服务主体；没有 record，你的系统没有事实依据。

3. 为什么 profile 一定是中心实体

你后面会做很多功能：聊天、主动提醒、WebSocket 推送、会话历史、健康分析。看起来它们差异很大，但最后都必须回答同一个问题：

这条消息、这条记录、这次提醒，到底是给哪位老人准备的？

所以 profile 并不是普通用户表，而是整个系统的中心锚点。

它的字段不只是身份信息，还包括画像信息。例如：

字段你应该怎么理解它
name最基础的展示名，也是后面提示词里会用到的人称对象
age很多健康建议都要参考年龄
chronic_disease体现系统不是空聊天，而是带健康背景的辅助
medicine_note后续主动提醒和对话建议都可能用到
usual_sleep_time / usual_wake_time后续主动时间窗和生活建议可以参考
emergency_contact_*不是今天就用，但能体现可扩展性

4. 为什么 record 第一版先做通用表

这是今天最重要的设计理解之一。

理论上，血压、心率、血糖、睡眠、服药、运动都可以拆成单独表。但对于你当前阶段，最重要的不是极致规范，而是先把健康数据统一接住。

所以你用通用记录表，通常会保留这些字段：

  • profile_id：说明这条数据属于谁。
  • record_type：说明这是什么类型的数据。
  • value_json：保存具体内容。
  • source：说明来自手动录入、设备、系统生成等。
  • recorded_at：说明记录发生时间。

这套设计真正解决的是“异构健康数据统一落库”的问题。

你不是不知道拆细更规范，而是现在优先选择一个更适合比赛落地速度的结构。

5. 代码层每一步在做什么

5.1 主键不是装饰，是后面一切关联的基础

每个表里的 id 都不是例行公事。它直接决定：

  1. 消息能不能关联到会话。
  2. 记录能不能关联到老人。
  3. 主动日志能不能关联到触发对象。

5.2 时间字段不是为了好看，而是为了“最近”这个概念

你的聊天要拿最近消息，主动触发要看最近健康记录，记录列表要按时间排序，所以时间字段很关键。

如果没有这些时间字段，后面很多“上下文”“最近一次”“今天是否已提醒”都做不干净。

5.3 ForeignKey 体现的是业务依赖关系

record.profile_id 指向 profiles.id，不是数据库技巧，而是在表达一个业务事实：

健康记录不能脱离老人独立存在。

6. 今天最容易犯的错误

  • 把老人档案字段做得过于少，后面聊天和主动提醒拿不到画像信息。
  • 一上来把健康记录拆成很多表，结果开发节奏被拖慢。
  • 没有时间字段，后面所有“最近”“今日”相关逻辑都很难做。
  • 没有建立 profile_id 关联，只把记录当成孤立数据。

7. 今天完成标准

  • 你能清楚说出 profile 和 record 分别代表什么。
  • 你能解释为什么记录表第一版用通用结构。
  • 数据库里已经有这两张表的基础模型定义。
  • 你知道后面聊天和主动触发都会依赖这两张表。

8. 明天衔接

明天开始把 profile 从“模型”做成“完整业务模块”，也就是补上 schema、repo、service、route，让它真正能从前端请求一路走到数据库。
