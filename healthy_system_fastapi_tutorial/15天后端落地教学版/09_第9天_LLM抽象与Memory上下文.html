Day 9 / 15理解上下文

第9天：LLM 抽象与 Memory 上下文

今天不是单纯接模型，而是让“模型能力”和“你自己的业务数据”开始对接。也就是说，从今天起，聊天回复应该逐渐带上老人画像和健康记录的味道。

1. 今天要完成哪些文件

  • app/services/llm_service.py
  • app/services/memory_service.py
  • chat_service.py 中把这两个服务接进去

2. 为什么模型调用一定要抽成独立服务

如果你把模型调用代码直接写在聊天路由或聊天 service 里，短期看起来可以跑，长期一定会出现三个问题：

  1. 以后换模型供应商，改动范围很大。
  2. 调试时想用 Mock 回复，会很难切换。
  3. 主动消息生成也想复用模型时，代码会开始复制。

所以抽出 llm_service 的本质，是把“模型能力入口”统一起来。

3. 为什么要有 Mock 模型

这是很多人容易忽略，但特别实用的一点。

比赛版开发时，网络、密钥、额度、平台稳定性都可能影响联调。如果没有 Mock 版本，你的整个系统会被模型可用性绑架。

所以 Mock 的工程价值是：

  • 没配真实模型也能跑完整链路。
  • 你能先验证业务流程，再验证模型质量。
  • 答辩或演示前，遇到网络问题也不至于全盘失效。

4. MemoryService 真正在做什么

很多人会把“上下文”理解得很玄。其实你现在做的第一版上下文，非常朴素：

  1. 拿最近几条消息。
  2. 补上老人画像。
  3. 补上最近健康记录。
  4. 一起交给模型。

这就是滑动窗口上下文。

你现在不用一上来做向量库、长期记忆，因为第一版真正要解决的是：

让回复不脱离老人本人，也不脱离最近健康状态。

5. 代码层每一步到底在做什么

5.1 get_llm_service()：按配置返回不同实现

这个工厂函数的价值，是让业务层只关心“我要生成一条回复”，而不需要知道底层到底是 Mock、DashScope，还是别的平台。

这是典型的“面向能力编程，而不是面向供应商编程”。

5.2 build_messages()：把业务数据翻译成模型能理解的上下文

MemoryService 做的不是数据库查询本身，而是更高一层的工作：把多种来源的数据，整合成模型能消费的消息列表。

这里通常会组合：

  • 系统提示词。
  • 老人画像摘要。
  • 最近血压、心率等记录。
  • 最近几轮对话消息。
  • 当前用户问题。

这就是“上下文构造”真正的含义。

5.3 为什么最近几条消息就够第一版使用

因为你当前阶段要解决的是“上下文感”，不是“超长记忆系统”。

最近几轮消息足以支撑：

  1. 前后话题不完全断裂。
  2. 系统知道用户刚刚在聊什么。
  3. 健康建议能参考最近场景。

这就已经足够支撑基础功能完成。

6. 今天最重要的思维升级

你现在要明白：所谓“智能回复”，不是模型自己 magically 变懂，而是你后端把正确的上下文准备好了。

模型只负责生成，理解用户场景的大部分准备工作，其实是你的后端在做。

7. 今天最容易错在哪

  • 把模型调用硬塞进 chat_service，没有单独抽象。
  • 不做 Mock，实现一开始就被外部 API 绑住。
  • 上下文只拿消息，不拿老人画像和健康记录。
  • 试图一上来做特别复杂的长期记忆，拖慢项目节奏。

8. 今天完成标准

  • 聊天服务已经通过 llm_service 调模型，而不是自己直接写死。
  • 上下文构造已单独放进 memory_service。
  • 你能解释“最近消息 + 老人画像 + 最近记录”为什么足够支撑第一版。
  • 你明白“后端负责准备上下文，模型负责生成文本”这个职责边界。
