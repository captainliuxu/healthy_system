Day 2 / 15工程起步

第2天：项目骨架、配置、数据库会话

今天做的代码，表面看起来“业务味道不强”，但它决定了后面所有模块的稳定性。今天的任务不是堆功能，而是把项目的基础骨架搭正确。

1. 今天要做哪些文件

  • app/core/config.py
  • app/core/exception.py
  • app/core/response.py
  • app/db/session.py
  • app/api/deps.py
  • app/main.py 的最小可启动版本
  • app/api/routes/health.py

2. 为什么先做这些，而不是马上做业务表

因为后面所有业务模块都会重复依赖这些能力：

  1. 配置从哪里读取。
  2. 数据库连接怎么创建、怎么关闭。
  3. 接口返回格式是否统一。
  4. 抛出业务错误时返回什么结构。
  5. FastAPI 服务怎么初始化。

如果这些基础点今天不先定住，后面每做一个模块你就要重新纠结一遍。

3. 先理解每个文件到底在干什么

文件作用今天你必须理解的点
config.py统一读取环境配置让数据库地址、模型配置、跨域设置都有唯一来源
exception.py统一处理业务错误和未知错误后端不能到处手写错误返回
response.py统一成功响应结构前后端联调要有稳定返回格式
session.py创建 SQLAlchemy 引擎和会话每个请求都应该拿到独立数据库会话
deps.py给路由提供依赖注入入口后面所有接口都能统一获取 db
main.py应用装配中心把中间件、路由、异常处理接起来

4. 代码层每一步到底在做什么

4.1 Settings 不是普通类，它是在集中管理系统开关

在 config.py 里定义 Settings，本质是在做一件事：把“写死在各处的配置值”集中到一个地方。

这段代码背后真正的工程含义是：

  1. 数据库地址以后可能变。
  2. 模型供应商以后可能变。
  3. 跨域配置以后可能变。
  4. 调度器开关以后可能变。

所以配置不应该散在项目每个角落。

4.2 create_engine 和 SessionLocal 是数据库入口

你在 session.py 里做的不是“随便连一下数据库”，而是在给整个项目建立统一的数据访问入口。

里面最关键的三个对象分别负责：

  • Base：所有模型类的共同基类。
  • engine：数据库连接引擎。
  • SessionLocal：每次业务操作时拿到数据库会话。

get_db() 这个生成器函数尤其重要。它的真实意义不是语法技巧，而是：

每次接口请求进来，都拿一个新的数据库会话；请求结束后，再把它安全关闭。

4.3 统一响应是在降低前端复杂度

如果每个接口返回结构都不一样，前端会越来越难处理。所以你统一成：

{
  "code": 0,
  "message": "success",
  "data": ...
}

这件事的本质是给前后端约定一个稳定合同。

4.4 统一异常是在降低后端混乱度

业务错误和系统错误不是一回事：

  1. 业务错误：比如老人档案不存在。
  2. 系统错误：比如未预期异常、数据库问题。

你单独做 BusinessException，就是为了明确区分“业务失败”和“系统崩溃”。

4.5 main.py 是装配中心，不是业务处理中心

今天你在 main.py 做的事情主要有：

  1. 创建 FastAPI 应用。
  2. 加 CORS 中间件。
  3. 挂健康检查路由。
  4. 注册异常处理器。

也就是说，main.py 的任务是“组装”，不是“执行业务”。

5. 今天最容易犯的错误

  • 把数据库连接直接写在每个路由里。
  • 每个接口各自返回不同 JSON 结构。
  • 发现报错就直接 return {"error": ...}，没有统一异常出口。
  • 把配置值写死在多个文件里。

6. 今天完成标准

  • 服务能启动。
  • /api/v1/health 能返回成功结构。
  • 你知道一个请求如何拿到数据库会话。
  • 你知道业务异常和未知异常为什么要分开处理。
