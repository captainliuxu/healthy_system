Day 4 / 15第一个完整模块

第4天：把 Profile 做成完整业务链路

今天是你第一次真正把“一个模型”扩成“一个模块”。从今天开始，你不只是会建表，而是会把一次请求完整送进系统，再完整返回给前端。

1. 今天要完成哪些文件

  • app/schemas/profile.py
  • app/repositories/profile_repo.py
  • app/services/profile_service.py
  • app/api/routes/profiles.py
  • app/main.py 中把 profiles_router 挂上去

2. 为什么今天一定要先把 Profile 做完整

因为它是系统中心实体。后面你再做 record、conversation、chat、proactive，都会频繁依赖“先找到某位老人”。

如果 Profile 模块本身都不完整，后面所有模块都会不停重复补逻辑。

3. 从请求到落库，一次完整流程到底怎么走

环节负责文件本质动作
接收请求routes/profiles.py拿到请求参数，交给 service
校验数据schemas/profile.py确保输入形状正确、字段类型正确
业务判断services/profile_service.py决定创建、查询、更新、删除应该怎么做
数据库操作repositories/profile_repo.py把数据真正写入或读出数据库
统一返回response.py给前端稳定响应结构

今天最重要的理解是：这一条链路不是为了显得“正规”，而是为了让你的代码职责清晰。

4. 每个文件今天到底在做什么

4.1 schemas/profile.py：定义接口合同

这里你至少会定义三类结构：

  1. ProfileCreate：创建时允许传什么。
  2. ProfileUpdate：更新时哪些字段可选。
  3. ProfileRead：返回给前端时长什么样。

真正含义是：输入合同和输出合同分开。

比如创建时可能必须要有 name，但查询返回时会多出 id、created_at、updated_at。

4.2 profile_repo.py：只做数据库动作，不做业务判断

这个文件主要封装：

  • 创建老人档案。
  • 按 id 查询老人档案。
  • 分页查询老人列表。
  • 更新老人。
  • 删除老人。

这里最关键的理解是：repo 不应该思考“这条档案该不该创建”，它只负责“怎么创建”。

4.3 profile_service.py：把业务规则放这里

Service 层做的是“决定流程”。例如：

  1. 查询不到老人时，不是简单返回空，而是抛业务异常。
  2. 更新时只更新用户真正传入的字段。
  3. 删除前先确认对象存在。

这就是为什么 service 才是后端的大脑。

4.4 routes/profiles.py：让 HTTP 请求接上业务能力

你今天写的每个接口，本质都在做同一个动作：

从 HTTP 世界进入业务世界，再从业务世界返回 HTTP 世界。

所以路由文件应该尽量轻，只保留：

  • 请求参数定义。
  • 依赖注入 db。
  • 调用 service。
  • 返回 success_response。

5. 今天你应该如何理解 CRUD

今天的 CRUD 不只是增删改查练手，它是在帮你建立一种固定模式。你后面做 Record、Conversation 甚至主动配置，都会复用这套模块结构。

也就是说，Profile 模块是你的第一个“模板模块”。

6. 今天最容易错在哪

  • 把校验逻辑写到路由里，而不是 schema 或 service 里。
  • repo 里开始抛业务异常，职责混乱。
  • 更新接口直接整对象覆盖，没有做增量更新。
  • 返回数据库对象时没有统一成功结构。

7. 今天完成标准

  • 能创建老人档案。
  • 能获取老人列表。
  • 能获取单个老人详情。
  • 能更新和删除档案。
  • 你能说清 Schema、Repo、Service、Route 在这个模块里分别承担什么。
