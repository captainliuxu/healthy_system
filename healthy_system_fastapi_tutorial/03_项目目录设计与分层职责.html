03. 项目目录设计与分层职责

1. 为什么目录设计这么重要

在小脚本里，目录乱一点问题不大。
但只要进入“项目开发”，目录结构就会直接影响：

  • 你找代码快不快
  • 功能能不能继续扩展
  • 你以后敢不敢重构
  • 别人看你项目时会不会一脸懵

所以，目录设计不是形式主义，它本质上是在做 复杂度管理。

2. 推荐目录结构 

backend/
├─ app/

│  ├─ core/
│  │  ├─ config.py

│  │  ├─ exceptions.py
│  │  └─ logging.py
│  ├─ api/
│  │  └─ routes/
│  │     ├─ health.py
│  │     ├─ system.py
│  │     ├─ chat.py
│  │     ├─ profile.py
│  │     └─ record.py

│  │  ├─ common.py
│  │  ├─ chat.py
│  │  ├─ profile.py
│  │  └─ record.py
│  ├─ services/

│  │  ├─ profile_service.py

│  └─ utils/

├─ requirements.txt
└─ .env

3. 每一层的职责

3.1 app/main.py

应用入口。
只做“组装”，不做具体业务。

负责：

  • 创建 FastAPI() 实例
  • 注册中间件
  • 注册路由
  • 注册异常处理器
  • 设置标题和文档信息

不负责：

  • 直接写聊天逻辑
  • 直接查数据库
  • 直接写复杂业务

3.2 core/

全局基础设施层。

config.py

负责：

  • 读取 .env
  • 暴露配置项
  • 做配置的默认值和环境判断

response.py

负责：

  • 定义统一成功 / 失败返回格式

exceptions.py

负责：

  • 自定义业务异常
  • 全局异常处理逻辑

logging.py

负责：

  • 日志初始化
  • 输出格式

3.3 api/routes/

接口层。

特点：

  • 每个文件按业务域划分
  • URL 规则在这里定义
  • 尽量薄

例如：

health.py

  • /health

chat.py

  • /api/chat

profile.py

  • /api/profiles

record.py

  • /api/records

3.4 schemas/

数据结构层。

它的价值非常大，因为它让你的代码不再“靠猜”。

典型作用

  • 定义请求体
  • 定义响应体
  • 定义字段校验
  • 让文档自动生成

例如：

  • ChatRequest
  • ProfileCreate
  • RecordCreate

3.5 services/

业务逻辑层。

这是后端项目最核心的地方之一。
它负责把“业务逻辑”和“HTTP”解耦。

比如：

  • 聊天服务：如何调用模型、如何拼接 system prompt
  • 档案服务：如何创建和查询档案
  • 记录服务：如何保存健康记录，如何聚合

以后如果你接数据库、缓存、队列，也大多是在这里串联。

3.6 data/

第一阶段先放“伪数据存储”。

为什么推荐加这一层？

因为你还没上数据库，但总不能所有文件都各自保存临时数据。
可以先集中放：

  • profiles = []
  • records = []

以后接数据库时，这层可以替换成 DAO / repository / ORM。

3.7 utils/

通用工具层。

只放真正通用的工具，比如：

  • 时间格式转换
  • 字符串辅助
  • ID 生成

不要把业务逻辑偷塞进 utils。

4. 推荐的命名习惯

文件命名

用小写加下划线：

  • chat_service.py
  • profile_service.py

schema 类命名

用业务名 + 动作：

  • ChatRequest
  • ChatResponse
  • ProfileCreate
  • ProfileUpdate
  • RecordItem

路由变量命名

尽量语义清晰：

  • profile_id
  • record_id
  • current_user

5. 为什么不建议一开始加太多层

有些教程会教你一上来就写：

  • controller
  • service
  • repository
  • dao
  • model
  • entity
  • vo
  • dto

这在你现在这个阶段会把你绕晕。

第一阶段推荐最小分层

只保留：

  • routes
  • schemas
  • services
  • core

够用了。

6. 路由按“业务域”拆，不按 HTTP 方法拆

错误示例：

  • get_routes.py
  • post_routes.py

正确思路：

  • chat.py
  • profile.py
  • record.py

因为你是在写业务系统，不是在写 HTTP 教材。

7. 真实开发里的抽象边界怎么判断

你以后总会纠结：
“这个应该放哪层？”

可以用下面的判断方式：

属于 HTTP 协议层的问题

放 routes
例如：

  • 路径参数
  • 查询参数
  • 状态码
  • header

属于数据格式的问题

放 schemas
例如：

  • 字段类型
  • 字段校验
  • 响应结构

属于业务规则的问题

放 services
例如：

  • 老人档案创建逻辑
  • 重复提交校验
  • 聊天 prompt 拼接
  • 健康记录聚合

属于全局基础设施的问题

放 core
例如：

  • 配置
  • 日志
  • 异常
  • 统一响应

8. 一个接口从请求到返回的完整流转

以 POST /api/chat 为例：

  1. 前端发 JSON
  2. 路由层接收 ChatRequest
  3. 路由层调用 chat_service.reply()
  4. service 层调用模型接口
  5. service 层返回业务结果
  6. 路由层包成统一响应
  7. FastAPI 按 schema 序列化给前端

这个流转一旦形成习惯，你写任何功能都会更稳。

9. 一句话总结本文件

目录结构不是摆样子，而是在提前管理项目未来的复杂度。你现在最该建立的是“每一层该做什么”的感觉。
