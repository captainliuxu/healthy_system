# 07. 接口设计规范与前后端联调

这一部分会直接影响你和前端协作的效率。  
很多项目不是写不出功能，而是接口设计混乱，导致联调很痛苦。

---

## 1. 为什么接口设计要提前规范

如果接口随便写，后面会出现：

- URL 风格不统一
- 有的用 GET，有的用 POST，没有规律
- 返回格式不统一
- 参数命名混乱
- 前端每个页面都要单独适配

所以即使是个人项目，也应该尽量统一规则。

---

## 2. 推荐的 URL 风格

### 使用名词，不要用动词堆 URL

推荐：
- `GET /api/profiles`
- `POST /api/profiles`
- `GET /api/profiles/{profile_id}`
- `GET /api/records`
- `POST /api/records`

不推荐：
- `/api/getProfiles`
- `/api/createProfile`
- `/api/addRecord`

原因：
REST 风格更清晰，也更容易维护。

---

## 3. 推荐的 HTTP 方法使用方式

### GET
用于获取数据
例如：
- 查询档案
- 查询记录
- 查询系统信息

### POST
用于创建数据或发起动作
例如：
- 创建档案
- 创建健康记录
- 聊天请求

### PUT / PATCH
以后用于更新
例如：
- 修改档案
- 修改提醒设置

### DELETE
以后用于删除
例如：
- 删除记录
- 删除档案

---

## 4. 你的项目第一阶段推荐接口清单

### 系统相关
- `GET /health`
- `GET /api/system/info`

### 聊天相关
- `POST /api/chat`

### 档案相关
- `POST /api/profiles`
- `GET /api/profiles`
- `GET /api/profiles/{profile_id}`

### 记录相关
- `POST /api/records`
- `GET /api/records`
- `GET /api/records/{profile_id}`

---

## 5. 参数应该放哪里

### 路径参数
用于标识资源 ID
例如：
- `/api/profiles/{profile_id}`

### 查询参数
用于过滤、搜索、分页
例如：
- `/api/records?profile_id=1`
- `/api/records?record_type=blood_pressure`

### 请求体 JSON
用于提交复杂数据
例如：
- 创建档案
- 创建记录
- 聊天请求

### 你的一个建议
将来可以把
`GET /api/records/{profile_id}`
改成
`GET /api/records?profile_id=1`

因为它更适合继续扩展：
- `profile_id`
- `record_type`
- `page`
- `page_size`

---

## 6. 返回格式为什么一定要统一

推荐始终返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

不要有时返回对象，有时返回数组，有时返回 message。  
统一之后，前端可以写固定逻辑：

```js
if (res.data.code === 0) {
  // 成功
} else {
  // 失败提示
}
```

---

## 7. 给前端的接口文档最少要包含什么

每个接口至少应该说明：

- 请求方法
- URL
- 请求参数
- 示例请求
- 示例响应
- 失败时可能返回什么

---

## 8. 以聊天接口为例写一份简洁接口文档

### 接口
`POST /api/chat`

### 请求体
```json
{
  "message": "我最近总觉得头晕怎么办"
}
```

### 成功响应
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "reply": "如果持续头晕，建议先测量血压并注意休息，如症状明显请及时就医。"
  }
}
```

### 失败响应
```json
{
  "code": 2001,
  "message": "模型调用失败",
  "data": null
}
```

## 9. 以档案接口为例写给前端看的约定

### 创建档案
`POST /api/profiles`

请求体：
```json
{
  "name": "张三",
  "age": 70,
  "gender": "男",
  "disease_history": "高血压",
  "emergency_contact": "儿子 138xxxx"
}
```

成功响应：
```json
{
  "code": 0,
  "message": "档案创建成功",
  "data": {
    "id": 1,
    "name": "张三",
    "age": 70,
    "gender": "男",
    "disease_history": "高血压",
    "emergency_contact": "儿子 138xxxx"
  }
}
```

## 10. 前后端联调时的推荐顺序

### 第一步：只测 `/health`
确保后端服务真的启动了。

### 第二步：只测 `/docs`
确认接口定义和请求体正确。

### 第三步：用 Apifox / Postman 或 Swagger 自测
不要一上来就让前端帮你测。

### 第四步：让前端先接最简单接口
比如系统信息、档案列表。

### 第五步：再接聊天和记录
因为这两块更容易涉及数据结构变化。

---

## 11. 常见联调问题与排查思路

### 问题 1：前端请求到了，但拿不到数据
先看：
- 后端有没有统一返回 `data`
- 前端是不是写错了解析路径

### 问题 2：跨域报错
检查：
- CORS 是否允许前端地址
- 是否用错端口

### 问题 3：422 校验错误
通常是：
- 前端传的 JSON 字段名不对
- schema 字段缺失
- 类型不匹配

### 问题 4：聊天接口报错
通常检查：
- `.env` 是否读到 key
- 模型服务是否正常
- 后端异常有没有打印日志

---

## 12. 对你项目的一个实用建议

你前端已有页面：
- chat
- record
- profile

所以你后端最好优先保证以下接口稳定：

- `POST /api/chat`
- `GET /api/profiles`
- `POST /api/profiles`
- `GET /api/records`
- `POST /api/records`

这样前端页面最容易开始联调。

---

## 13. 一句话总结本文件

> 后端接口设计的目的，不只是“返回数据”，而是给前端一个稳定、可预测、能长期协作的协议层。
