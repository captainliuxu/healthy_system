healthy_system 后端改造详细教程（FastAPI 工程化版）

适用对象：已经会一点 Flask、刚入门 FastAPI、想把自己的课程/比赛项目改造成真正能展示的项目的人。

本教程完全围绕你的仓库现状来设计：当前仓库已经是前后端分离结构，根目录有 backend 和 frontend，前端已经有 chat / record / profile 页面路由，但后端目前仍以单文件入口为主，主要只有一个直接调用模型的聊天接口。目标不是“把代码写得更花”，而是把后端改成一个可维护、可扩展、可联调、可写进简历的项目。

0. 先看清你现在项目的问题

你现在项目并不是“完全没后端”，而是处于一个很典型的阶段：

  • 目录上已经分成了 backend 和 frontend
  • 前端路由已经有 /、/chat、/record、/profile
  • 后端已经能启动 FastAPI，配置了 CORS，并暴露了一个 GET /chat 接口
  • 聊天接口里直接读取 .env，直接调用 Qwen SDK
  • README 还很薄，几乎没有交接信息

这说明你已经走出了“从零开始”的阶段，但现在还没走到“工程化项目”的阶段。

为什么现在不能继续维持单文件写法

因为单文件后端有 5 个明显问题：

  1. 所有东西都挤在入口文件里：配置、接口、模型调用、异常处理混在一起。
  2. 功能一多必乱：今天只有聊天还能撑住，明天加档案、健康记录、鉴权、数据库就会越来越难改。
  3. 前后端联调不稳定：没有统一响应结构，前端一旦接入多个页面，接口风格不一致就会难受。
  4. 不利于协作：别人打开你的仓库，很难一下子看懂“业务写在哪”。
  5. 不利于展示：面试/答辩时，人家一看就是“能跑的 demo”，不是“会做项目的人做的项目”。

所以你这次改造的核心目标不是“会几个新语法”，而是：

把后端从“能跑”升级成“像一个真实项目”。

1. 改造总目标

改造完成后，你的后端至少要达到这 10 条：

  1. 多文件结构，不再只有一个 main.py
  2. 路由、业务逻辑、数据访问分层清晰
  3. 所有接口都有统一前缀 /api/v1
  4. 所有接口都有统一响应格式
  5. 所有请求体和响应体都有 Pydantic schema
  6. 数据落库，而不是只存在内存里
  7. 数据库结构由迁移工具管理
  8. 有全局异常处理和基础日志
  9. 前端能稳定联调 profile / record / chat
  10. README、.env.example、启动步骤完整

这 10 条做到以后，你的项目就已经不是“练习代码”，而是一个合格的学生后端项目。

│  │  ├─ deps.py
│  │  └─ routes/

│  ├─ db/
│  │  ├─ base.py
│  │  ├─ session.py
│  │  └─ models/
│  │     ├─ profile.py
│  │     ├─ record.py

这个结构每层的作用

  • app/main.py：应用入口，只负责组装，不写业务。
  • api/routes/：真正的接口地址定义位置。
  • schemas/：请求和响应的数据结构。
  • services/：业务逻辑层。
  • repositories/：数据库读写层。
  • db/models/：数据库表结构。
  • core/：配置、异常、统一响应、日志等全局能力。
  • tests/：基础测试。
  • alembic/：数据库迁移。

这一步的作用：让你一开始就知道目标长什么样，后面每创建一个文件，都知道它为什么存在，而不是“先建再说”。

3. 第 1 步：先保存当前版本，避免改崩

操作

先在项目根目录执行：

git add .
git commit -m "保存单文件后端版本"
git checkout -b refactor/backend-v1

这一步的作用

很多新手一改结构就把项目改崩，最后既回不去，又不知道哪里出错。先保存当前版本的意义是：

  • 给自己留退路
  • 后面重构出错可以随时对比
  • 这是一个真实开发习惯，不是形式主义

真正做项目的人，在大改结构前都会先留一个稳定点。

4. 第 2 步：确定第一版只做哪些功能

操作

先把本次后端改造的功能范围写下来，不要边做边扩：

第一版后端只做：

  1. 健康检查接口
  2. 老人档案 CRUD
  3. 健康记录 CRUD
  4. 聊天接口
  5. 统一响应格式
  6. 全局异常处理
  7. 数据库存储
  8. 基础日志
  9. 基础测试
  10. 完整 README

暂时不做：

  • 登录注册
  • 权限系统
  • 向量库
  • Redis
  • 消息队列
  • 定时任务
  • 多角色后台

这一步的作用

项目最容易失败的原因之一，就是“一边学一边加功能，最后所有东西都半成品”。

你现在最重要的是先做出一个闭环：

  • 前端有页面
  • 后端有接口
  • 数据能保存
  • 项目能跑
  • 别人能看懂

而不是“功能特别多”。

5. 第 3 步：先装依赖，统一开发环境

操作

进入 backend 目录：

cd backend

建议先创建虚拟环境，如果你已经有就激活现有环境：

python -m venv .venv

Windows 激活：

.venv\Scripts\activate

然后安装第一批依赖：

pip install fastapi uvicorn python-dotenv sqlalchemy alembic pydantic-settings dashscope pytest httpx

然后生成 requirements.txt：

pip freeze > requirements.txt

这一步的作用

这一步不是“机械安装包”，而是为后面每个模块铺路：

  • fastapi：后端框架
  • uvicorn：运行服务器
  • python-dotenv：读环境变量
  • sqlalchemy：数据库 ORM
  • alembic：数据库迁移
  • pydantic-settings：规范管理配置
  • dashscope：调用你当前用的模型
  • pytest、httpx：测试接口

没有依赖统一，你后面写出来的教程步骤别人无法复现，你自己下次换电脑也会忘。

6. 第 4 步：创建新的目录结构

操作

在 backend 下手动创建这些目录和文件：

backend/
├─ app/
│  ├─ api/
│  │  └─ routes/
│  ├─ core/
│  ├─ db/
│  │  └─ models/
│  ├─ schemas/
│  ├─ repositories/
│  ├─ services/
│  └─ utils/
├─ tests/

并在这些目录里加上 __init__.py（至少 app/、api/、routes/、db/、models/、schemas/、repositories/、services/ 这些 Python 包目录建议都加）。

这一步的作用

这一步是把“代码堆在一起”的后端，改成“按职责划分”的后端。它的价值在于：

  • 你以后不会再问“这个逻辑该写哪”
  • 后续加功能不会继续挤爆入口文件
  • 队友或老师打开仓库能直接理解模块分工

目录结构本身就是一种“工程化表达能力”。

7. 第 5 步：先写配置中心 core/config.py

操作

创建 backend/app/core/config.py：

from pathlib import Path
from pydantic_settings import BaseSettings, SettingsConfigDict


class Settings(BaseSettings):
    APP_NAME: str = "healthy_system backend"
    DEBUG: bool = True
    API_V1_PREFIX: str = "/api/v1"
    DATABASE_URL: str = "sqlite:///./healthy_system.db"
    DASHSCOPE_API_KEY: str = ""
    ALLOWED_ORIGINS: str = "*"

    model_config = SettingsConfigDict(
        env_file=str(Path(__file__).resolve().parent.parent.parent / ".env"),
        env_file_encoding="utf-8",
        extra="ignore",
    )


settings = Settings()

同时创建 .env.example

在 backend/.env.example 中写：

APP_NAME=healthy_system backend
DEBUG=true
API_V1_PREFIX=/api/v1
DATABASE_URL=sqlite:///./healthy_system.db
DASHSCOPE_API_KEY=your_key_here
ALLOWED_ORIGINS=*

这一步的作用

这一步是为了让“配置”和“代码”分开。

你现在项目里是直接在入口文件里读 .env。短期能用，但长期会有几个问题：

  • 哪些配置存在，不清楚
  • 不同文件到处 os.getenv()，很乱
  • 换环境时容易漏
  • README 难写清楚

配置中心的核心价值是：

  1. 所有配置只有一个入口
  2. 所有人一眼知道要填哪些环境变量
  3. 以后换数据库、换接口前缀、换模型 key 都不用全局搜字符串

项目里凡是“将来可能变”的东西，都应该尽量从配置中心读，而不是写死在业务逻辑里。

8. 第 6 步：重写 main.py，让入口文件只负责组装

操作

创建 backend/app/main.py：

from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

from app.core.config import settings
from app.api.routes.health import router as health_router

app = FastAPI(title=settings.APP_NAME)

app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"] if settings.ALLOWED_ORIGINS == "*" else settings.ALLOWED_ORIGINS.split(","),
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

app.include_router(health_router, prefix=settings.API_V1_PREFIX)

这一步的作用

你当前项目最大的结构问题，就是入口文件做了太多事：

  • 读配置
  • 配中间件
  • 定义接口
  • 调模型
  • 异常处理

真正像样的入口文件应该很“瘦”，只负责组装应用。这样做的好处：

  • 你打开 main.py 就能看懂整个项目挂了哪些模块
  • 业务逻辑不再污染入口
  • 新增接口时只要新增 router 并注册即可

入口文件越薄，项目越像项目。

9. 第 7 步：先做最简单的 health 路由

操作

创建 backend/app/api/routes/health.py：

from fastapi import APIRouter

router = APIRouter(tags=["health"])


@router.get("/health")
def health_check():
    return {
        "code": 0,
        "message": "success",
        "data": {"status": "ok"}
    }

启动项目：

uvicorn app.main:app --reload

浏览器访问：

http://127.0.0.1:8000/api/v1/health

这一步的作用

很多人一上来就写最复杂的业务接口，结果一旦出错，不知道是：

  • 启动有问题
  • 路由没注册
  • 中间件有问题
  • 业务逻辑有问题

而 health 接口的价值就在于：

  • 它不依赖数据库
  • 不依赖模型 SDK
  • 不依赖前端
  • 最容易验证“后端基本框架已经通了”

以后你每做一步大改，都先验证 /api/v1/health 是否还正常，这是非常专业的习惯。

10. 第 8 步：先建立统一响应模型

操作

创建 backend/app/schemas/common.py：

from typing import Generic, Optional, TypeVar
from pydantic import BaseModel

T = TypeVar("T")


class ApiResponse(BaseModel, Generic[T]):
    code: int = 0
    message: str = "success"
    data: Optional[T] = None

创建 backend/app/core/response.py：

from app.schemas.common import ApiResponse


def success_response(data=None, message: str = "success"):
    return ApiResponse(code=0, message=message, data=data)

然后把 health.py 改成：

from fastapi import APIRouter
from app.core.response import success_response

router = APIRouter(tags=["health"])


@router.get("/health")
def health_check():
    return success_response({"status": "ok"})

这一步的作用

统一响应不是“花里胡哨”，而是前后端联调效率的核心。

如果没有统一响应，后面就会出现：

  • A 接口返回 {"reply": ...}
  • B 接口返回 {"status": 200, "data": ...}
  • C 接口报错返回字符串

前端处理起来会非常痛苦。

统一响应格式之后，你的前端拿到任何接口结果，都会按同一套结构处理：

  • code：是否成功
  • message：提示信息
  • data：真正的数据

这一步的价值是给整个后端“立规矩”。

11. 第 9 步：建立全局异常处理

操作

创建 backend/app/core/exception.py：

from fastapi import Request
from fastapi.responses import JSONResponse


class BusinessException(Exception):
    def __init__(self, code: int, message: str):
        self.code = code
        self.message = message


async def business_exception_handler(request: Request, exc: BusinessException):
    return JSONResponse(
        status_code=400,
        content={
            "code": exc.code,
            "message": exc.message,
            "data": None,
        },
    )


async def generic_exception_handler(request: Request, exc: Exception):
    return JSONResponse(
        status_code=500,
        content={
            "code": 50000,
            "message": "服务器内部错误",
            "data": None,
        },
    )

然后在 main.py 注册：

from app.core.exception import (
    BusinessException,
    business_exception_handler,
    generic_exception_handler,
)

app.add_exception_handler(BusinessException, business_exception_handler)
app.add_exception_handler(Exception, generic_exception_handler)

这一步的作用

你当前聊天接口的写法是把异常直接 try/except 后返回字符串。这种写法有两个问题：

  1. 每个接口都可能各写各的异常风格
  2. 真出错时没有统一结构，前端很难处理

全局异常处理的意义：

  • 业务异常统一输出
  • 系统异常统一兜底
  • 以后路由里不用到处写 try/except
  • 项目更整洁

当你把“异常”也做成统一能力时，你的后端才真正开始像一个项目。

12. 第 10 步：设计第一个真实业务模块 profile

为什么先做 profile

因为你的前端已经有 /profile 页面，且“老人档案”是这个项目里几乎所有业务的基础：

  • 健康记录依赖老人档案
  • 聊天建议以后也可能依赖老人档案
  • 将来提醒、统计、个性化建议都依赖老人档案

所以 profile 必须是你第一个完整打通的业务模块。

13. 第 11 步：先定义 profile 的接口数据结构

操作

创建 backend/app/schemas/profile.py：

from datetime import date, datetime
from typing import Optional
from pydantic import BaseModel, Field


class ProfileBase(BaseModel):
    name: str = Field(..., min_length=1, max_length=50)
    gender: str = Field(..., description="男/女/其他")
    birth_date: Optional[date] = None
    phone: Optional[str] = None
    emergency_contact_name: Optional[str] = None
    emergency_contact_phone: Optional[str] = None
    chronic_diseases: Optional[str] = None
    allergies: Optional[str] = None
    remark: Optional[str] = None


class ProfileCreate(ProfileBase):
    pass


class ProfileUpdate(BaseModel):
    name: Optional[str] = None
    gender: Optional[str] = None
    birth_date: Optional[date] = None
    phone: Optional[str] = None
    emergency_contact_name: Optional[str] = None
    emergency_contact_phone: Optional[str] = None
    chronic_diseases: Optional[str] = None
    allergies: Optional[str] = None
    remark: Optional[str] = None


class ProfileRead(ProfileBase):
    id: int
    created_at: datetime
    updated_at: datetime

    model_config = {
        "from_attributes": True
    }

这一步的作用

很多新手容易混淆：

  • 数据库模型
  • 接口请求体
  • 接口响应体

这一步就是在建立一个非常重要的意识：

接口的数据结构，不等于数据库表结构。

为什么要单独写 schema：

  • 请求时只接收允许用户传的字段
  • 返回时只返回允许前端看到的字段
  • 校验规则写在这里更清晰
  • 文档会自动生成

从这里开始，你就不再是“直接收 dict”，而是在按接口契约开发。

14. 第 12 步：接数据库，先用 SQLite 跑通

为什么先用 SQLite

你当然可以以后换 MySQL，但当前最优解是：

  • 先用 SQLite 跑通本地开发流程
  • 代码结构按正式项目写
  • 以后只换连接串，不大动业务代码

这样你不会一边学 FastAPI，一边被本地数据库环境折磨。

操作：创建数据库会话

创建 backend/app/db/session.py：

from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker, DeclarativeBase
from app.core.config import settings


class Base(DeclarativeBase):
    pass


engine = create_engine(
    settings.DATABASE_URL,
    connect_args={"check_same_thread": False} if settings.DATABASE_URL.startswith("sqlite") else {},
)

SessionLocal = sessionmaker(bind=engine, autoflush=False, autocommit=False)


def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

创建 backend/app/api/deps.py：

from app.db.session import get_db

这一步的作用

这一步是后面所有“真实落库”的前提。它的价值在于：

  • 统一管理数据库连接
  • 接口里可以按依赖注入拿到 db
  • 以后换数据库时，主要改这里和配置即可

这一步开始，你的项目才真正准备从“演示”走向“有状态的应用”。

15. 第 13 步：建立 profile 数据库表模型

操作

创建 backend/app/db/models/profile.py：

from datetime import datetime
from sqlalchemy import String, Date, DateTime, Text
from sqlalchemy.orm import Mapped, mapped_column
from app.db.session import Base


class Profile(Base):
    __tablename__ = "profiles"

    id: Mapped[int] = mapped_column(primary_key=True, index=True)
    name: Mapped[str] = mapped_column(String(50), nullable=False)
    gender: Mapped[str] = mapped_column(String(20), nullable=False)
    birth_date: Mapped[Date | None] = mapped_column(Date, nullable=True)
    phone: Mapped[str | None] = mapped_column(String(20), nullable=True)
    emergency_contact_name: Mapped[str | None] = mapped_column(String(50), nullable=True)
    emergency_contact_phone: Mapped[str | None] = mapped_column(String(20), nullable=True)
    chronic_diseases: Mapped[str | None] = mapped_column(Text, nullable=True)
    allergies: Mapped[str | None] = mapped_column(Text, nullable=True)
    remark: Mapped[str | None] = mapped_column(Text, nullable=True)
    created_at: Mapped[datetime] = mapped_column(DateTime, default=datetime.utcnow)
    updated_at: Mapped[datetime] = mapped_column(DateTime, default=datetime.utcnow, onupdate=datetime.utcnow)

再创建 backend/app/db/models/__init__.py：

from app.db.models.profile import Profile

这一步的作用

数据库模型是“数据最终存哪、长什么样”的定义。它的意义是：

  • 真正把数据落到数据库里
  • 定义字段类型、长度、是否允许为空
  • 为后面迁移工具提供依据

Schema 是接口契约，Model 是表结构。二者必须分清。

16. 第 14 步：先不急着写路由，先写 repository

操作

创建 backend/app/repositories/profile_repo.py：

from sqlalchemy.orm import Session
from app.db.models.profile import Profile


class ProfileRepository:
    def create(self, db: Session, data: dict) -> Profile:
        obj = Profile(**data)
        db.add(obj)
        db.commit()
        db.refresh(obj)
        return obj

    def get_by_id(self, db: Session, profile_id: int):
        return db.query(Profile).filter(Profile.id == profile_id).first()

    def get_all(self, db: Session):
        return db.query(Profile).order_by(Profile.id.desc()).all()

    def update(self, db: Session, obj: Profile, data: dict):
        for key, value in data.items():
            setattr(obj, key, value)
        db.commit()
        db.refresh(obj)
        return obj

    def delete(self, db: Session, obj: Profile):
        db.delete(obj)
        db.commit()

这一步的作用

很多初学者会把所有数据库操作直接写到路由里，这会让接口越来越臃肿。

Repository 的作用是：

  • 专门负责数据库操作
  • 把“查表、增删改”从业务逻辑里拆出去
  • 以后同一套数据库操作可以被多个 service 重用

Route 不应该直接写一堆 SQLAlchemy 细节。先把“数据库访问层”独立出来，你的代码会干净很多。

17. 第 15 步：再写 service，处理业务规则

操作

创建 backend/app/services/profile_service.py：

from sqlalchemy.orm import Session
from app.core.exception import BusinessException
from app.repositories.profile_repo import ProfileRepository
from app.schemas.profile import ProfileCreate, ProfileUpdate

repo = ProfileRepository()


class ProfileService:
    def create_profile(self, db: Session, data: ProfileCreate):
        return repo.create(db, data.model_dump())

    def get_profile(self, db: Session, profile_id: int):
        profile = repo.get_by_id(db, profile_id)
        if not profile:
            raise BusinessException(40401, "档案不存在")
        return profile

    def list_profiles(self, db: Session):
        return repo.get_all(db)

    def update_profile(self, db: Session, profile_id: int, data: ProfileUpdate):
        profile = repo.get_by_id(db, profile_id)
        if not profile:
            raise BusinessException(40401, "档案不存在")
        update_data = data.model_dump(exclude_unset=True)
        return repo.update(db, profile, update_data)

    def delete_profile(self, db: Session, profile_id: int):
        profile = repo.get_by_id(db, profile_id)
        if not profile:
            raise BusinessException(40401, "档案不存在")
        repo.delete(db, profile)
        return True


profile_service = ProfileService()

这一步的作用

Service 层就是“业务规则层”。

为什么不能让路由直接调 repository：

  • 路由应该尽量薄
  • 校验某条数据是否存在，这是业务规则，不是路由责任
  • 以后如果创建档案前要检查手机号是否重复，也应该写在 service

简单理解：

  • repository 只管“怎么查库”
  • service 只管“业务该怎么走”

from typing import List
from fastapi import APIRouter, Depends
from sqlalchemy.orm import Session

from app.core.response import success_response
from app.db.session import get_db
from app.schemas.profile import ProfileCreate, ProfileRead, ProfileUpdate
from app.services.profile_service import profile_service

    profile = profile_service.update_profile(db, profile_id, data)
    return success_response(ProfileRead.model_validate(profile).model_dump())


@router.delete("/{profile_id}", response_model=dict)

    profile_service.delete_profile(db, profile_id)

20. 第 18 步：再做 record 模块，补足项目业务感

为什么 record 很重要

你前端已经有 /record 页面。如果你只有聊天和档案，那这个项目还不像“健康管理”项目。

健康记录模块一做出来，项目业务感会立刻上一个台阶。

建议的第一版设计

不要一开始拆血压表、血糖表、睡眠表、服药表四五张表。第一版更适合做成“统一记录表”：

  • profile_id
  • record_type：例如 blood_pressure / blood_sugar / medicine
  • value_json：存具体值
  • record_time
  • note

这样做的优点：

  • 开发快
  • 一张表先够用
  • 后面如果你要精细化拆表，也还能演进

操作：创建 schema

backend/app/schemas/record.py

再写 model

from datetime import datetime
from sqlalchemy import Integer, String, DateTime, Text, ForeignKey
from sqlalchemy.orm import Mapped, mapped_column

class Record(Base):
    __tablename__ = "records"

做项目时，业务模块的存在感非常重要。record 就是你这个项目从“聊天 demo”变成“健康管理项目”的关键一步。

21. 第 19 步：把聊天接口改成真正适合项目的形式

你现在的问题

你当前聊天接口是：

  • GET /chat
  • 通过 query 参数传消息
  • 路由函数里直接调用 SDK
  • 异常直接字符串返回

这更像调试接口，不像正式项目接口。

正确改法

schema

backend/app/schemas/chat.py

from typing import Optional
from pydantic import BaseModel


class ChatRequest(BaseModel):
    profile_id: Optional[int] = None
    message: str


class ChatResponse(BaseModel):
    reply: str

service

backend/app/services/chat_service.py

import dashscope
from app.core.config import settings
from app.core.exception import BusinessException


dashscope.api_key = settings.DASHSCOPE_API_KEY


class ChatService:
    def chat(self, message: str) -> str:
        if not settings.DASHSCOPE_API_KEY:
            raise BusinessException(40002, "后端未读取到 DASHSCOPE_API_KEY，请检查 .env")

        response = dashscope.Generation.call(
            model="qwen-turbo",
            messages=[
                {"role": "system", "content": "你是一个慢病健康管理助手，回答简洁、温和"},
                {"role": "user", "content": message},
            ],
        )
        return response["output"]["text"]


chat_service = ChatService()

route

backend/app/api/routes/chat.py

from fastapi import APIRouter
from app.core.response import success_response
from app.schemas.chat import ChatRequest
from app.services.chat_service import chat_service

router = APIRouter(prefix="/chat", tags=["chat"])


@router.post("", response_model=dict)

注册到 main.py：

这一步非常关键，因为它体现了你是不是“在做项目”，而不是“还在写 demo”。

改成 POST /api/v1/chat 的意义：

22. 第 20 步：把数据库创建方式从 create_all 升级成 Alembic

为什么要升级

Base.metadata.create_all() 只适合最开始测试，不适合长期项目。

一旦你后面改字段：

  • 改表名
  • 加字段
  • 改字段类型

你就会很痛苦。

Alembic 的意义就是：

  • 记录数据库结构演进历史
  • 每次变更都能追踪
  • 别人拉你的代码可以同步同样的表结构

操作

在 backend 目录执行：

alembic init alembic

编辑 alembic.ini，把数据库 URL 改成和 .env 对应，或者在 env.py 中读取项目配置。

然后在 alembic/env.py 中导入你的 Base：

from app.db.session import Base
from app.db.models import *

target_metadata = Base.metadata

生成第一次迁移：

alembic revision --autogenerate -m "create profiles and records tables"

应用迁移：

alembic upgrade head

这一步的作用

这一步的价值很大，因为它意味着：

  • 你的数据库结构是“可管理的”
  • 表结构变更不是靠手改
  • 项目交接时别人能复现同样的数据库

会 Alembic 不只是“会一个工具”，而是在学真实项目的数据库管理方式。

## 27. 第 25 步：哪些抽象是你现在就该预留的 {#27-第-25-步哪些抽象是你现在就该预留的 }

你说不只是基础，还要有“实际开发中可能需要做的功能的抽象”。这点很重要。

但注意：抽象不是越多越高级，而是**只抽未来一定会变的部分**。

### 现在就应该预留的抽象 {#现在就应该预留的抽象 }

#### 1. LLM 调用抽象 {#1-llm-调用抽象 }

今天你调 Qwen，明天可能换 DeepSeek、GLM。

所以你应该让：

- 路由只调用 `chat_service`
- `chat_service` 以后再调用 `llm_client`

这样换模型时不用动路由层。

#### 2. 数据访问抽象 {#2-数据访问抽象 }

今天你是 SQLite，明天可能换 MySQL。

所以：

- 数据库连接统一走 `session.py`
- 数据操作统一走 `repository`

#### 3. 统一响应和异常抽象 {#3-统一响应和异常抽象 }

所有接口都共用一套成功响应和异常响应。

#### 4. API 前缀版本抽象 {#4-api-前缀版本抽象 }

统一用 `settings.API_V1_PREFIX`。

### 现在不要过度抽象的东西 {#现在不要过度抽象的东西 }

- 不要急着写 BaseRepository、BaseService 一堆父类
- 不要急着上工厂模式
- 不要现在就为微服务做抽象

### 这一步的作用 {#这一步的作用-23 }

真正的抽象，目的是：

- 降低未来修改成本
- 让变化点收口
- 让结构更稳定

而不是“为了显得懂设计模式”。

> **抽象变化点，不抽象一切。** 这是你现在最该建立的工程观。

## 30. 最后给你的开发原则 {#30-最后给你的开发原则 }

最后我给你 8 条最重要的原则，今后做任何后端项目都适用：

1. **入口文件越薄越好**
2. **每个接口都先想清楚 schema**
3. **路由只负责接和回，不负责业务**
4. **业务规则写在 service，不写在 route**
5. **数据库操作单独收口到 repository**
6. **所有可变配置都从 settings 统一读取**
7. **先做能跑的闭环，再做高级功能**
8. **抽象未来一定会变的地方，不要为了抽象而抽象**

## 31. 你下一步应该怎么继续 {#31-你下一步应该怎么继续 }

你现在最该做的，不是再去看别人项目，而是：

1. 先按本教程把目录结构搭出来
2. 先把 `health + profile` 跑通
3. 再做 `record + chat`
4. 最后补 Alembic、日志、测试、README

等你把这套第一版做完，再来补：

- 鉴权
- 向量库
- 定时任务
- Redis 缓存
- 提醒策略

那时你会轻松很多，因为你的地基已经稳了。

## 32. 给你的最直接建议 {#32-给你的最直接建议 }

不要追求“一次写成最强版本”。

真正拿得出手的项目，不是功能最多，而是：

- 结构清楚
- 模块完整
- 接口规范
- 能稳定跑
- 别人能复现

你这次只要把这套后端骨架搭稳，已经会比绝大多数“只会把功能塞进一个文件”的同学强很多。
