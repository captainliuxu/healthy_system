healthy_system 后端开发教程：阅读顺序与总路线

适用对象：会 Flask、刚入门 FastAPI、希望直接围绕自己项目做开发的人
项目仓库：captainliuxu/healthy_system

1. 这套教程解决什么问题

你的项目当前已经是前后端分离结构，仓库里有 backend 和 frontend 两部分；前端已经有 /、/chat、/record、/profile 这些页面路由，而后端目前还是一个单文件 FastAPI 入口，主要完成了 .env 读取、CORS 配置和一个直接调用 Qwen 的 /chat 接口。
这说明你现在最缺的不是“FastAPI 语法大全”，而是：

  1. 如何把自己的项目从“能跑”升级成“能持续开发”
  2. 如何从 Flask 思维迁移到 FastAPI 的项目式开发
  3. 如何在不接入数据库、向量库的情况下，先搭好可扩展后端骨架
  4. 如何提前为真实开发常见需求预留抽象层

2. 你学这套教程时的目标

学完这套教程，你应该做到：

  • 能理解 FastAPI 项目的推荐分层
  • 能拆出 routes / schemas / services / core
  • 能写基本接口、统一响应、异常处理、配置管理
  • 能为以后接数据库、Redis、鉴权、日志、向量库预留接口
  • 能继续独立扩展你的项目

3. 不建议你现在先做什么

这阶段不要一口气做这些：

  • 不要上来就接数据库、ORM、Redis、Celery、消息队列
  • 不要先去啃一个几十个文件的成熟 FastAPI 项目
  • 不要为了“像大厂架构”而过早抽象十几层

原因不是这些不重要，而是你现在的瓶颈是 后端结构感，不是技术栈不够多。

4. 推荐的学习与开发顺序

第 1 阶段：项目骨架

先完成这些：

  • 目录拆分
  • 配置管理
  • 健康检查接口
  • 聊天接口改造为 POST
  • 档案接口
  • 记录接口的最小实现

第 2 阶段：工程能力

继续加：

  • 统一响应格式
  • 全局异常处理
  • 日志
  • .env 配置分层
  • 服务层抽象

第 3 阶段：业务抽象

按老年健康项目的实际场景加：

  • 老人档案
  • 健康记录
  • 健康摘要
  • 聊天建议
  • 定时提醒的任务接口预留
  • 权限和多用户角色预留

第 4 阶段：后续可扩展项

以后再逐步加入：

  • 数据库
  • 用户登录
  • 向量库 / RAG
  • 后台管理接口
  • 文件上传
  • 定时任务
  • 缓存

6. 这套教程的使用方法

最好的用法不是“先全部看完”，而是：

  • 打开教程
  • 一边读，一边改你的项目目录
  • 一边写，一边启动测试
  • 每完成一个阶段，就在浏览器 / Swagger 文档里验证

你应该把它当成一份 项目落地手册，而不是理论笔记。

7. 你的项目这阶段最重要的思想

你现在最该记住的一句话是：

FastAPI 不是拿来把 Flask 写法照搬过去的，而是拿来帮助你更自然地写“项目化后端”。

也就是说，你现在不只是要会写接口，而是要开始形成下面这种思维：

  • 路由层只负责接请求和返回结果
  • 数据格式要用 schema 明确表达
  • 业务逻辑不要直接写在接口函数里
  • 配置不要散落在不同文件
  • 未来可能复杂的地方，要先预留一层抽象

8. 本教程中的原则

这套教程遵循 5 个原则：

原则 1：先能跑，再优雅

先做最小可运行版本，然后再逐步抽象。

原则 2：只抽象真实会复用的部分

不要为了“看起来专业”而过度设计。

原则 3：接口先稳定，内部可以慢慢重构

前后端联调的关键是接口格式稳定。

原则 4：分层不是为了复杂，而是为了后面不乱

分层的价值在项目变大以后最明显。

原则 5：为真实需求留口子

哪怕你现在不做鉴权、数据库、向量库，也应该知道以后接在哪里。

9. 你读完后应该得到什么

读完这套教程后，你应当能产出：

  • 一个清晰的 backend/app 结构
  • 若干个基础 API
  • 一个可以继续扩展的服务层
  • 一套自己能解释清楚的后端结构

如果你能做到“我知道每个文件为什么在这里”，这套教程就成功了。
