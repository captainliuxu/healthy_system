Healthy System FastAPI 第二阶段详细教程

1. 这一阶段到底要解决什么

第二阶段只做三件核心事:

  1. User: 系统里到底是谁在使用
  2. Auth: 这个人怎么注册、登录、带着身份访问受保护接口
  3. Profile: 这个人的健康档案是什么

这一阶段的目标不是把所有业务都做完，而是先把“用户身份”和“用户画像上下文”稳稳落到数据库里。

因为后面的这些模块，几乎都会依赖它:

  1. 健康记录
  2. 聊天建议
  3. 主动提醒
  4. 个性化推荐
  5. 多角色扩展

如果第二阶段不先做好，后面所有业务都会陷入两个问题:

  1. 不知道数据到底归谁
  2. 不知道系统该基于什么上下文给建议

2. 第二阶段完成后的验收标准

做到下面这些，就算第二阶段合格:

  1. 用户可注册
  2. 用户可登录
  3. 用户访问受保护接口时，后端可以识别身份
  4. 用户可以查看自己的基本信息
  5. 用户可以修改自己的基本信息
  6. 每个用户只能维护自己的 profile
  7. profile 能创建、查看、更新

3. 推荐开发顺序为什么是 User -> Auth -> Profile

这个顺序不要反。

3.1 先做 User

因为:

  • 所有身份认证都要挂到用户表
  • 所有业务表最终都要有 user_id

3.2 再做 Auth

因为:

  • 没有认证，你就不知道谁在调用接口
  • 没有 JWT 和当前用户依赖注入，后面所有“我的数据”接口都没法写

3.3 最后做 Profile

因为:

  • profile 不是账号本身
  • profile 是这个账号对应的“健康档案 / 用户画像”

也就是说:

  • User 解决“你是谁”
  • Auth 解决“怎么证明你是你”
  • Profile 解决“系统应该如何理解你”

5. 这一阶段的数据关系先定死

先把关系想清楚，不然后面一定返工。

5.1 User 表职责

只负责账号身份相关的信息:

  1. 登录名
  2. 密码哈希
  3. 邮箱
  4. 手机号
  5. 账号状态
  6. 创建更新时间

5.2 Profile 表职责

只负责健康档案和用户画像:

  1. 姓名
  2. 年龄
  3. 性别
  4. 身高
  5. 体重
  6. 慢病史
  7. 过敏史
  8. 紧急联系人
  9. 备注

5.3 User 和 Profile 的关系

第二阶段最推荐先做成:

  • 一个 User
  • 对应一个 Profile

也就是 1 对 1

为什么先这样做:

  1. 比赛项目里通常一个账号对应一个核心使用者
  2. 后面健康记录、建议、提醒都可以直接挂到这个人的 profile 上
  3. 一对一最简单，不容易把业务关系绕乱

以后如果你要扩展成:

  • 一个监护人账号管理多个老人 profile

那是第三阶段以后再做，不要在第二阶段就过度设计。

9. User 模块设计

这一阶段的 User，只做最基础可用版本，不要把它做成“全功能用户中心”。

9.1 建议字段

username
email

is_active
created_at
updated_at

9.2 每个字段到底是干什么的

字段 作用 备注
id 用户主键 给别的表做外键
username 登录名 应该唯一
email 邮箱 可以为空，但如果有值建议唯一
phone 手机号 可以为空，但如果有值建议唯一
password_hash 密码哈希 绝对不能存明文密码
is_active 账号是否可用 后面可支持禁用用户
created_at 创建时间 排查和审计有用
updated_at 更新时间 方便看最后修改时间

9.3 app/models/user.py

完整代码如下:

from __future__ import annotations

from datetime import UTC, datetime

from sqlalchemy import Boolean, DateTime, String
from sqlalchemy.orm import Mapped, mapped_column, relationship

from app.db.session import Base


class User(Base):
    __tablename__ = "users"

    id: Mapped[int] = mapped_column(primary_key=True, index=True)
    username: Mapped[str] = mapped_column(String(50), unique=True, index=True, nullable=False)
    email: Mapped[str | None] = mapped_column(String(100), unique=True, index=True, nullable=True)
    phone: Mapped[str | None] = mapped_column(String(20), unique=True, index=True, nullable=True)
    password_hash: Mapped[str] = mapped_column(String(255), nullable=False)
    is_active: Mapped[bool] = mapped_column(Boolean, default=True, nullable=False)
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        default=lambda: datetime.now(UTC),
        nullable=False,
    )
    updated_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        default=lambda: datetime.now(UTC),
        onupdate=lambda: datetime.now(UTC),
        nullable=False,
    )

    profile: Mapped["Profile | None"] = relationship(
        "Profile",
        back_populates="user",
        uselist=False,
        cascade="all, delete-orphan",
    )

9.4 这段模型代码的关键点

username 唯一

因为登录时通常会按用户名查用户。

password_hash

这里故意不叫 password。

这是在提醒你:

  • 数据库存的是哈希值
  • 不是明文密码

profile 关系 

profile: Mapped["Profile | None"] = relationship(...)

作用:

  • 告诉 SQLAlchemy，这个用户有一个 profile
  • 后面你可以通过 user.profile 直接访问

10.3 app/models/profile.py

完整代码如下:

from __future__ import annotations

from datetime import UTC, datetime

from sqlalchemy import DateTime, Float, ForeignKey, Integer, String, Text
from sqlalchemy.orm import Mapped, mapped_column, relationship

from app.db.session import Base


class Profile(Base):
    __tablename__ = "profiles"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, index=True)
    user_id: Mapped[int] = mapped_column(
        ForeignKey("users.id", ondelete="CASCADE"),
        unique=True,
        index=True,
        nullable=False,
    )
    name: Mapped[str] = mapped_column(String(50), nullable=False)
    age: Mapped[int | None] = mapped_column(Integer, nullable=True)
    gender: Mapped[str | None] = mapped_column(String(20), nullable=True)
    height: Mapped[float | None] = mapped_column(Float, nullable=True)
    weight: Mapped[float | None] = mapped_column(Float, nullable=True)
    chronic_history: Mapped[str | None] = mapped_column(Text, nullable=True)
    allergy_history: Mapped[str | None] = mapped_column(Text, nullable=True)
    emergency_contact: Mapped[str | None] = mapped_column(String(100), nullable=True)
    remark: Mapped[str | None] = mapped_column(Text, nullable=True)
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        default=lambda: datetime.now(UTC),
        nullable=False,
    )
    updated_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        default=lambda: datetime.now(UTC),
        onupdate=lambda: datetime.now(UTC),
        nullable=False,
    )

    user: Mapped["User"] = relationship("User", back_populates="profile")

10.4 这段 Profile 模型最关键的一行 

user_id: Mapped[int] = mapped_column(..., unique=True, nullable=False)

它的含义是:

  • 每个 profile 必须属于某个 user
  • 一个 user 只能有一个 profile

这就是第二阶段最关键的数据约束。

12. Schema 层怎么拆

第二阶段最容易犯的错，就是把“数据库模型”和“接口数据结构”混在一起。

你要记住:

  • models/ 是数据库表
  • schemas/ 是接口收发的数据结构

它们不是一回事。

13. Auth 模块的 Schema

13.1 app/schemas/auth.py

完整代码如下:

from pydantic import BaseModel, EmailStr, Field, model_validator


class UserRegisterRequest(BaseModel):
    username: str = Field(..., min_length=3, max_length=50)
    email: EmailStr | None = None
    phone: str | None = Field(default=None, min_length=11, max_length=20)
    password: str = Field(..., min_length=8, max_length=32)
    confirm_password: str = Field(..., min_length=8, max_length=32)

    @model_validator(mode="after")
    def validate_passwords(self):
        if self.password != self.confirm_password:
            raise ValueError("password and confirm_password do not match")
        return self


class Token(BaseModel):
    access_token: str
    token_type: str = "bearer"


class TokenPayload(BaseModel):
    sub: str
    exp: int

13.2 每个类的作用

UserRegisterRequest

负责接收注册请求体。

这里面最重要的是:

@model_validator(mode="after")

作用:

  • 防止用户两次输入的密码不一致

Token

这是登录成功后返回的数据结构。

它必须长这样:

{
  "access_token": "xxxxx",
  "token_type": "bearer"
}

因为 FastAPI 的 OAuth2 Bearer 文档授权按钮默认就认识这种结构。

TokenPayload

这是“解码 JWT 后，我期望拿到什么字段”的结构描述。

14.2 为什么 UserRead 里没有 password_hash

因为密码哈希虽然不等于明文密码，但它仍然属于敏感信息。

所以:

  • 数据库存
  • 后端服务内部用
  • 接口绝不回给前端

这就是 schema 分层的意义之一。

15.2 为什么 ProfileCreate 和 ProfileUpdate 要分开

因为它们的语义不同:

  • Create 通常要求必填字段更完整
  • Update 一般允许只改部分字段

如果你把它们混成一个类，后面更新接口就会很别扭。

16. Auth 模块的核心: 密码哈希 + JWT

第二阶段最核心的安全逻辑，都放在 app/core/security.py。

16.1 app/core/security.py

完整代码如下:

from datetime import datetime, timedelta, timezone

import jwt
from fastapi.security import OAuth2PasswordBearer
from jwt.exceptions import InvalidTokenError
from pwdlib import PasswordHash

from app.core.config import settings


password_hash = PasswordHash.recommended()

oauth2_scheme = OAuth2PasswordBearer(
    tokenUrl=f"{settings.API_V1_PREFIX}/auth/login"
)


def hash_password(password: str) -> str:
    return password_hash.hash(password)


def verify_password(plain_password: str, hashed_password: str) -> bool:
    return password_hash.verify(plain_password, hashed_password)


def create_access_token(subject: str | int, expires_minutes: int | None = None) -> str:
    expire = datetime.now(timezone.utc) + timedelta(
        minutes=expires_minutes or settings.JWT_ACCESS_TOKEN_EXPIRE_MINUTES
    )
    payload = {
        "sub": str(subject),
        "exp": expire,
    }
    return jwt.encode(
        payload,
        settings.JWT_SECRET_KEY,
        algorithm=settings.JWT_ALGORITHM,
    )


def decode_access_token(token: str) -> dict:
    return jwt.decode(
        token,
        settings.JWT_SECRET_KEY,
        algorithms=[settings.JWT_ALGORITHM],
    )


__all__ = [
    "InvalidTokenError",
    "oauth2_scheme",
    "hash_password",
    "verify_password",
    "create_access_token",
    "decode_access_token",
]

16.2 这几个函数分别做什么

hash_password

注册时把明文密码变成哈希值。

verify_password

登录时验证:

  • 用户输入的明文密码
  • 是否能匹配数据库里的哈希值

create_access_token

登录成功后签发 JWT。

decode_access_token

访问受保护接口时，解析 JWT。

16.3 为什么这里用 sub

JWT 里最常用来表示“当前主体是谁”的字段就是 sub。

本教程建议把 user.id 放进去。

好处:

  1. 主键最稳定
  2. 用户以后改用户名也不会影响 token 识别

16.4 为什么 tokenUrl 要写成 /api/v1/auth/login

因为 OAuth2PasswordBearer 需要知道:

  • Swagger 文档的授权按钮应该去哪里拿 token

你如果写错了:

  • 文档里的 Authorize 按钮就会失效

17. 当前用户依赖注入

这一层是把“Bearer Token -> 当前用户对象”真正串起来的关键。

17.1 app/api/deps.py

完整代码如下:

from fastapi import Depends
from sqlalchemy.orm import Session

from app.core.exception import BusinessException
from app.core.security import InvalidTokenError, decode_access_token, oauth2_scheme
from app.db.session import get_db
from app.models.user import User


def get_current_user(
    db: Session = Depends(get_db),
    token: str = Depends(oauth2_scheme),
) -> User:
    credentials_error = BusinessException(
        code=40100,
        message="invalid or expired token",
        status_code=401,
    )

    try:
        payload = decode_access_token(token)
    except InvalidTokenError:
        raise credentials_error

    user_id = payload.get("sub")
    if not user_id:
        raise credentials_error

    user = db.get(User, int(user_id))
    if not user:
        raise credentials_error

    return user


def get_current_active_user(
    current_user: User = Depends(get_current_user),
) -> User:
    if not current_user.is_active:
        raise BusinessException(
            code=40300,
            message="inactive user",
            status_code=403,
        )
    return current_user


__all__ = ["get_db", "get_current_user", "get_current_active_user"]

17.2 这两个依赖分别是干什么的

get_current_user

负责:

  1. 从请求头拿 Bearer Token
  2. 解码 Token
  3. 从数据库拿用户
  4. 返回当前用户对象

get_current_active_user

负责:

  1. 在 get_current_user 基础上
  2. 再检查账号是否可用

所以后面需要保护的接口，基本都依赖这个:

current_user: User = Depends(get_current_active_user)

这就是第二阶段最重要的“路由鉴权入口”。

18. Service 层怎么拆才不乱

这阶段推荐至少拆 3 个 service:

  1. AuthService
  2. UserService
  3. ProfileService

这样做的原因:

  1. 路由层只负责接收请求和返回响应
  2. 真正的业务校验放 service
  3. 后面你要改逻辑，不用在路由里到处找

19. AuthService

19.1 app/services/auth_service.py

完整代码如下:

from sqlalchemy import select
from sqlalchemy.orm import Session

from app.core.exception import BusinessException
from app.core.security import create_access_token, hash_password, verify_password
from app.models.user import User
from app.schemas.auth import Token, UserRegisterRequest


class AuthService:
    def get_by_username(self, db: Session, username: str) -> User | None:
        return db.scalar(select(User).where(User.username == username))

    def get_by_email(self, db: Session, email: str) -> User | None:
        return db.scalar(select(User).where(User.email == email))

    def get_by_phone(self, db: Session, phone: str) -> User | None:
        return db.scalar(select(User).where(User.phone == phone))

    def register(self, db: Session, payload: UserRegisterRequest) -> User:
        if self.get_by_username(db, payload.username):
            raise BusinessException(
                code=40001,
                message="username already exists",
                status_code=400,
            )

        if payload.email and self.get_by_email(db, payload.email):
            raise BusinessException(
                code=40002,
                message="email already exists",
                status_code=400,
            )

        if payload.phone and self.get_by_phone(db, payload.phone):
            raise BusinessException(
                code=40003,
                message="phone already exists",
                status_code=400,
            )

        user = User(
            username=payload.username,
            email=payload.email,
            phone=payload.phone,
            password_hash=hash_password(payload.password),
            is_active=True,
        )

        db.add(user)
        db.commit()
        db.refresh(user)
        return user

    def authenticate(self, db: Session, username: str, password: str) -> User:
        user = self.get_by_username(db, username)
        if not user:
            raise BusinessException(
                code=40101,
                message="incorrect username or password",
                status_code=401,
            )

        if not verify_password(password, user.password_hash):
            raise BusinessException(
                code=40101,
                message="incorrect username or password",
                status_code=401,
            )

        if not user.is_active:
            raise BusinessException(
                code=40301,
                message="user is inactive",
                status_code=403,
            )

        return user

    def login(self, db: Session, username: str, password: str) -> Token:
        user = self.authenticate(db, username, password)
        access_token = create_access_token(subject=user.id)
        return Token(access_token=access_token, token_type="bearer")

from sqlalchemy.orm import Session

from app.core.exception import BusinessException
from app.models.user import User

class UserService:
    def update_basic_info(
        self,
        db: Session,
        current_user: User,
        payload: UserUpdate,
    ) -> User:
        update_data = payload.model_dump(exclude_unset=True)

        if "username" in update_data and update_data["username"] != current_user.username:
            exists = db.scalar(select(User).where(User.username == update_data["username"]))
            if exists:
                raise BusinessException(
                    code=40004,
                    message="username already exists",
                    status_code=400,
                )

        if "email" in update_data and update_data["email"] != current_user.email:
            exists = db.scalar(select(User).where(User.email == update_data["email"]))
            if exists:

                )

            if exists:
                raise BusinessException(
                    code=40006,
                    message="phone already exists",
                    status_code=400,
                )

        for field, value in update_data.items():
            setattr(current_user, field, value)

        db.add(current_user)
        db.commit()
        db.refresh(current_user)
        return current_user

from sqlalchemy import select
from sqlalchemy.orm import Session

from app.core.exception import BusinessException
from app.models.profile import Profile
from app.schemas.profile import ProfileCreate, ProfileUpdate


class ProfileService:
    def get_by_user_id(self, db: Session, user_id: int) -> Profile | None:
        return db.scalar(select(Profile).where(Profile.user_id == user_id))

    def create_for_user(
        self,
        db: Session,
        user_id: int,
        payload: ProfileCreate,
    ) -> Profile:
        if self.get_by_user_id(db, user_id):
            raise BusinessException(
                code=40011,
                message="profile already exists",
                status_code=400,
            )

        profile = Profile(
            user_id=user_id,
            **payload.model_dump(),
        )
        db.add(profile)
        db.commit()
        db.refresh(profile)
        return profile

    def get_or_raise(self, db: Session, user_id: int) -> Profile:
        profile = self.get_by_user_id(db, user_id)
        if not profile:
            raise BusinessException(
                code=40411,
                message="profile not found",
                status_code=404,
            )
        return profile

    def update_for_user(
        self,
        db: Session,
        user_id: int,
        payload: ProfileUpdate,
    ) -> Profile:
        profile = self.get_or_raise(db, user_id)
        update_data = payload.model_dump(exclude_unset=True)

        for field, value in update_data.items():
            setattr(profile, field, value)

        db.add(profile)
        db.commit()
        db.refresh(profile)
        return profile

@router.post(
    "/register",
    response_model=ApiResponse[UserRead],
    status_code=status.HTTP_201_CREATED,
)

    payload: UserRegisterRequest,
    db: Session = Depends(get_db),

        data=UserRead.model_validate(user),
        message="register success",
    )


@router.post(

    response_model=Token,
)

    db: Session = Depends(get_db),
):
    return auth_service.login(db, form_data.username, form_data.password)

23.2 为什么登录接口这里没有包 code/message/data

这是第二阶段一个很重要的工程细节。

因为 FastAPI 的 OAuth2 文档授权按钮，默认要求 token 接口返回:

{
  "access_token": "xxx",
  "token_type": "bearer"
}

如果你把它包成:

{
  "code": 0,
  "message": "success",
  "data": {
    "access_token": "xxx",
    "token_type": "bearer"
  }
}

Swagger 的 Authorize 按钮通常就不能直接用了。

所以:

  • 普通业务接口继续统一响应
  • 登录 token 接口允许作为例外直接返回 token 对象

这是合理的，不是架构错误。

23.3 登录接口为什么用表单而不是 JSON

因为 OAuth2PasswordRequestForm 遵循 OAuth2 password flow 规范:

  • 字段名必须叫 username
  • 字段名必须叫 password
  • 提交方式是 application/x-www-form-urlencoded

这样你在 /docs 里可以直接点 Authorize 来测试受保护接口。

30. 第二阶段常见错误

错误 1: 密码存明文

这是绝对不能做的。

正确做法:

  • 注册时 hash_password
  • 登录时 verify_password

错误 2: JWT 里塞太多数据

不要把用户完整信息都塞进 token。

第二阶段最推荐只放:

{
  "sub": "1",
  "exp": 9999999999
}

因为:

  1. Token 应该尽量小
  2. 用户信息以数据库为准

错误 3: 用用户名做 profile 外键

不要这样做。

正确做法:

  • profiles.user_id -> users.id

因为用户名未来可能改。

错误 4: 让前端传 user_id 来操作自己的 profile

这会带来权限风险。

正确做法:

  • 从 Bearer Token 解析当前用户
  • 直接操作 /profiles/me

错误 5: 登录接口也强行包统一响应

如果你要保留 Swagger 的 OAuth2 Authorize 按钮可用，登录 token 接口最好直接返回:

{
  "access_token": "...",
  "token_type": "bearer"
}

错误 6: UserRead 把 password_hash 回给前端

不要这么做。

这属于敏感字段泄漏。

错误 7: Profile 不加唯一约束

如果 user_id 不唯一，你后面很容易产生多个 profile，业务逻辑会变得混乱。

错误 8: 把大量校验都写在路由里

路由一旦变胖，后面非常难维护。

正确做法:

  • 路由只负责收发
  • 业务校验放 service

错误 9: 当前用户依赖注入只解析 token，不查数据库

如果 token 里的人已经被删了，或者被禁用了，你不能继续信任它。

正确做法:

  • 解析 token 后还要去数据库查用户

错误 10: Profile 和 User 职责混在一起

比如把:

  • chronic_history
  • weight
  • remark

这类健康档案字段直接塞进 users 表。

这样后面一定会越来越乱。

31. 结合你当前仓库，第二阶段最值得注意的地方

你当前仓库已经有这些东西:

  1. backend/app/api/routes/auth.py 雏形
  2. backend/app/schemas/auth.py 雏形
  3. backend/app/db/models/profile.py 雏形
  4. backend/app/repositories/profile_repo.py 雏形

但这些还没有形成完整的标准链路。

第二阶段你自己改的时候，最建议优先做下面这些统一动作:

  1. 明确引入真正的 User 表
  2. 把认证逻辑收进 core/security.py + services/auth_service.py
  3. 把“当前用户”依赖注入补完整
  4. 给 Profile 加上 user_id
  5. 把 profile 改成“每个用户自己的档案”

31.1 你当前的 profile.py 和本教程的差异

你现在的 profile 模型字段更丰富，比如:

  1. nickname
  2. preferred_call_name
  3. living_alone
  4. medicine_note
  5. usual_sleep_time

这些字段不是不能保留。

只是第二阶段最推荐先收缩成最小可用结构:

  1. 先把 user_id 关系打通
  2. 先确保创建、查看、更新的流程稳定
  3. 再逐步把扩展字段加回来

换句话说:

  • 本教程是“第二阶段最稳的基准版”
  • 不是说你当前那些字段不对

33. 一句话总结第二阶段

第二阶段的本质，不是在“多写几个接口”，而是在给后面所有业务建立一个稳定前提:

  • 我知道当前是谁
  • 我知道这个人的账号信息
  • 我知道这个人的基础健康档案

只有这个前提成立，第三阶段往后的:

  1. 健康记录
  2. 聊天建议
  3. 主动提醒
  4. 个性化推荐

才有稳定的归属和上下文。
