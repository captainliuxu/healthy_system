# 06. 业务建模：老人档案、健康记录、聊天服务

本文件开始讲“真实业务怎么抽象”。  
这部分很重要，因为项目不是只有接口，还要有业务对象和业务关系。

---

## 1. 什么叫业务建模

业务建模不是数据库建表专属概念。  
你现在即使先不用数据库，也应该思考：

- 系统里有哪些核心对象
- 这些对象之间是什么关系
- 每个接口到底在操作什么对象

你的项目核心业务对象至少有 3 个：

1. 老人档案 `Profile`
2. 健康记录 `Record`
3. 聊天会话 / 建议 `Chat`

---

## 2. 老人档案应该包含什么

第一版建议先包含：

- `id`
- `name`
- `age`
- `gender`
- `disease_history`
- `emergency_contact`

以后可扩展：
- 身高体重
- 常用药物
- 过敏史
- 家属联系人
- 居住信息
- 风险等级
- 标签

### 为什么档案要先做
因为：
- 记录是挂在某个老人上的
- 聊天建议以后也应该知道“对谁说”
- 定时提醒也要绑定用户对象

也就是说，档案是整个系统的数据根。

---

## 3. 健康记录怎么抽象

很多人会一上来把：
- 血压
- 血糖
- 心率
- 睡眠
- 用药
全部分成很多张表和很多 schema。

这对你现在太重。

### 第一阶段更推荐统一抽象
先统一成一个通用记录模型：

- `id`
- `profile_id`
- `record_type`
- `value`
- `note`
- `created_at`

例如：

```json
{
  "profile_id": 1,
  "record_type": "blood_pressure",
  "value": "120/80",
  "note": "早晨测量"
}
```

### 这样做的好处
- 先快速跑通记录功能
- 前后端简单
- 以后再逐步细分

### 以后怎么升级
以后可以继续拆成更明确的记录类型：

#### 血压记录
- systolic
- diastolic

#### 血糖记录
- blood_sugar_value
- fasting / after_meal

#### 心率记录
- bpm

但第一阶段先别急着拆。

---

## 4. 聊天服务不只是“把用户话发给模型”

聊天服务真正的业务意义是：

- 根据项目定位设定助手角色
- 根据业务场景限制回答风格
- 后续接入档案和记录信息
- 给用户提供健康管理建议

### 第一阶段
你先做到：
- system prompt 固定成健康管理助手
- 接收一个 message
- 返回一段 reply

### 第二阶段
可以逐步升级：
- 根据老人档案动态拼 prompt
- 根据最近记录给出建议
- 加上下文历史
- 加敏感词、风险词处理

---

## 5. service 层的真正职责

很多人会觉得 service 层只是“把路由里的代码挪出去”。  
这只是最初级的一步。

真正的 service 层应该承担这些职责：

### 5.1 业务规则
比如：
- 先检查老人档案是否存在
- 再允许创建记录

### 5.2 业务流程编排
比如：
- 创建记录后自动生成简短提示
- 聊天时先读档案摘要再调模型

### 5.3 外部依赖调用
比如：
- 调用大模型
- 调用数据库
- 调用短信 / 提醒服务

### 5.4 对上层屏蔽实现细节
路由层不需要知道模型是怎么调的，也不需要知道记录是怎么存的。

## 6. 一个更贴近真实项目的聊天服务抽象

示例：

```python
class ChatService:
    def build_system_prompt(self, profile=None):
        base_prompt = "你是一个慢病健康管理助手，回答简洁、温和。"
        if profile:
            base_prompt += f" 当前服务对象：{profile['name']}，年龄 {profile['age']}。"
        return base_prompt

    def reply(self, message: str, profile=None):
        system_prompt = self.build_system_prompt(profile)
        # 调模型
        ...
```

### 这样设计的意义
以后你接更多上下文时，不用改路由，只改 service。

## 7. 记录服务应该具备哪些“业务意识”

一个成熟一点的 `record_service` 不应该只是 append 数据。  
至少要开始考虑：

- 对应的 `profile_id` 是否存在
- `record_type` 是否允许
- 是否需要时间格式标准化
- 是否需要统计最近几条记录

### 允许的记录类型示例
```python
ALLOWED_RECORD_TYPES = {
    "blood_pressure",
    "blood_sugar",
    "heart_rate",
    "sleep",
    "medicine"
}
```

---

## 8. 记录服务未来可以扩展出的能力

### 能力 1：按类型筛选
- 查询某老人所有血压记录

### 能力 2：按时间区间筛选
- 查询最近 7 天记录

### 能力 3：统计摘要
- 最近血压平均值
- 最近睡眠情况

### 能力 4：异常识别
- 多次偏高给出提醒

### 能力 5：联动聊天服务
- “最近血压偏高，建议咨询医生并监测作息”

---

## 9. 你的项目为什么特别适合先做“对象—服务”这种抽象

因为它不是纯 CRUD 系统。  
它有明显的“智能辅助”特征：

- 有健康管理场景
- 有面向老人和家属的业务语义
- 有记录和建议的联动空间

所以你现在就应该建立“业务对象 + 业务服务”的思维，而不是只停留在“增删改查接口”。

---

## 10. 推荐的业务主线

你后续写功能时，尽量按这条主线展开：

### 第一步：先有老人对象
创建档案

### 第二步：围绕老人积累数据
增加健康记录

### 第三步：基于数据做智能反馈
聊天与建议

### 第四步：未来再做长期机制
提醒、统计、趋势、RAG、后台管理

---

## 11. 一句话总结本文件

> 一个好的后端不是“接口列表”，而是围绕核心业务对象组织起来的一套服务系统。对你这个项目来说，最核心的对象就是档案、记录和聊天建议。
