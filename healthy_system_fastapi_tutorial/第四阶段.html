Healthy System FastAPI 第四阶段详细教程

1. 这一阶段到底要解决什么

第四阶段先做一件很重要的事:

把聊天相关的数据结构先稳定下来，也就是先把:

  1. Conversation
  2. Message

这两个模块建好。

注意，这一阶段先不要急着接大模型。

因为聊天功能本质上至少有两层:

  1. 会话层 conversation
  2. 消息层 message

如果这两层的数据模型不先稳定，后面你一接模型调用，就会不停返工。

所以第四阶段的目标不是“先聊起来”，而是:

先把聊天数据真正存得清楚、查得清楚、关联得清楚。

2. 第四阶段完成后的验收标准

做到下面这些，就算第四阶段合格:

  1. conversation 与 user 正确关联
  2. message 与 conversation 正确关联
  3. 用户可以创建会话
  4. 用户可以查看自己的会话列表
  5. 用户可以查看某个会话详情
  6. 用户可以修改会话标题
  7. 用户可以删除会话
  8. 用户可以查看某个会话下的历史消息
  9. 消息可以手动插入并落库
  10. 用户不能查看别人的会话和消息

3. 为什么这一阶段先做数据结构，不先接大模型

因为你后面真正的聊天接口，本质上一定会依赖这两张表:

  1. conversations
  2. messages

你以后无论是:

  1. 普通问答
  2. 上下文续聊
  3. 多轮记忆
  4. 历史记录回放
  5. 主动消息回显

都离不开这两层数据。

如果你现在不先把它们建稳，后面接大模型时会不断遇到这些问题:

  1. 消息不知道该挂到哪个会话下
  2. 会话不知道归哪个用户
  3. 历史消息查不出来
  4. 想补标题、状态、角色时又得返工改表

所以第四阶段最正确的思路是:

先把数据库结构和基础 CRUD 做好，再谈聊天主链路。

4. 这一阶段先不做什么

这一阶段先不要做这些内容:

  1. 不做流式输出
  2. 不做 WebSocket
  3. 不做复杂 prompt 编排
  4. 不做上下文截断策略
  5. 不做自动生成会话标题
  6. 不做 LLM 调用封装
  7. 不做 /chat 真正问答接口

这一阶段你只要做到:

  1. 用户能创建会话
  2. 用户能查历史消息
  3. 消息能落库

就已经完成目标了。

5. 先说明你当前项目里的草稿文件

你现在仓库里已经有这些草稿文件:

  1. backend/app/db/models/conversation.py
  2. backend/app/db/models/message.py
  3. backend/app/schemas/conversation.py
  4. backend/app/schemas/chat.py

但你前两阶段、第三阶段已经逐步统一成了正式结构:

  1. ORM 模型放在 app/models/
  2. Alembic 通过 app/db/base.py 找模型
  3. 路由放在 app/api/routes/
  4. service 放在 app/services/

所以第四阶段也要继续统一成这套正式结构。

5.1 这一阶段的处理原则

  1. 正式版 Conversation 放到 app/models/conversation.py
  2. 正式版 Message 放到 app/models/message.py
  3. app/db/models/conversation.py 和 app/db/models/message.py 先不要再继续用
  4. app/schemas/conversation.py 可以直接重写
  5. 新增 app/schemas/message.py
  6. app/schemas/chat.py 这一阶段先不要动

5.2 为什么这里不继续沿用 profile_id

你现在草稿里的 conversation.py 用的是 profile_id。

但第四阶段的验收标准写得很明确:

conversation 与 user 正确关联

所以这一阶段推荐直接改成:

Conversation -> user_id

这样做好处是:

  1. 和第二阶段的认证体系直接对上
  2. 当前用户权限判断最简单
  3. 这一阶段更容易演示
  4. 不依赖 profile 是否已创建

以后如果你要做“一个账号管理多个不同聊天对象”，再往 profile_id 扩展也不迟。

6. 第四阶段推荐的数据设计

6.1 Conversation 表

推荐字段:

字段 类型 说明
id int 主键
user_id int 归属用户
title str 会话标题
status str 会话状态
created_at datetime 创建时间
updated_at datetime 更新时间

为什么这里加 status

虽然这一阶段你还不会真的用到很多状态流转，但建议先把 status 留好。

因为以后你很可能会需要:

  1. 激活中的会话
  2. 归档会话
  3. 已关闭会话

第四阶段先保留最小状态字段，后面就不容易返工。

推荐先用:

  1. active
  2. archived

但这一阶段先只默认写 active，不做复杂状态切换接口。

6.2 Message 表

推荐字段:

字段 类型 说明
id int 主键
conversation_id int 所属会话
role str 角色
content text 消息内容
message_type str 消息类型
created_at datetime 创建时间

role 推荐值

  1. user
  2. assistant
  3. system

这三个值是聊天系统里最经典、最通用的一套角色设计。

第四阶段先固定成这三个值，后面你接大模型时就不用重构历史消息结构了。

message_type 为什么先保留

虽然这一阶段大多数消息都只是普通文本，但这个字段建议先留好。

后面你可能会扩展:

  1. text
  2. summary
  3. event
  4. reminder

第四阶段先统一默认成 text 就够了。

conversation -> messages

9. 开发顺序建议

这个顺序不要乱:

  1. 先做 Conversation 模型
  2. 再做 Message 模型
  3. 再补 app/models/__init__.py
  4. 再补 app/db/base.py
  5. 再写 Conversation 的 schema
  6. 再写 Message 的 schema
  7. 再写 ConversationService
  8. 再写 MessageService
  9. 再写 conversations.py
  10. 再写 messages.py
  11. 再补总路由 app/api/router.py
  12. 最后做 Alembic migration 和 Swagger 验收

原因:

  1. 没有模型，后面都没法写
  2. 没有 schema，请求和响应边界不清楚
  3. 没有 service，路由里会堆满业务逻辑
  4. 没有 base 导入，Alembic 很可能生成空 migration

11. 先建 Conversation ORM 模型

11.1 新建 app/models/conversation.py

完整代码如下:

from __future__ import annotations

from datetime import UTC, datetime

from sqlalchemy import DateTime, ForeignKey, Integer, String
from sqlalchemy.orm import Mapped, mapped_column

from app.db.session import Base


class Conversation(Base):
    __tablename__ = "conversations"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, index=True)
    user_id: Mapped[int] = mapped_column(
        ForeignKey("users.id", ondelete="CASCADE"),
        index=True,
        nullable=False,
    )
    title: Mapped[str] = mapped_column(String(100), nullable=False, default="New Conversation")
    status: Mapped[str] = mapped_column(String(20), nullable=False, default="active", index=True)
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        default=lambda: datetime.now(UTC),
        nullable=False,
    )
    updated_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        default=lambda: datetime.now(UTC),
        onupdate=lambda: datetime.now(UTC),
        nullable=False,
    )

11.2 这份模型代码在做什么

  1. user_id 表示这个会话属于哪个用户
  2. title 是会话标题
  3. status 是会话状态
  4. created_at 是创建时间
  5. updated_at 是最后更新时间

11.3 这里为什么先不加 relationship

这一阶段的核心目标是:

把会话和消息的数据结构先跑通。

所以第四阶段先只用外键表达关联关系就够了。

也就是说:

  1. Conversation.user_id -> users.id
  2. Message.conversation_id -> conversations.id

这已经足够支撑 CRUD 和历史消息查询。

后面如果你熟悉了 SQLAlchemy relationship，再补也不迟。

schema 要解决的核心问题是:

14.1 重写 app/schemas/conversation.py

你当前仓库里已经有这个文件，但内容还是很早期的草稿版。
第四阶段建议直接整文件替换。

完整代码如下:

from datetime import datetime
from enum import Enum

from pydantic import BaseModel, ConfigDict, Field


class ConversationStatus(str, Enum):
    active = "active"
    archived = "archived"


class ConversationCreate(BaseModel):
    title: str | None = Field(default=None, max_length=100)


class ConversationTitleUpdate(BaseModel):
    title: str = Field(..., min_length=1, max_length=100)


class ConversationRead(BaseModel):
    id: int
    user_id: int
    title: str
    status: ConversationStatus
    created_at: datetime
    updated_at: datetime

    model_config = ConfigDict(from_attributes=True)


class ConversationListData(BaseModel):
    items: list[ConversationRead]

14.2 每个 schema 的作用

ConversationStatus

作用:

把会话状态限制在固定范围内。

这一阶段先保留两个最小值:

  1. active
  2. archived

ConversationCreate

作用:

创建会话时使用。

这里让标题可不传，是为了让你在第四阶段可以快速创建一个空标题会话。

如果用户没传标题，后端会自动补默认值。

ConversationTitleUpdate

作用:

专门给“修改会话标题”接口使用。

这里把 title 设成必填，是因为这个接口只做一件事:

改标题。

ConversationRead

作用:

返回给前端的会话读取结构。

ConversationListData

作用:

统一会话列表的返回数据结构。

16. 写 ConversationService

service 层负责:

  1. 会话归属校验
  2. 创建会话
  3. 查询会话列表
  4. 查询会话详情
  5. 修改标题
  6. 删除会话

16.1 新建 app/services/conversation_service.py

完整代码如下:

from sqlalchemy import delete, select
from sqlalchemy.orm import Session

from app.core.exception import BusinessException
from app.models.conversation import Conversation
from app.models.message import Message
from app.schemas.conversation import ConversationCreate, ConversationTitleUpdate


class ConversationService:
    def create_for_user(
        self,
        db: Session,
        user_id: int,
        payload: ConversationCreate,
    ) -> Conversation:
        title = (payload.title or "").strip() or "New Conversation"

        conversation = Conversation(
            user_id=user_id,
            title=title,
            status="active",
        )
        db.add(conversation)
        db.commit()
        db.refresh(conversation)
        return conversation

    def list_for_user(
        self,
        db: Session,
        user_id: int,
    ) -> list[Conversation]:
        stmt = (
            select(Conversation)
            .where(Conversation.user_id == user_id)
            .order_by(Conversation.updated_at.desc(), Conversation.id.desc())
        )
        return list(db.scalars(stmt).all())

    def get_or_raise(
        self,
        db: Session,
        user_id: int,
        conversation_id: int,
    ) -> Conversation:
        stmt = select(Conversation).where(
            Conversation.id == conversation_id,
            Conversation.user_id == user_id,
        )
        conversation = db.scalar(stmt)
        if not conversation:
            raise BusinessException(
                code=40431,
                message="conversation not found",
                status_code=404,
            )
        return conversation

    def update_title_for_user(
        self,
        db: Session,
        user_id: int,
        conversation_id: int,
        payload: ConversationTitleUpdate,
    ) -> Conversation:
        conversation = self.get_or_raise(db, user_id, conversation_id)
        title = payload.title.strip()

        if not title:
            raise BusinessException(
                code=40031,
                message="title cannot be empty",
                status_code=400,
            )

        conversation.title = title
        db.add(conversation)
        db.commit()
        db.refresh(conversation)
        return conversation

    def delete_for_user(
        self,
        db: Session,
        user_id: int,
        conversation_id: int,
    ) -> None:
        conversation = self.get_or_raise(db, user_id, conversation_id)

        db.execute(
            delete(Message).where(Message.conversation_id == conversation.id)
        )
        db.delete(conversation)
        db.commit()


conversation_service = ConversationService()

16.2 这份 service 代码最关键的点

1. 所有查询都必须带 user_id

这一阶段最重要的权限原则就是:

会话必须归当前用户所有。

所以不管是详情、改标题还是删除，都不能只写:

db.get(Conversation, conversation_id)

因为这样没有做用户归属限制。

正确思路必须是:

where(Conversation.id == conversation_id, Conversation.user_id == user_id)

2. 创建会话时，user_id 不能来自前端

它必须来自当前登录用户。

也就是说:

  1. 前端负责传标题
  2. 后端负责把当前用户 id 写进会话

3. 删除会话时，为什么这里手动先删消息

你当前项目默认数据库是 SQLite。
而 SQLite 的外键级联删除行为，很多初学者环境里并不会默认稳定生效。

所以为了让你在第四阶段少踩坑，这里直接在 service 里手动删:

  1. 先删 messages
  2. 再删 conversation

这样逻辑最稳，也最容易理解。

17. 写 MessageService

MessageService 主要负责:

  1. 验证会话是否属于当前用户
  2. 查询某个会话下的消息列表
  3. 手动插入消息用于调试

17.1 新建 app/services/message_service.py

完整代码如下:

from datetime import UTC, datetime

from sqlalchemy import select
from sqlalchemy.orm import Session

from app.core.exception import BusinessException
from app.models.message import Message
from app.schemas.message import MessageCreate
from app.services.conversation_service import conversation_service


class MessageService:
    def list_for_conversation(
        self,
        db: Session,
        user_id: int,
        conversation_id: int,
    ) -> list[Message]:
        conversation_service.get_or_raise(db, user_id, conversation_id)

        stmt = (
            select(Message)
            .where(Message.conversation_id == conversation_id)
            .order_by(Message.created_at.asc(), Message.id.asc())
        )
        return list(db.scalars(stmt).all())

    def create_debug_for_conversation(
        self,
        db: Session,
        user_id: int,
        conversation_id: int,
        payload: MessageCreate,
    ) -> Message:
        conversation = conversation_service.get_or_raise(
            db, user_id, conversation_id
        )

        content = payload.content.strip()
        if not content:
            raise BusinessException(
                code=40041,
                message="content cannot be empty",
                status_code=400,
            )

        message_type = payload.message_type.strip()
        if not message_type:
            raise BusinessException(
                code=40042,
                message="message_type cannot be empty",
                status_code=400,
            )

        message = Message(
            conversation_id=conversation_id,
            role=payload.role.value,
            content=content,
            message_type=message_type,
        )

        db.add(message)

        conversation.updated_at = datetime.now(UTC)
        db.add(conversation)

        db.commit()
        db.refresh(message)
        return message


message_service = MessageService()

17.2 这份 service 代码最关键的点

1. 先验证会话归属，再查消息

消息本身没有直接挂 user_id，所以权限判断不能直接从 Message 表做。

正确做法是:

先验证:

  1. 这个 conversation_id 存在
  2. 并且它属于当前用户

然后再去查这个会话下面的消息。

2. 为什么插入消息时要更新 conversation.updated_at

这是一个非常容易忽略的点。

如果你新增了一条消息，但不更新会话的 updated_at，那会发生什么?

会话列表排序可能就不准确。

比如:

  1. 会话 A 昨天创建
  2. 会话 B 今天创建
  3. 你今天又往会话 A 里新增了一条消息

按业务直觉，会话 A 应该排到更前面。
所以插入消息时，最好顺手刷新会话的 updated_at。

这一步非常重要。

19.2 这份消息路由为什么这样设计

1. 为什么消息接口挂在会话下面

因为消息一定从属于某个会话。

所以:

/conversations/{conversation_id}/messages

比单独搞一个:

/messages?conversation_id=xxx

更直观，也更不容易绕晕。

2. 为什么手动插入消息接口叫 /debug

因为第四阶段你还没真正接 LLM。

所以这个接口的本质是:

手动往数据库里插消息，方便调试历史消息链路。

它不是最终聊天入口，所以路径里明确标个 /debug 更合适。

25. 这一阶段最容易写错的地方

25.1 继续把 Conversation 设计成挂 profile_id

这是你当前草稿最容易延续下来的问题。

但第四阶段的验收标准要求是:

conversation 与 user 正确关联

所以这一阶段正式版必须是:

Conversation.user_id

25.2 查会话时只按 conversation_id 查，不按 user_id 限制

这是最典型的越权 bug。

错误写法:

conversation = db.get(Conversation, conversation_id)

正确思路:

where(Conversation.id == conversation_id, Conversation.user_id == user_id)

25.3 查消息时直接按 conversation_id 查，没有先校验会话归属

消息表本身没有 user_id，所以你不能直接假设:

这个 conversation_id 一定属于当前用户。

正确流程一定是:

  1. 先校验这个会话归当前用户
  2. 再查该会话下的消息

25.4 插入消息时忘了更新 conversation.updated_at

这会导致会话列表排序不符合实际使用顺序。

25.5 删除会话时只删会话，不删消息

这样很容易留下脏数据。

第四阶段为了稳定起见，直接在 service 里手动先删消息，再删会话。

25.6 现在就急着去写 /chat

第四阶段先不要急。

这一阶段的核心任务只是:

  1. 会话建好
  2. 消息建好
  3. 历史消息能查
  4. 消息能落库

把这一步做好，后面接大模型才顺。
