# 05. 真实开发中的抽象：配置、响应、异常、日志

这一部分非常关键。  
它决定了你的项目以后是“越写越稳”，还是“越写越乱”。

---

## 1. 为什么这些基础抽象这么重要

很多新手写项目时只关注接口能不能返回数据。  
但真实开发里，你还必须考虑：

- 线上报错怎么排查
- 前端收到错误怎么统一处理
- 不同环境怎么切配置
- 业务异常怎么优雅提示
- 日志怎么记录关键链路

这就是为什么你不能只会写 route。

---

## 2. 配置管理：为什么不能在多个文件里到处 `os.getenv`

错误做法：

```python
# chat_service.py
api_key = os.getenv("DASHSCOPE_API_KEY")

# another.py
debug = os.getenv("DEBUG")
```

问题：
- 配置来源散乱
- 不方便统一管理默认值
- 很难知道系统到底依赖哪些配置
- 后面换环境不方便

### 正确做法
统一放在 `core/config.py`。

示例：

```python
from pathlib import Path
from dotenv import load_dotenv
import os

ROOT_DIR = Path(__file__).resolve().parent.parent.parent
load_dotenv(ROOT_DIR / ".env")

APP_NAME = os.getenv("APP_NAME", "healthy_system backend")
APP_ENV = os.getenv("APP_ENV", "dev")
DEBUG = os.getenv("DEBUG", "true").lower() == "true"

DASHSCOPE_API_KEY = os.getenv("DASHSCOPE_API_KEY", "")
ALLOWED_ORIGINS = os.getenv("ALLOWED_ORIGINS", "*")
```

### 真实开发中的进一步做法
以后你可以继续扩展：
- 数据库地址
- Redis 地址
- JWT 密钥
- 上传目录
- 向量库目录

## 3. 统一响应格式：不要让前端每个接口都单独猜

不统一的坏处：
- 前端每个页面都要写一套解析逻辑
- 有的接口返回 `{"msg": ...}`
- 有的返回 `{"message": ...}`
- 有的直接返回对象
- 很难维护

### 推荐格式

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

失败时：

```json
{
  "code": 1001,
  "message": "档案不存在",
  "data": null
}
```

### 为什么这个格式实用
- `code` 适合前端逻辑判断
- `message` 适合 toast / 提示文案
- `data` 适合承载真正返回数据

### 工具函数示例

```python
def success(data=None, message="success", code=0):
    return {
        "code": code,
        "message": message,
        "data": data
    }

def fail(message="fail", code=1, data=None):
    return {
        "code": code,
        "message": message,
        "data": data
    }
```

## 4. 业务异常：别让所有错误都靠 `HTTPException`

`HTTPException` 很有用，但不够表达业务语义。

例如：
- 老人档案不存在
- 健康记录类型非法
- 当前配置缺失
- 模型调用失败
- 用户无权限

这些其实更适合定义成“业务异常”。

### 示例：`core/exceptions.py`

```python
from fastapi import Request
from fastapi.responses import JSONResponse

class BizException(Exception):
    def __init__(self, message: str, code: int = 1000):
        self.message = message
        self.code = code

async def biz_exception_handler(request: Request, exc: BizException):
    return JSONResponse(
        status_code=200,
        content={
            "code": exc.code,
            "message": exc.message,
            "data": None
        }
    )
```

然后在 `main.py` 注册：

```python
from app.core.exceptions import BizException, biz_exception_handler

app.add_exception_handler(BizException, biz_exception_handler)
```

### 使用示例
```python
from app.core.exceptions import BizException

def get_profile(profile_id: int):
    ...
    raise BizException("档案不存在", code=1001)
```

这样前端会收到统一格式，而不是直接拿 404 HTML 或不同结构的报错。

## 5. 为什么很多团队会让“业务错误也返回 200”

这点你以后会遇到。  
有些团队会规定：

- HTTP 200 表示“请求到了服务端并被正常处理”
- 业务成功失败由 `code` 决定

这样前端只要统一判断 `code` 即可。

当然，也有团队会：
- 参数错误用 400
- 未登录用 401
- 无权限用 403
- 资源不存在用 404
- 服务器错误用 500

### 对你现在的建议
第一阶段可以简单一点：
- 系统级明显错误仍可使用 HTTP 状态码
- 业务级错误统一成业务异常

## 6. 日志：为什么不能只靠 `print`

`print` 的问题：
- 没有级别
- 不方便搜索
- 不方便持久化
- 上线后很难排查问题

### 推荐最小日志方案

`core/logging.py`

```python
import logging

def get_logger(name: str = "app"):
    logger = logging.getLogger(name)
    if not logger.handlers:
        logger.setLevel(logging.INFO)

        handler = logging.StreamHandler()
        formatter = logging.Formatter(
            "[%(asctime)s] [%(levelname)s] %(name)s - %(message)s"
        )
        handler.setFormatter(formatter)
        logger.addHandler(handler)

    return logger
```

### 使用示例

```python
from app.core.logging import get_logger

logger = get_logger("chat_service")

def get_chat_reply(message: str):
    logger.info(f"收到聊天请求: {message}")
    ...
```

## 7. 真实开发里哪些地方应该记日志

建议至少记录：

### info 级别
- 服务启动
- 收到关键业务请求
- 成功创建档案
- 成功保存记录

### warning 级别
- 配置缺失
- 查询不到对象
- 非法参数组合

### error / exception 级别
- 第三方调用失败
- 数据库异常
- 未捕获异常

## 8. 模型调用失败时怎么抽象更合理

不要在接口函数里直接：

```python
except Exception as e:
    return {"reply": f"调用失败：{str(e)}"}
```

因为：
- 错误结构不统一
- 前端没法稳定处理
- 业务和错误展示耦合了

### 更好的写法
在 service 层里：
- 记录错误日志
- 抛出业务异常

```python
from app.core.exceptions import BizException
from app.core.logging import get_logger

logger = get_logger("chat_service")

def get_chat_reply(message: str):
    try:
        ...
    except Exception as e:
        logger.exception("模型调用失败")
        raise BizException(f"模型调用失败：{str(e)}", code=2001)
```

## 9. 一个成熟后端最少要具备的“基础工程能力”

即便不接数据库，你也应该尽量具备：

- 统一配置入口
- 统一响应格式
- 统一异常处理
- 基础日志

这四项就是你从“写功能”迈向“写工程”的关键一步。

---

## 10. 一句话总结本文件

> 真正决定项目能不能长期维护的，往往不是业务功能本身，而是你有没有把配置、响应、异常、日志这几件基础设施先打好。
