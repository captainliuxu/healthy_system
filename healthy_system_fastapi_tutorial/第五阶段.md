# Healthy System FastAPI 第五阶段详细教程

## 1. 这一阶段到底要解决什么

第五阶段要做的事情非常关键:

把聊天主链路真正打通。

也就是说，这一阶段你要实现这样一条最小可用链路:

1. 用户发消息
2. 后端校验当前会话是否属于当前用户
3. 保存一条 `user message`
4. 读取最近若干条历史消息
5. 组装给大模型的消息列表
6. 调用大模型
7. 保存一条 `assistant message`
8. 把回复结果返回给前端

这一阶段的意义非常大。

因为到这里为止，你的项目才第一次从:

“静态管理系统”

真正变成:

“会说话、会回复的智能应用”

---

## 2. 第五阶段完成后的验收标准

做到下面这些，就算第五阶段合格:

1. 发一条消息能拿到 AI 回复
2. 用户消息能入库
3. AI 回复能入库
4. 历史消息能回显
5. 会话归属校验正确
6. 用户不能给别人的会话发消息

---

## 3. 为什么第五阶段先做同步版本

这一阶段强烈建议你先做普通同步 HTTP 版本，不要一开始就做流式输出。

也就是说:

前端发一个 HTTP 请求，后端处理完整条链路后，一次性返回结果。

为什么这样做:

1. 好调试
2. 好定位报错
3. 便于先验证数据库链路
4. 逻辑更容易看懂
5. 代码颗粒度更适合你现在学习

如果你一上来就做:

1. SSE
2. WebSocket
3. 流式 token 推送

那你很容易把:

1. 路由问题
2. 数据库问题
3. 模型调用问题
4. 前端展示问题

全搅在一起，最后根本不知道哪一层错了。

所以第五阶段最稳的做法就是:

先把同步版本做通。

## 4. 这一阶段默认建立在第四阶段已经完成的前提上

第五阶段不是从第二阶段直接跳过来的。

它默认你已经完成第四阶段，也就是至少已经有:

因为第五阶段的聊天主链路一定依赖:

1. `Conversation`

所以这一阶段的前提很明确:

先有聊天数据结构，再有聊天主链路。

## 5. 第五阶段推荐实现的最小版本

## 5.1 前端传什么

前端发来:

1. `conversation_id`
2. 用户消息内容

建议请求体最小结构就是:

```json
{
  "conversation_id": 1,
  "content": "我今天血压有点高，应该注意什么？"
}
```

## 5.2 后端做什么

后端最小流程建议固定成:

1. 校验 `conversation` 是否属于当前用户
2. 保存 `user message`
3. 读取最近若干条历史消息
4. 组装 `messages`
5. 调用模型
6. 保存 `assistant message`
7. 返回结果

这个顺序不要乱。

尤其是:

不要先调模型，再保存用户消息。

因为用户消息本来就是对话历史的一部分，应该先写入数据库，再进入后续流程。

你当前仓库里的 `app/schemas/chat.py` 还是比较早期的草稿版本，里面有:

### 11.1 修改 `app/schemas/chat.py`

如果你当前这个文件已经存在，直接整文件替换成下面内容。

完整代码如下:

```python
from datetime import datetime

from pydantic import BaseModel, Field


class ChatSendRequest(BaseModel):
    conversation_id: int = Field(..., ge=1)
    content: str = Field(..., min_length=1, max_length=2000)


class ChatSendData(BaseModel):
    conversation_id: int
    user_message_id: int
    assistant_message_id: int
    reply: str
    replied_at: datetime
```

### 11.2 这份 schema 在做什么

#### `ChatSendRequest`

作用:

定义前端发消息时的请求体。

为什么这里不再允许 `conversation_id` 可选:

因为第五阶段的最小目标是:

在一个已经存在的会话里完成消息往返。

“没有会话就自动建会话”这种能力，可以放到后面再做。

#### `ChatSendData`

作用:

定义后端回复给前端的数据结构。

这里保留:

1. `conversation_id`
2. `user_message_id`
3. `assistant_message_id`
4. `reply`
5. `replied_at`

这样前端拿到结果后:

1. 可以直接显示 AI 回复
2. 也能知道数据库里两条消息各自的 id

## 12. 先补好 `MessageService`

第五阶段非常依赖消息服务，所以建议你把 `app/services/message_service.py` 升级成一个更通用的版本。

如果你第四阶段已经有这个文件，建议整文件替换成下面内容。

### 12.1 修改 `app/services/message_service.py`

完整代码如下:

```python
from datetime import UTC, datetime

from sqlalchemy import select
from sqlalchemy.orm import Session

from app.core.exception import BusinessException
from app.models.message import Message
from app.schemas.message import MessageCreate
from app.services.conversation_service import conversation_service


class MessageService:
    def list_for_conversation(
        self,
        db: Session,
        user_id: int,
        conversation_id: int,
    ) -> list[Message]:
        conversation_service.get_or_raise(db, user_id, conversation_id)

        stmt = (
            select(Message)
            .where(Message.conversation_id == conversation_id)
            .order_by(Message.created_at.asc(), Message.id.asc())
        )
        return list(db.scalars(stmt).all())

    def get_recent_for_conversation(
        self,
        db: Session,
        user_id: int,
        conversation_id: int,
        limit: int = 10,
    ) -> list[Message]:
        conversation_service.get_or_raise(db, user_id, conversation_id)

        stmt = (
            select(Message)
            .where(Message.conversation_id == conversation_id)
            .order_by(Message.created_at.desc(), Message.id.desc())
            .limit(limit)
        )
        items = list(db.scalars(stmt).all())
        items.reverse()
        return items

    def create_for_conversation(
        self,
        db: Session,
        user_id: int,
        conversation_id: int,
        role: str,
        content: str,
        message_type: str = "text",
    ) -> Message:
        conversation = conversation_service.get_or_raise(
            db, user_id, conversation_id
        )

        cleaned_content = content.strip()
        if not cleaned_content:
            raise BusinessException(
                code=40041,
                message="content cannot be empty",
                status_code=400,
            )

        cleaned_message_type = message_type.strip()
        if not cleaned_message_type:
            raise BusinessException(
                code=40042,
                message="message_type cannot be empty",
                status_code=400,
            )

        message = Message(
            conversation_id=conversation_id,
            role=role,
            content=cleaned_content,
            message_type=cleaned_message_type,
        )

        db.add(message)

        conversation.updated_at = datetime.now(UTC)
        db.add(conversation)

        db.commit()
        db.refresh(message)
        return message

    def create_debug_for_conversation(
        self,
        db: Session,
        user_id: int,
        conversation_id: int,
        payload: MessageCreate,
    ) -> Message:
        return self.create_for_conversation(
            db=db,
            user_id=user_id,
            conversation_id=conversation_id,
            role=payload.role.value,
            content=payload.content,
            message_type=payload.message_type,
        )


message_service = MessageService()
```

### 12.2 这份消息服务代码最关键的地方

#### 1. `list_for_conversation`

作用:

拿某个会话下的全部历史消息，并且按正序返回。

正序的意思是:

越早的消息越在前面。

#### 2. `get_recent_for_conversation`

作用:

拿最近若干条历史消息，专门给大模型上下文使用。

这里先按倒序取最近 `limit` 条，再 `reverse()` 变回正序，是一个很常见的写法。

为什么这样做:

因为“取最近 N 条”最容易用倒序加 `limit` 实现。

#### 3. `create_for_conversation`

作用:

封装通用的消息创建逻辑。

这是第五阶段最关键的方法之一。

因为:

1. 用户消息要靠它创建
2. AI 回复消息也要靠它创建

#### 4. 为什么创建消息时还要更新会话 `updated_at`

因为一个会话只要有新消息，它就应该被视为“最近活跃过”。

如果你不更新 `updated_at`，后面会话列表排序就不准确。

#### 5. 为什么这里每次都先校验会话归属

因为消息表本身没有直接挂 `user_id`。

所以做消息操作时，最安全的方式就是:

先验证这个会话属于当前用户，再继续。

## 13. 新建 `app/services/llm_service.py`

这个文件的目标非常明确:

把真正调用大模型的代码单独封装起来。

这样做的好处是:

1. router 里不会塞满模型调用细节
2. chat 业务编排和模型请求实现分离
3. 将来更换模型提供商更容易

### 13.1 新建 `app/services/llm_service.py`

完整代码如下:

```python
import httpx

from app.core.config import settings
from app.core.exception import BusinessException


class LLMService:
    def chat(self, messages: list[dict[str, str]]) -> str:
        if not settings.LLM_API_KEY:
            raise BusinessException(
                code=50041,
                message="LLM_API_KEY is not configured",
                status_code=500,
            )

        if not settings.LLM_BASE_URL:
            raise BusinessException(
                code=50042,
                message="LLM_BASE_URL is not configured",
                status_code=500,
            )

        if not settings.LLM_MODEL_NAME:
            raise BusinessException(
                code=50043,
                message="LLM_MODEL_NAME is not configured",
                status_code=500,
            )

        url = settings.LLM_BASE_URL.rstrip("/") + "/chat/completions"
        headers = {
            "Authorization": f"Bearer {settings.LLM_API_KEY}",
            "Content-Type": "application/json",
        }
        payload = {
            "model": settings.LLM_MODEL_NAME,
            "messages": messages,
            "temperature": 0.7,
        }

        try:
            with httpx.Client(timeout=60.0) as client:
                response = client.post(url, headers=headers, json=payload)
                response.raise_for_status()
        except httpx.HTTPStatusError as exc:
            detail = exc.response.text[:300]
            raise BusinessException(
                code=50241,
                message=f"llm request failed: {detail}",
                status_code=502,
            )
        except httpx.HTTPError as exc:
            raise BusinessException(
                code=50242,
                message=f"llm request error: {exc}",
                status_code=502,
            )

        data = response.json()

        try:
            content = data["choices"][0]["message"]["content"]
        except (KeyError, IndexError, TypeError):
            raise BusinessException(
                code=50243,
                message="invalid llm response format",
                status_code=502,
            )

        cleaned_content = str(content).strip()
        if not cleaned_content:
            raise BusinessException(
                code=50244,
                message="llm returned empty content",
                status_code=502,
            )

        return cleaned_content


llm_service = LLMService()
```

### 13.2 这份 `llm_service.py` 在做什么

#### 1. 先检查配置是否存在

这里先检查:

1. `LLM_API_KEY`
2. `LLM_BASE_URL`
3. `LLM_MODEL_NAME`

这样好处是:

如果你忘了配 `.env`，错误会在这里尽早暴露。

#### 2. 构造统一的模型请求

这里使用的是 OpenAI-compatible 的 `chat/completions` 接口形式。

也就是说，只要你的服务端支持这种风格，大体上都能复用这一套代码。

#### 3. 把网络错误和返回格式错误都转成业务异常

这一步很重要。

因为你不应该把原始 HTTP 异常直接扔给前端。

而是应该把它整理成:

1. 有业务码
2. 有明确 message
3. 统一 JSON 结构

## 14. 新建 `app/services/chat_service.py`

这个文件是第五阶段真正的核心。

它负责把:

1. 会话校验
2. 用户消息落库
3. 历史消息读取
4. prompt 组装
5. 模型调用
6. assistant 消息落库

全部串起来。

### 14.1 新建 `app/services/chat_service.py`

完整代码如下:

```python
from sqlalchemy.orm import Session

from app.core.exception import BusinessException
from app.schemas.chat import ChatSendData, ChatSendRequest
from app.services.conversation_service import conversation_service
from app.services.llm_service import llm_service
from app.services.message_service import message_service


class ChatService:
    HISTORY_LIMIT = 10
    SYSTEM_PROMPT = (
        "你是一名健康管理助手。"
        "请基于用户当前输入和历史聊天内容，给出简洁、清晰、友好的中文回复。"
        "如果问题涉及明显风险，请提醒用户及时线下就医。"
        "不要编造用户没有提供的检查结果。"
    )

    def _build_llm_messages(self, history_messages) -> list[dict[str, str]]:
        messages: list[dict[str, str]] = [
            {"role": "system", "content": self.SYSTEM_PROMPT}
        ]

        for item in history_messages:
            if item.role not in {"user", "assistant", "system"}:
                continue

            content = (item.content or "").strip()
            if not content:
                continue

            messages.append(
                {
                    "role": item.role,
                    "content": content,
                }
            )

        return messages

    def send_message(
        self,
        db: Session,
        user_id: int,
        payload: ChatSendRequest,
    ) -> ChatSendData:
        conversation_service.get_or_raise(
            db, user_id, payload.conversation_id
        )

        cleaned_content = payload.content.strip()
        if not cleaned_content:
            raise BusinessException(
                code=40051,
                message="content cannot be empty",
                status_code=400,
            )

        user_message = message_service.create_for_conversation(
            db=db,
            user_id=user_id,
            conversation_id=payload.conversation_id,
            role="user",
            content=cleaned_content,
            message_type="text",
        )

        history_messages = message_service.get_recent_for_conversation(
            db=db,
            user_id=user_id,
            conversation_id=payload.conversation_id,
            limit=self.HISTORY_LIMIT,
        )

        llm_messages = self._build_llm_messages(history_messages)
        assistant_reply = llm_service.chat(llm_messages)

        assistant_message = message_service.create_for_conversation(
            db=db,
            user_id=user_id,
            conversation_id=payload.conversation_id,
            role="assistant",
            content=assistant_reply,
            message_type="text",
        )

        return ChatSendData(
            conversation_id=payload.conversation_id,
            user_message_id=user_message.id,
            assistant_message_id=assistant_message.id,
            reply=assistant_message.content,
            replied_at=assistant_message.created_at,
        )


chat_service = ChatService()
```

### 14.2 这份聊天服务代码最关键的地方

#### 1. 先校验会话归属

这是整个阶段最重要的安全点之一。

只有在确认:

1. 这个 `conversation_id` 存在
2. 并且它属于当前登录用户

之后，才能继续发消息。

#### 2. 为什么先写 `user message`，再读历史消息

因为用户刚发的这句话，本身就应该成为上下文的一部分。

所以最合理的顺序是:

1. 先把用户消息入库
2. 再把最近消息读出来
3. 再送给模型

#### 3. 为什么要限制历史消息条数

第五阶段先用最小版本，不建议一开始把整段历史全塞给模型。

原因:

1. 容易浪费 token
2. 调试复杂
3. 后期消息越来越多时会变慢

所以第五阶段先保守一点，比如最近 `10` 条消息，就够了。

#### 4. 为什么要有系统提示词 `SYSTEM_PROMPT`

因为大模型需要一个最小的角色设定。

第五阶段不做复杂 prompt 编排，但至少要告诉模型:

1. 它是谁
2. 应该怎么回答
3. 风险场景下要怎么提醒

#### 5. 为什么 `llm_service` 和 `chat_service` 要分开

因为它们的职责不同:

1. `llm_service` 只负责“怎么调模型”
2. `chat_service` 负责“整个聊天业务流程怎么编排”

这样分开以后，代码会更清楚。

## 15. 新建聊天路由 `app/api/routes/chat.py`

router 层只负责:

1. 接收请求
2. 注入依赖
3. 调用 service
4. 返回统一响应

不要把模型调用代码直接塞到 router 里。

### 15.1 新建 `app/api/routes/chat.py`

完整代码如下:

```python
from fastapi import APIRouter, Depends
from sqlalchemy.orm import Session

from app.api.deps import get_current_active_user, get_db
from app.core.response import success_response
from app.models.user import User
from app.schemas.chat import ChatSendData, ChatSendRequest
from app.schemas.common import ApiResponse
from app.services.chat_service import chat_service

router = APIRouter(prefix="/chat", tags=["chat"])


@router.post(
    "/send",
    response_model=ApiResponse[ChatSendData],
)
def send_chat_message(
    payload: ChatSendRequest,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_active_user),
):
    result = chat_service.send_message(
        db=db,
        user_id=current_user.id,
        payload=payload,
    )
    return success_response(
        data=result,
        message="chat reply generated",
    )
```

### 15.2 这份路由代码在做什么

#### `POST /chat/send`

作用:

这是第五阶段最核心的新接口。

它做的事情只有一件:

把整条聊天主链路交给 `chat_service.send_message()` 去处理。

你要特别记住:

router 越薄越好。

因为真正复杂的聊天逻辑，不应该写在路由里。

## 18. 第五阶段的聊天主链路，逐步解释一遍

现在我们把整个 `POST /chat/send` 的执行过程串起来。

## 第一步: 路由接收请求

文件:

`app/api/routes/chat.py`

这里负责:

1. 接收 `conversation_id`
2. 接收 `content`
3. 注入 `db`
4. 注入 `current_user`
5. 调用 `chat_service.send_message`

## 第二步: 校验会话归属

文件:

`app/services/chat_service.py`

这里先调用:

```python
conversation_service.get_or_raise(...)
```

确保:

1. 会话存在
2. 会话属于当前用户

## 第三步: 先保存 user message

文件:

`app/services/chat_service.py`

调用:

```python
message_service.create_for_conversation(...)
```

把用户输入先落库。

## 第四步: 读取最近历史消息

文件:

`app/services/chat_service.py`

调用:

```python
message_service.get_recent_for_conversation(...)
```

拿最近若干条上下文。

## 第五步: 组装给大模型的 messages

文件:

`app/services/chat_service.py`

这里会:

1. 先加一条 `system` 提示词
2. 再把最近消息按顺序放进去

## 第六步: 调用模型

文件:

`app/services/llm_service.py`

调用:

```python
llm_service.chat(llm_messages)
```

拿到一段 AI 文本回复。

## 第七步: 保存 assistant message

文件:

`app/services/chat_service.py`

再次调用:

```python
message_service.create_for_conversation(...)
```

不过这次写入的是:

1. `role="assistant"`
2. `content=assistant_reply`

## 第八步: 返回给前端

文件:

`app/api/routes/chat.py`

最后把结果包装成统一响应返回。

## 19. 为什么第五阶段建议这样分层

这一阶段最推荐的分层结构是:

1. `chat.py` 路由层
2. `chat_service.py` 业务编排层
3. `llm_service.py` 模型调用层
4. `message_service.py` 消息读写层

你可以这样理解它们的职责:

### `chat.py`

负责:

“收 HTTP 请求”

### `chat_service.py`

负责:

“把聊天主链路串起来”

### `llm_service.py`

负责:

“真正调模型”

### `message_service.py`

负责:

“消息怎么存、怎么查”

这样分层的好处是:

1. 容易读
2. 容易改
3. 容易定位 bug
4. 后面切换模型更简单

## 20. 第五阶段最容易写错的地方

### 20.1 直接在 router 里写模型调用

这是最典型的错误之一。

错误思路:

1. 路由里自己查会话
2. 路由里自己写 user message
3. 路由里自己调模型
4. 路由里自己写 assistant message

这样会让 router 很快变得又长又乱。

正确思路是:

把聊天主链路收口到 `chat_service.py`。

### 20.2 先调模型，再写 user message

这会破坏聊天历史的完整性。

因为用户刚发的这句话，本来就应该成为历史消息的一部分。

正确顺序一定是:

1. 先写 user message
2. 再读历史
3. 再调模型

### 20.3 查会话时不校验归属

错误写法:

```python
conversation = db.get(Conversation, conversation_id)
```

这样很容易导致用户给别人的会话发消息。

正确思路:

必须通过当前用户 + 会话 id 一起校验。

### 20.4 把所有历史消息都扔给模型

第五阶段最小版本不需要这样。

一开始就全量历史，很容易:

1. 太慢
2. 太耗 token
3. 太难调试

先取最近 `10` 条就够了。

### 20.5 插入 assistant message 时忘了落库

有些人会做到:

1. 成功调到模型
2. 也成功把回复返回前端

但忘了把 assistant 消息写进数据库。

这样一刷新历史记录就没了。

所以第五阶段一定要确认:

用户消息和 AI 回复都入库。

### 20.6 配置没填，结果一直报模型调用错误

第五阶段非常依赖 `.env`。

如果这些值没填:

1. `LLM_API_KEY`
2. `LLM_BASE_URL`
3. `LLM_MODEL_NAME`

那后端是一定没法正常调用模型的。

## 21. 第五阶段为什么不建议现在就做流式输出

很多人一到聊天就会本能地想做“像 ChatGPT 那样边生成边输出”。

但你现在不适合。

因为你当前真正需要先确认的是:

1. conversation 归属有没有校验对
2. user message 有没有入库
3. 历史消息有没有正确读取
4. prompt 有没有正确组装
5. 模型到底有没有调用成功
6. assistant message 有没有入库

这些问题如果还没完全稳定，你再叠加流式输出，只会让排错更困难。

所以第五阶段先做同步返回，是最正确的工程顺序。

## 22. 第五阶段怎么做 Swagger 验收

下面给你一个推荐的 Swagger 演示顺序。

## 22.1 先完成登录

第五阶段所有聊天接口都需要当前登录用户。

所以先:

1. 注册
2. 登录
3. 点 Swagger 的 `Authorize`

## 22.2 先确保你已经有一个会话

如果第四阶段已经完成，就先用:

`POST /api/v1/conversations`

创建一个会话。

比如请求体:

```json
{
  "title": "今天的健康咨询"
}
```

记住返回的 `conversation_id`。

## 22.3 发送一条聊天消息

调用:

`POST /api/v1/chat/send`

请求体示例:

```json
{
  "conversation_id": 1,
  "content": "我今天血压有点高，应该注意什么？"
}
```

如果一切正常，你应该拿到:

1. `conversation_id`
2. `user_message_id`
3. `assistant_message_id`
4. `reply`
5. `replied_at`

## 22.4 再查看会话历史消息

调用:

`GET /api/v1/conversations/{conversation_id}/messages`

你应该能看到至少两条消息:

1. 用户刚发的 `user` 消息
2. AI 刚回复的 `assistant` 消息

这一步非常关键。

因为它验证的是:

消息不只是“返回给前端了”，而是真的落库了。

## 22.5 再连续发送第二条消息

比如:

```json
{
  "conversation_id": 1,
  "content": "那我今天饮食上应该怎么控制？"
}
```

然后再查消息列表。

你应该看到:

1. 第一轮 user
2. 第一轮 assistant
3. 第二轮 user
4. 第二轮 assistant

这说明多轮聊天已经跑起来了。
