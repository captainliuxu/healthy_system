# Healthy System FastAPI 第一阶段详细教程

## 1. 这份文档解决什么问题

这份教程只讲第一阶段: `基础骨架层`。

目标不是立刻堆业务功能, 而是先把 FastAPI 后端搭成一个后面可以持续加功能、又不容易失控的标准工程。

这份教程会覆盖你要求的 6 件事:

1. 项目目录结构
2. 配置管理
3. 数据库连接与会话管理
4. 通用返回格式
5. 全局异常处理
6. Alembic 迁移初始化

这份教程不会做的事:

1. 不做聊天功能
2. 不做 WebSocket
3. 不做 scheduler
4. 不写复杂业务逻辑

你当前仓库里已经有 `backend/`、`frontend/`，所以本教程默认:

- 后端根目录是 `D:\vue_fastapi\backend`
- 文档给出的代码，都是以 `backend/` 为起点
- 我这里只提供教学文件，不直接改你的后端代码

## 2. 第一阶段完成后应该长什么样

完成后，后端至少应该满足这几个验收条件:

1. 项目能启动
2. `/docs` 可访问
3. 数据库能连通
4. 接口返回结构统一
5. Alembic 能正常生成和执行 migration

## 3. 每个模块到底负责什么

先把职责分清楚，后面写代码才不会乱。

| 模块 | 作用 | 这一阶段是否要写 |
| --- | --- | --- |
| `app/api/` | HTTP 接口入口, 放路由、依赖注入 | 要写 |
| `app/core/` | 全局配置、异常、统一响应这类“全局规则” | 要写 |
| `app/models/` | SQLAlchemy ORM 模型 | 先留空, 第二阶段再补业务表 |
| `app/schemas/` | Pydantic 请求体、响应体结构 | 要写 |
| `app/services/` | 业务编排层 | 第一阶段先留空 |
| `app/db/` | 数据库引擎、会话、Alembic 关联 | 要写 |
| `app/utils/` | 纯工具函数 | 第一阶段先留空 |
| `app/main.py` | 应用装配入口 | 要写 |

一句话记忆:

- `api` 管“收发请求”
- `services` 管“业务流程”
- `models` 管“数据库表”
- `schemas` 管“数据结构”
- `db` 管“连接数据库”
- `core` 管“项目规则”

---

## 4. 第一阶段开发顺序

推荐顺序不要乱:

1. 先建目录
2. 再做配置管理
3. 再做数据库连接与会话
4. 再做统一响应
5. 再做全局异常
6. 再做健康检查路由
7. 最后做 Alembic 初始化

原因很简单:

- 没配置, 其他模块没法稳定读参数
- 没数据库会话, 后面所有表操作都会乱
- 没统一返回和异常, 前后端联调会越来越痛苦
- 没 Alembic, 你后面每加一张表都要手改数据库

## 5. 先准备环境变量

### 5.1 `.env.example`

这个文件是“模板”，不是正式配置文件。

你应该把它提交到 Git，方便别人知道项目需要哪些环境变量。

完整代码如下:

```dotenv
PROJECT_NAME=healthy_system_fastapi
APP_ENV=dev
DEBUG=true
API_V1_PREFIX=/api/v1

# 本地开发先用 SQLite，最快
DATABASE_URL=sqlite:///./healthy_system.db

# 如果你后面要切 MySQL，可以改成下面这种格式
# DATABASE_URL=mysql+pymysql://root:123456@127.0.0.1:3306/healthy_system

JWT_SECRET_KEY=replace_this_in_production
JWT_ALGORITHM=HS256
JWT_ACCESS_TOKEN_EXPIRE_MINUTES=120
JWT_REFRESH_TOKEN_EXPIRE_DAYS=7

LLM_PROVIDER=dashscope
LLM_API_KEY=
LLM_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
LLM_MODEL_NAME=qwen-plus

BACKEND_CORS_ORIGINS=http://localhost:5173,http://127.0.0.1:5173
```

每个配置项的作用:

- `PROJECT_NAME`: FastAPI 文档标题和项目名
- `APP_ENV`: 当前环境，常见值是 `dev`、`test`、`prod`
- `DEBUG`: 是否开启调试
- `API_V1_PREFIX`: 接口统一前缀
- `DATABASE_URL`: 数据库连接地址
- `JWT_SECRET_KEY`: JWT 签名密钥
- `JWT_ALGORITHM`: JWT 算法
- `JWT_ACCESS_TOKEN_EXPIRE_MINUTES`: 访问令牌有效期
- `JWT_REFRESH_TOKEN_EXPIRE_DAYS`: 刷新令牌有效期
- `LLM_PROVIDER`: 大模型提供方标识
- `LLM_API_KEY`: 大模型密钥
- `LLM_BASE_URL`: 大模型网关地址
- `LLM_MODEL_NAME`: 默认模型名
- `BACKEND_CORS_ORIGINS`: 允许跨域的前端地址列表

### 5.2 `.env`

正式运行时应该读取 `.env`，而不是读取 `.env.example`。

也就是说:

- `.env.example` 负责“告诉别人要填什么”
- `.env` 负责“真正给程序读取”

你可以直接复制一份:

```powershell
Copy-Item .env.example .env
```

## 6. 配置管理: `app/core/config.py`

这是第一阶段最关键的文件之一。

它的核心作用不是“存几个变量”，而是:

1. 把所有配置集中管理
2. 避免在各个文件里写死字符串
3. 让数据库、JWT、LLM、CORS 都从一个入口读取

完整代码如下:

```python
from functools import lru_cache
from pathlib import Path

from pydantic import field_validator
from pydantic_settings import BaseSettings, SettingsConfigDict

BASE_DIR = Path(__file__).resolve().parent.parent.parent


class Settings(BaseSettings):
    PROJECT_NAME: str = "healthy_system_fastapi"
    APP_ENV: str = "dev"
    DEBUG: bool = True
    API_V1_PREFIX: str = "/api/v1"

    DATABASE_URL: str = "sqlite:///./healthy_system.db"

    JWT_SECRET_KEY: str = "replace_this_in_production"
    JWT_ALGORITHM: str = "HS256"
    JWT_ACCESS_TOKEN_EXPIRE_MINUTES: int = 120
    JWT_REFRESH_TOKEN_EXPIRE_DAYS: int = 7

    LLM_PROVIDER: str = "dashscope"
    LLM_API_KEY: str = ""
    LLM_BASE_URL: str = ""
    LLM_MODEL_NAME: str = "qwen-plus"

    BACKEND_CORS_ORIGINS: list[str] = [
        "http://localhost:5173",
        "http://127.0.0.1:5173",
    ]

    @field_validator("BACKEND_CORS_ORIGINS", mode="before")
    @classmethod
    def parse_cors_origins(cls, value):
        if isinstance(value, str):
            if not value.strip():
                return []
            return [item.strip() for item in value.split(",") if item.strip()]
        return value

    model_config = SettingsConfigDict(
        env_file=BASE_DIR / ".env",
        env_file_encoding="utf-8",
        extra="ignore",
    )


@lru_cache
def get_settings() -> Settings:
    return Settings()


settings = get_settings()
```

### 6.1 这段代码每一部分的作用

#### `BASE_DIR`

```python
BASE_DIR = Path(__file__).resolve().parent.parent.parent
```

作用:

- 定位 `backend/` 根目录
- 保证程序无论从哪个目录启动，都能找到 `.env`

#### `Settings(BaseSettings)`

作用:

- 把环境变量自动映射成 Python 配置对象
- 你以后在任何文件里都可以通过 `settings.xxx` 读取配置

#### `field_validator("BACKEND_CORS_ORIGINS")`

作用:

- 把 `.env` 里用逗号写的一串地址
- 自动转成 Python 的 `list[str]`

比如:

```dotenv
BACKEND_CORS_ORIGINS=http://localhost:5173,http://127.0.0.1:5173
```

会变成:

```python
[
    "http://localhost:5173",
    "http://127.0.0.1:5173",
]
```

#### `@lru_cache`

作用:

- 保证 `Settings()` 只初始化一次
- 避免每次导入都重复读取环境变量

### 6.2 这一阶段为什么一定要把 JWT 和 LLM 先写进配置

因为这两个东西虽然这一阶段不用实现业务，但它们都是后面一定会依赖的“全局基础参数”。

现在先把口子留好，后面:

- 做登录时直接接 JWT
- 做聊天时直接接 LLM

就不会到处重构配置文件。

### 6.3 你当前项目里最值得修正的一点

你现在的 `config.py` 是直接读取 `.env.example`。

这不适合正式开发流程。

正确做法是:

- 程序读 `.env`
- 仓库保存 `.env.example`

这也是本教程推荐的版本。

## 7. 数据库连接与会话管理

这一层的目标是解决 4 件事:

1. 创建 SQLAlchemy engine
2. 创建 `SessionLocal`
3. 定义 `Base`
4. 提供请求级数据库依赖注入

### 7.1 `app/db/session.py`

完整代码如下:

```python
from collections.abc import Generator

from sqlalchemy import create_engine
from sqlalchemy.orm import DeclarativeBase, Session, sessionmaker

from app.core.config import settings


class Base(DeclarativeBase):
    pass


connect_args = {}
if settings.DATABASE_URL.startswith("sqlite"):
    connect_args = {"check_same_thread": False}


engine = create_engine(
    settings.DATABASE_URL,
    pool_pre_ping=True,
    connect_args=connect_args,
)


SessionLocal = sessionmaker(
    autocommit=False,
    autoflush=False,
    bind=engine,
    class_=Session,
)


def get_db() -> Generator[Session, None, None]:
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
```

### 7.2 `app/db/session.py` 每个对象的作用

#### `Base`

```python
class Base(DeclarativeBase):
    pass
```

作用:

- 所有 ORM 模型的共同父类
- Alembic 会通过它拿到所有表的元数据

#### `engine`

作用:

- 真正负责和数据库建立连接
- 这是 SQLAlchemy 的底层连接入口

这里加 `pool_pre_ping=True` 的原因:

- 数据库连接池里的连接如果失效
- SQLAlchemy 会先探测一下，降低“连接已断开”的概率

#### `SessionLocal`

作用:

- 每次业务操作都从这里拿一个数据库会话
- 不要自己在路由里手写 `Session(engine)`，统一从这里出

#### `get_db()`

作用:

- 给 FastAPI 的依赖注入系统使用
- 每个请求拿到一个独立会话
- 请求结束后自动关闭

这就是标准的“请求级数据库会话”。

### 7.3 `app/db/base.py`

这个文件是给 Alembic 用的。

完整代码如下:

```python
from app.db.session import Base
from app.models import *  # noqa: F401,F403
```

这行看起来简单，但作用很关键:

- Alembic 只认识 `Base.metadata`
- 但前提是所有模型都要先被导入
- 所以后面你只要把 ORM 模型统一导入到 `app/models/__init__.py`
- Alembic 就能自动识别新表

### 7.4 `app/models/__init__.py`

第一阶段先留空，代码如下:

```python
# 第一阶段先不放业务表。
# 第二阶段开始，把所有 ORM 模型统一导入到这里，供 Alembic 自动发现。
```

### 7.5 `app/api/deps.py`

完整代码如下:

```python
from app.db.session import get_db

__all__ = ["get_db"]
```

作用:

- 这是“路由依赖出口”
- 以后所有路由想拿数据库会话，都从这里导入 `get_db`
- 这样后面如果你加用户鉴权、分页参数、租户信息，也都可以放进 `deps.py`

### 7.6 为什么建议把 `models` 和 `db` 分开

很多初学者会把表模型放到 `db/` 下面。

短期看没问题，长期会混乱。

因为:

- `db/` 应该只负责“怎么连数据库”
- `models/` 才负责“数据库里有哪些表”

职责分开后，后面加表、迁移、服务层都更清楚。

## 8. 通用返回格式

你的所有成功接口，建议统一长这样:

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

这样做的好处:

1. 前端永远知道从哪拿数据
2. 不同接口结构统一，联调成本低
3. 错误响应也能复用相同外壳结构

### 8.1 `app/schemas/common.py`

完整代码如下:

```python
from typing import Generic, TypeVar

from pydantic import BaseModel, ConfigDict

T = TypeVar("T")


class ApiResponse(BaseModel, Generic[T]):
    code: int = 0
    message: str = "success"
    data: T | None = None

    model_config = ConfigDict(from_attributes=True)
```

作用说明:

- `ApiResponse` 是统一响应壳
- `code` 表示业务状态码
- `message` 表示提示信息
- `data` 才是真正业务数据

### 8.2 `app/core/response.py`

完整代码如下:

```python
from typing import Any

from app.schemas.common import ApiResponse


def success_response(data: Any = None, message: str = "success") -> ApiResponse[Any]:
    return ApiResponse(code=0, message=message, data=data)


def error_response(code: int, message: str, data: Any = None) -> ApiResponse[Any]:
    return ApiResponse(code=code, message=message, data=data)
```

作用说明:

- `success_response()` 给成功接口使用
- `error_response()` 给异常处理器使用

这样做的意义是:

- 路由层不用反复手写 `{"code": 0, "message": "...", "data": ...}`
- 异常层也能共用同一套响应壳

## 9. 全局异常处理

这是第一阶段必须做的。

你要求处理的异常包括:

1. 参数错误
2. 业务错误
3. 404
4. 500

最怕的写法是:

- 每个路由都自己 `return {"error": "..."}`
- 一会儿返回 `detail`
- 一会儿返回 `msg`
- 一会儿又返回 `error`

这样前端会崩溃，后端自己也不好维护。

### 9.1 `app/core/exception.py`

完整代码如下:

```python
import logging
from typing import Any

from fastapi import FastAPI, Request
from fastapi.exceptions import RequestValidationError
from fastapi.responses import JSONResponse
from starlette.exceptions import HTTPException as StarletteHTTPException

from app.core.response import error_response

logger = logging.getLogger(__name__)


class BusinessException(Exception):
    def __init__(
        self,
        code: int,
        message: str,
        status_code: int = 400,
        data: Any = None,
    ):
        self.code = code
        self.message = message
        self.status_code = status_code
        self.data = data


def build_json_response(
    status_code: int,
    code: int,
    message: str,
    data: Any = None,
) -> JSONResponse:
    payload = error_response(code=code, message=message, data=data)
    return JSONResponse(status_code=status_code, content=payload.model_dump())


async def business_exception_handler(
    request: Request,
    exc: BusinessException,
) -> JSONResponse:
    return build_json_response(
        status_code=exc.status_code,
        code=exc.code,
        message=exc.message,
        data=exc.data,
    )


async def http_exception_handler(
    request: Request,
    exc: StarletteHTTPException,
) -> JSONResponse:
    message = exc.detail if isinstance(exc.detail, str) else "request error"

    error_code_map = {
        401: 40100,
        403: 40300,
        404: 40400,
    }
    code = error_code_map.get(exc.status_code, exc.status_code * 100)

    return build_json_response(
        status_code=exc.status_code,
        code=code,
        message=message,
    )


async def validation_exception_handler(
    request: Request,
    exc: RequestValidationError,
) -> JSONResponse:
    error_list = []
    for item in exc.errors():
        error_list.append(
            {
                "field": ".".join(str(part) for part in item.get("loc", [])),
                "message": item.get("msg", "invalid input"),
            }
        )

    return build_json_response(
        status_code=422,
        code=42200,
        message="request validation error",
        data=error_list,
    )


async def generic_exception_handler(
    request: Request,
    exc: Exception,
) -> JSONResponse:
    logger.exception("Unhandled exception: %s", exc)
    return build_json_response(
        status_code=500,
        code=50000,
        message="internal server error",
    )


def register_exception_handlers(app: FastAPI) -> None:
    app.add_exception_handler(BusinessException, business_exception_handler)
    app.add_exception_handler(StarletteHTTPException, http_exception_handler)
    app.add_exception_handler(RequestValidationError, validation_exception_handler)
    app.add_exception_handler(Exception, generic_exception_handler)
```

### 9.2 这里每种异常分别处理什么

#### `BusinessException`

这是你自己定义的业务异常。

适合场景:

- 用户不存在
- 档案不存在
- 记录状态不允许
- 重复提交

也就是说:

- 业务没过
- 但不是系统崩了

举例:

```python
raise BusinessException(code=40001, message="profile not found", status_code=404)
```

#### `RequestValidationError`

这是 FastAPI 参数校验失败时自动抛出来的异常。

比如:

- 少传字段
- 字段类型不对
- 长度不符合要求

你不需要在每个路由里自己抓它，全局处理就行。

#### `StarletteHTTPException`

这个用来兜 FastAPI/Starlette 里的 HTTP 异常，尤其是:

- 404 路由不存在
- 手动抛出的 401、403、404

#### `Exception`

这是最后的兜底。

只要前面没抓住，就走这里，返回 500。

### 9.3 为什么错误也要保持统一结构

因为前端最好永远按同一种格式解析:

```json
{
  "code": 40400,
  "message": "Not Found",
  "data": null
}
```

而不是一会儿:

```json
{"detail": "Not Found"}
```

一会儿又:

```json
{"msg": "error"}
```

## 10. 路由层先做最小可运行版本

第一阶段不要写业务接口。

你只需要一个健康检查路由，证明三件事:

1. 服务起来了
2. 统一响应生效了
3. 数据库依赖注入能用

### 10.1 `app/api/routes/health.py`

完整代码如下:

```python
from fastapi import APIRouter, Depends
from sqlalchemy import text
from sqlalchemy.orm import Session

from app.api.deps import get_db
from app.core.response import success_response

router = APIRouter(prefix="/health", tags=["health"])


@router.get("")
def health_check():
    return success_response(
        data={"service": "ok"},
        message="service is running",
    )


@router.get("/db")
def health_db(db: Session = Depends(get_db)):
    db.execute(text("SELECT 1"))
    return success_response(
        data={"database": "connected"},
        message="database is ok",
    )
```

### 10.2 这两个接口分别证明什么

#### `GET /api/v1/health`

证明:

- FastAPI 已启动
- 路由已注册
- 统一响应格式已经能用

#### `GET /api/v1/health/db`

证明:

- 依赖注入拿到了数据库会话
- SQLAlchemy 连接可用
- 你的数据库链路是通的

### 10.3 `app/api/router.py`

完整代码如下:

```python
from fastapi import APIRouter

from app.api.routes.health import router as health_router

api_router = APIRouter()
api_router.include_router(health_router)
```

作用:

- 统一收口所有子路由
- 后面你加 `auth`、`profile`、`record` 时，都只往这里挂

## 11. 应用入口: `app/main.py`

`main.py` 的任务不是写业务逻辑。

它只负责装配:

1. 创建 FastAPI 对象
2. 注册 CORS
3. 挂总路由
4. 注册异常处理器

完整代码如下:

```python
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

from app.api.router import api_router
from app.core.config import settings
from app.core.exception import register_exception_handlers


def create_application() -> FastAPI:
    app = FastAPI(
        title=settings.PROJECT_NAME,
        debug=settings.DEBUG,
        version="0.1.0",
    )

    app.add_middleware(
        CORSMiddleware,
        allow_origins=settings.BACKEND_CORS_ORIGINS or ["*"],
        allow_credentials=True,
        allow_methods=["*"],
        allow_headers=["*"],
    )

    app.include_router(api_router, prefix=settings.API_V1_PREFIX)
    register_exception_handlers(app)

    return app


app = create_application()
```

### 11.1 这段代码每部分的作用

#### `FastAPI(...)`

作用:

- 创建应用对象
- 决定文档标题、debug 状态等基础信息

#### `CORSMiddleware`

作用:

- 允许前端跨域访问后端
- Vue 前端常见是 `5173`

为什么放到配置里:

- 开发环境和生产环境允许的域名通常不一样
- 不能把跨域来源写死在 `main.py`

#### `app.include_router(...)`

作用:

- 把所有 API 路由统一挂载到 `/api/v1`

这样接口会长成:

- `/api/v1/health`
- `/api/v1/health/db`

#### `register_exception_handlers(app)`

作用:

- 统一接管全局异常响应

这样:

- 参数错了，不会散乱返回
- 404，不会默认返回裸 `detail`
- 500，也能回统一结构

### 11.2 CORS 一个很容易踩的点

如果你未来要带 cookie 或跨站凭证，生产环境不要长期用 `["*"]`。

建议:

- 开发环境可以灵活一些
- 生产环境一定写死白名单域名

## 12. Alembic 迁移初始化

这是第一阶段必须做的，不然后面表一多就会非常难维护。

### 12.1 为什么一定要上 Alembic

因为后面你至少会不断增加:

- 用户表
- 老人档案表
- 健康记录表
- 对话表
- 消息表

如果每次都手改数据库:

- 环境不同步
- 同学机器不一致
- 上线无法回滚

Alembic 就是专门解决这个问题的。

### 12.2 初始化命令

进入后端目录:

```powershell
cd D:\vue_fastapi\backend
```

初始化 Alembic:

```powershell
alembic init alembic
```

执行后一般会生成:

```text
backend/
├─ alembic/
│  ├─ versions/
│  ├─ env.py
│  ├─ README
│  └─ script.py.mako
└─ alembic.ini
```

其中:

- `README` 和 `script.py.mako` 可以先保留默认
- 你主要需要改的是 `alembic.ini` 和 `alembic/env.py`

### 12.3 `alembic.ini`

完整示例代码如下:

```ini
[alembic]
script_location = alembic
prepend_sys_path = .
path_separator = os
sqlalchemy.url = sqlite:///./healthy_system.db

[post_write_hooks]

[loggers]
keys = root,sqlalchemy,alembic

[handlers]
keys = console

[formatters]
keys = generic

[logger_root]
level = WARN
handlers = console

[logger_sqlalchemy]
level = WARN
handlers =
qualname = sqlalchemy.engine

[logger_alembic]
level = INFO
handlers =
qualname = alembic

[handler_console]
class = StreamHandler
args = (sys.stderr,)
level = NOTSET
formatter = generic

[formatter_generic]
format = %(levelname)-5.5s [%(name)s] %(message)s
```

这里最关键的不是 `sqlalchemy.url` 这一行本身，而是:

- `prepend_sys_path = .`

这会让 Alembic 能从当前项目根路径导入 `app.*` 模块。

### 12.4 `alembic/env.py`

完整代码如下:

```python
from logging.config import fileConfig

from alembic import context
from sqlalchemy import engine_from_config, pool

from app.core.config import settings
from app.db.base import Base

config = context.config

if config.config_file_name is not None:
    fileConfig(config.config_file_name)

config.set_main_option("sqlalchemy.url", settings.DATABASE_URL)

target_metadata = Base.metadata


def run_migrations_offline() -> None:
    url = config.get_main_option("sqlalchemy.url")
    context.configure(
        url=url,
        target_metadata=target_metadata,
        literal_binds=True,
        compare_type=True,
    )

    with context.begin_transaction():
        context.run_migrations()


def run_migrations_online() -> None:
    connectable = engine_from_config(
        config.get_section(config.config_ini_section, {}),
        prefix="sqlalchemy.",
        poolclass=pool.NullPool,
    )

    with connectable.connect() as connection:
        context.configure(
            connection=connection,
            target_metadata=target_metadata,
            compare_type=True,
        )

        with context.begin_transaction():
            context.run_migrations()


if context.is_offline_mode():
    run_migrations_offline()
else:
    run_migrations_online()
```

### 12.5 `env.py` 里最关键的两行

#### 第一行

```python
from app.db.base import Base
```

作用:

- 拿到你的项目元数据入口

#### 第二行

```python
target_metadata = Base.metadata
```

作用:

- 告诉 Alembic: “请根据这个 metadata 来比较数据库结构差异”

这就是自动迁移生成的基础。

### 12.6 第一阶段没有业务表, migration 怎么做

这是很多人会卡住的地方。

答案是:

- 第一阶段就算还没有正式业务表
- 你也应该先把 Alembic 链路打通

你可以先生成一个空迁移，验证流程没问题:

```powershell
alembic revision --autogenerate -m "init empty"
alembic upgrade head
```

如果当前还没有 ORM 模型，这个迁移文件可能是空的，这完全正常。

空迁移大概会长这样:

```python
"""init empty"""

from alembic import op

revision = "0001_init_empty"
down_revision = None
branch_labels = None
depends_on = None


def upgrade() -> None:
    pass


def downgrade() -> None:
    pass
```

它的意义不是“创建了什么表”，而是:

- 验证 Alembic 能运行
- 验证项目能正常识别 metadata
- 给后续真正建表打基础

### 12.7 第二阶段开始要做什么

一旦你开始写 `Profile`、`Record` 等 ORM 模型:

1. 把模型导入 `app/models/__init__.py`
2. 执行 `alembic revision --autogenerate -m "create xxx table"`
3. 执行 `alembic upgrade head`

这样建表就不靠手工改数据库了。

## 14. 验收时你应该看到什么

### 14.1 `/api/v1/health`

```json

  "message": "service is running",
```

### 14.2 `/api/v1/health/db`

预期返回:

```json

  "code": 0,

  "data": {
    "database": "connected"
  }
}
```

### 14.3 参数错误时

比如你后面某个接口参数不对，应该返回统一结构，类似:

```json
{

  "message": "request validation error",
  "data": [
    {
      "field": "body.username",

    }
  ]
}
```

### 14.4 路由不存在时

比如访问不存在的接口，应该类似:

```json
{
  "code": 40400,
  "message": "Not Found",
  "data": null
}
```

## 15. 这一阶段最常见的 10 个错误

### 错误 1: 程序直接读取 `.env.example`

问题:

- 这是模板，不是运行配置

正确做法:

- 程序读 `.env`
- 仓库保存 `.env.example`

### 错误 2: 路由里自己创建数据库连接

错误示例思路:

- 每个接口里自己 `create_engine`
- 或自己手动创建 Session

问题:

- 很快会出现连接泄漏和重复代码

正确做法:

- 统一用 `get_db()`

### 错误 3: 成功接口和失败接口返回格式不一致

问题:

- 前端每个接口都要写不同解析逻辑

正确做法:

- 统一 `code/message/data`

### 错误 4: 把 `HTTPException` 当全部异常方案

问题:

- 404、参数错误、业务错误、系统错误混在一起

正确做法:

- 全局异常分类处理

### 错误 5: 把 ORM 模型继续堆在 `db/` 目录里

问题:

- `db` 和 `models` 职责混乱

正确做法:

- `db` 管连接
- `models` 管表

### 错误 6: `main.py` 什么都往里塞

问题:

- 到后面会变成几百行垃圾场

正确做法:

- `main.py` 只装配，不写业务

### 错误 7: 第一阶段就急着写聊天和 WebSocket

问题:

- 地基没打稳，后面越写越乱

正确做法:

- 先把配置、DB、响应、异常、迁移这层弄稳定

### 错误 8: Alembic 初始化了，但没接项目 metadata

问题:

- `--autogenerate` 永远扫不出你的表

正确做法:

- `target_metadata = Base.metadata`

### 错误 9: `app/models/__init__.py` 不导入模型

问题:

- Alembic 看不到新表

正确做法:

- 后续统一在这里导入所有 ORM 模型

### 错误 10: 生产环境也继续放开所有 CORS

问题:

- 安全边界不清晰

正确做法:

- 开发环境灵活
- 生产环境白名单

## 16. 这个阶段不要做什么

你在这个阶段应该克制住不去碰下面这些:

1. 聊天上下文设计
2. WebSocket 推送
3. APScheduler 定时任务
4. 大量 service 业务代码
5. 复杂 repository 抽象
6. 多角色权限系统

原因不是这些不重要，而是现在做会把基础层和业务层搅在一起。

第一阶段只做一件事:

把后面所有功能都会依赖的工程底座搭稳。
