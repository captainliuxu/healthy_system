Healthy System FastAPI 第八阶段详细教程

1. 这一阶段到底要解决什么

第八阶段要做的，不是新增新的业务核心，而是把前面已经做好的能力变得：

  1. 更实时
  2. 更自动
  3. 更像真实产品

如果说第七阶段已经让系统具备了：

“根据规则生成主动消息”

那么第八阶段要把它推进成：

“系统会自动检查，并且一旦生成主动消息，可以实时推到前端”

一句话理解第八阶段：

让主动服务从“手动触发可运行”，升级成“自动巡检 + 实时送达”。

2. 第八阶段完成后的验收标准

做到下面这些，就算第八阶段合格：

  1. 前端可以建立 WebSocket 连接
  2. 当前用户连接成功后，能收到一条欢迎消息
  3. 后端可以向当前用户推送一条测试实时消息
  4. proactive_service 创建主动消息后，可以选配实时推送
  5. 调度器能按固定周期自动扫描规则
  6. 自动扫描后仍然通过原有 active_log 留痕
  7. 即使 WebSocket 推送失败，也不会影响主动消息落库

3. 为什么 WebSocket 和 Scheduler 都要放在后面

因为它们解决的都不是“核心业务是否成立”，而是：

  1. WebSocket 解决结果怎么更快送达
  2. Scheduler 解决业务怎么自动执行

但这两层都依赖前面的业务已经稳定：

  1. 第四阶段的 Conversation / Message
  2. 第五阶段的聊天主链路
  3. 第六阶段的 TriggerRule / ActiveLog
  4. 第七阶段的 ProactiveWindow / ProactiveMessage / ProactiveService

如果这些前置层没打稳，你第八阶段只会得到：

  1. 一个会连上但没业务可推的 WebSocket
  2. 一个会定时空转甚至定时报错的 Scheduler

所以第八阶段在工程上一定是增强层，不是起步层。

4. 先明确第八阶段和第七阶段的衔接点

第七阶段已经有一条完整主链路：

trigger_rule_service.evaluate_for_user()

-> proactive_window_service.allow_trigger_now()

-> proactive_service.execute_rule_for_user()

-> ProactiveMessage

-> ActiveLog

第八阶段不会改这条主链路的核心职责。

它只是往这条链路外面再接两层：

4.1 WebSocket 层

当 proactive_service 成功创建一条 ProactiveMessage 后：

  1. 消息照样落库
  2. 日志照样落库
  3. 如果用户当前在线，再尝试实时推一份到前端

4.2 Scheduler 层

定时去调用：

proactive_service.execute_rule_for_user()

也就是说，Scheduler 不是再写一遍主动逻辑，而是：

帮你定时按下第七阶段那颗“执行主动规则”的按钮。

5. 第八阶段推荐的最小实现策略

为了让第八阶段既好讲，又不把系统搞复杂，我建议你固定用下面这套最小版：

5.1 WebSocket 最小版

先只做三件事：

  1. 建立当前用户连接
  2. 连接成功后发一条欢迎消息
  3. 支持向当前用户推送一条文本测试消息

不要一开始就做：

  1. 房间系统
  2. 多种事件总线
  3. 在线状态矩阵
  4. 聊天流式 token 推送

5.2 Scheduler 最小版

先只做一个周期任务：

proactive_scan_job

它每隔一段时间做一次：

  1. 查活跃用户
  2. 查已启用规则
  3. 逐个用户执行主动规则
  4. 复用第七阶段主动服务

第八阶段先不要急着上：

  1. 多任务编排
  2. 分布式队列
  3. Celery
  4. 定时任务控制台

你现在最重要的是：

先把“自动巡检”这件事跑起来。

7. 第八阶段的核心设计原则

这一阶段最重要的不是代码量，而是边界不要乱。

7.1 WebSocket 不是业务层

WebSocket 只负责：

把已有业务结果实时送出去

它不负责：

  1. 判断是否触发
  2. 决定是否限流
  3. 决定生成什么主动消息

这些还是第七阶段业务层负责。

7.2 Scheduler 不是业务层

Scheduler 只负责：

定时调用已有业务服务

它不负责：

  1. 自己再写一遍规则判断
  2. 自己再写一遍主动消息生成

7.3 proactive_service 仍然是主动链路核心

也就是说，第八阶段新增的一切，都应该围绕：

proactive_service.execute_rule_for_user()

来做增强，而不是绕开它。

8. 第八阶段的事件流应该长什么样

这里建议你把这一张流程图记住，因为它就是答辩时最好讲的一页。

Scheduler 定时触发
        ↓
查活跃用户 + 已启用规则
        ↓
调用 proactive_service.execute_rule_for_user()
        ↓
规则命中 + 窗口允许 + 频率允许
        ↓
创建 ProactiveMessage
        ↓
写 ActiveLog(status="created")
        ↓
如果用户当前有 WebSocket 连接
        ↓
实时推送 proactive_message_created 事件

这里有一个非常关键的工程结论：

即使最后一步 WebSocket 没推成功，

前面的：

  1. 主动消息落库
  2. 主动日志落库

也都必须已经成功。

这说明：

实时推送是增强层，不是主事务核心。

9. 这一阶段不新增数据库表

这一点要讲清楚。

第八阶段虽然加了两个大能力：

  1. WebSocket
  2. Scheduler

但它本质上都是：

对第七阶段已有业务的增强

所以第八阶段正常情况下：

不需要新增数据库表

也就不需要新增 Alembic migration。

如果你发现自己第八阶段又开始想建一堆新表，通常说明设计开始偏了。

10. 先补依赖和配置

10.1 修改 backend/requirements.txt

第八阶段最重要的新依赖就是：

APScheduler

你不需要重写整份 requirements.txt，只要在文件末尾追加这一行：

APScheduler==3.10.4

为什么第八阶段这里选 APScheduler，而不是 Celery

因为你当前项目还是：

单体 FastAPI + 本地数据库 + 比赛型后端

这时候最适合的是：

先上一个轻量、易讲、易调试的调度器。

Celery 当然更强，但会立刻带来：

  1. broker
  2. worker
  3. 任务序列化
  4. 部署复杂度

这对当前阶段明显过重。

11. 先把实时连接管理器写出来

这一层是 WebSocket 最值得独立出来的一层。

它负责：

  1. 用户连接登记
  2. 断线清理
  3. 向某个用户发送 JSON 事件
  4. 给同步业务代码提供“线程安全推送入口”

11.1 新建 app/ws/manager.py

完整代码如下：

from __future__ import annotations

import asyncio
from collections import defaultdict
from threading import Lock
from typing import Any

from fastapi import WebSocket


class ConnectionManager:
    def __init__(self) -> None:
        self._connections: dict[int, set[WebSocket]] = defaultdict(set)
        self._lock = Lock()
        self._loop: asyncio.AbstractEventLoop | None = None

    def bind_loop(self, loop: asyncio.AbstractEventLoop) -> None:
        self._loop = loop

    async def connect(self, user_id: int, websocket: WebSocket) -> None:
        await websocket.accept()
        with self._lock:
            self._connections[user_id].add(websocket)

    def disconnect(self, user_id: int, websocket: WebSocket) -> None:
        with self._lock:
            sockets = self._connections.get(user_id)
            if not sockets:
                return

            sockets.discard(websocket)
            if not sockets:
                self._connections.pop(user_id, None)

    async def send_json_to_user(
        self,
        user_id: int,
        payload: dict[str, Any],
    ) -> int:
        with self._lock:
            sockets = list(self._connections.get(user_id, set()))

        delivered_count = 0
        broken_sockets: list[WebSocket] = []

        for websocket in sockets:
            try:
                await websocket.send_json(payload)
                delivered_count += 1
            except Exception:
                broken_sockets.append(websocket)

        for websocket in broken_sockets:
            self.disconnect(user_id, websocket)

        return delivered_count

    def send_json_to_user_sync(
        self,
        user_id: int,
        payload: dict[str, Any],
        timeout: float = 5,
    ) -> int:
        if self._loop is None or self._loop.is_closed():
            return 0

        future = asyncio.run_coroutine_threadsafe(
            self.send_json_to_user(user_id, payload),
            self._loop,
        )
        try:
            return future.result(timeout=timeout)
        except Exception:
            return 0

    def count_for_user(self, user_id: int) -> int:
        with self._lock:
            return len(self._connections.get(user_id, set()))

    def reset(self) -> None:
        with self._lock:
            self._connections.clear()
        self._loop = None


realtime_manager = ConnectionManager()

11.2 这份代码的关键点

1. 为什么这里按 user_id 管理连接

因为你当前正式后端主链路已经全面按 user_id 统一了。

比如：

  1. Conversation.user_id
  2. Record.user_id
  3. ActiveLog.user_id
  4. 第七阶段推荐的 ProactiveWindow.user_id
  5. 第七阶段推荐的 ProactiveMessage.user_id

所以第八阶段的 WebSocket 也应该继续按 user_id 管。

不要再回退到旧草稿那种按 profile_id 管理连接的写法。

2. 为什么这里要有 send_json_to_user_sync

这是第八阶段最容易忽略、但最关键的一点。

因为：

  1. WebSocket 发送本身是异步的
  2. 但你的 proactive_service 和 APScheduler 任务都是同步调用风格

所以这里必须有一个桥接方法，让同步业务代码也能安全触发实时推送。

3. 为什么推送失败时返回 0，而不是抛异常

因为实时推送是增强层。

它失败了，不应该把：

  1. 主动消息落库
  2. 主动日志落库

整个都搞失败。

这也是第八阶段最重要的稳定性原则之一。

12. 再定义一个最小实时推送 schema

虽然 WebSocket 不像 HTTP 那样强制 response_model，但第八阶段最好还是给测试推送准备一份干净的 schema。

13. 写实时路由

这一层建议同时提供：

  1. WebSocket 连接入口
  2. 一个 HTTP 测试推送入口

13.1 为什么这里不建议直接在 WebSocket 上做复杂认证依赖

很多人会想直接把：

Depends(get_current_active_user)

塞进 WebSocket 路由。

技术上不是完全不行，但有个问题：

get_db 这种请求级依赖，会把数据库会话跟着整个 WebSocket 长连接一起存活。

而 WebSocket 可能持续几分钟、几十分钟，甚至更久。

这会让数据库会话生命周期很别扭。

所以第八阶段更稳的做法是：

  1. WebSocket 只拿 token
  2. 自己短暂开一个 SessionLocal()
  3. 做完用户鉴权立刻关闭数据库会话

这样长连接更干净。

13.2 新建 app/api/routes/realtime.py

完整代码如下：

from __future__ import annotations

from datetime import UTC, datetime

from fastapi import APIRouter, Depends, Query, WebSocket, WebSocketDisconnect

from app.api.deps import get_current_active_user
from app.core.config import settings
from app.core.exception import BusinessException
from app.core.response import success_response
from app.core.security import InvalidTokenError, decode_access_token
from app.db.session import SessionLocal
from app.models.user import User
from app.schemas.common import ApiResponse
from app.schemas.realtime import RealtimePushTestData, RealtimePushTestRequest
from app.ws.manager import realtime_manager

router = APIRouter(prefix="/realtime", tags=["realtime"])


def authenticate_websocket_user(token: str) -> User:
    credentials_error = BusinessException(
        code=40100,
        message="invalid or expired token",
        status_code=401,
    )

    try:
        payload = decode_access_token(token)
    except InvalidTokenError:
        raise credentials_error

    user_id = payload.get("sub")
    if not user_id:
        raise credentials_error

    db = SessionLocal()
    try:
        user = db.get(User, int(user_id))
        if not user:
            raise credentials_error
        if not user.is_active:
            raise BusinessException(
                code=40300,
                message="inactive user",
                status_code=403,
            )
        return user
    finally:
        db.close()


@router.websocket("/ws")
async def websocket_endpoint(
    websocket: WebSocket,
    token: str = Query(...),
):
    if not settings.WS_ENABLED:
        await websocket.close(code=1008, reason="websocket is disabled")
        return

    try:
        current_user = authenticate_websocket_user(token)
    except BusinessException as exc:
        await websocket.close(code=1008, reason=exc.message)
        return

    await realtime_manager.connect(current_user.id, websocket)
    await realtime_manager.send_json_to_user(
        current_user.id,
        {
            "event": "connected",
            "message": "realtime channel connected",
            "user_id": current_user.id,
            "connected_at": datetime.now(UTC).isoformat(),
        },
    )

    try:
        while True:
            text = await websocket.receive_text()
            if text.strip().lower() == "ping":
                await websocket.send_json(
                    {
                        "event": "pong",
                        "message": "pong",
                        "sent_at": datetime.now(UTC).isoformat(),
                    }
                )
    except WebSocketDisconnect:
        realtime_manager.disconnect(current_user.id, websocket)
    except Exception:
        realtime_manager.disconnect(current_user.id, websocket)
        raise


@router.post(
    "/test-push/me",
    response_model=ApiResponse[RealtimePushTestData],
)
def send_test_push_to_current_user(
    payload: RealtimePushTestRequest,
    current_user: User = Depends(get_current_active_user),
):
    delivered_connection_count = realtime_manager.send_json_to_user_sync(
        current_user.id,
        {
            "event": "manual_test_push",
            "data": {
                "content": payload.content,
                "sent_at": datetime.now(UTC).isoformat(),
            },
        },
    )
    return success_response(
        data=RealtimePushTestData(
            event="manual_test_push",
            content=payload.content,
            delivered_connection_count=delivered_connection_count,
        ),
        message="realtime push sent",
    )

13.3 这份实时路由在做什么

1. GET / POST 都没有，WebSocket 走的是 @router.websocket

它最终会提供一个类似下面的连接入口：

ws://127.0.0.1:8000/api/v1/realtime/ws?token=你的JWT

这里要注意，因为这个路由挂在 api_router 里，所以路径前面仍然会带：

/api/v1

2. 为什么欢迎消息还要走 manager 发

因为这能保证：

你后续所有实时消息都走同一个发送口径。

也就是说：

  1. 测试消息走 manager
  2. 主动消息走 manager
  3. 欢迎消息也走 manager

这会让第八阶段结构更统一。

3. 为什么测试推送接口保留 HTTP

因为这能让你把 WebSocket 调试从业务调试里分离出来。

当你怀疑：

“为什么前端没收到实时消息”

你就可以先调用：

POST /api/v1/realtime/test-push/me

如果这都收不到，那问题通常就不在主动服务，而在：

  1. 连接没建好
  2. token 不对
  3. 前端监听没写对

api_router.include_router(trigger_rules_router)
api_router.include_router(active_logs_router)
api_router.include_router(proactive_router)
api_router.include_router(realtime_router)

14.2 为什么 WebSocket 路由也可以挂在 api_router

因为 FastAPI 的 APIRouter 不只是能注册 HTTP 路由，也能注册 WebSocket 路由。

这样做的好处是：

  1. 你的所有 API 入口仍然统一管理
  2. API_V1_PREFIX 仍然统一生效
  3. 结构上更整洁

15. 升级 proactive_service.py，让它在创建主动消息后可选实时推送

这一步是第八阶段衔接第七阶段最重要的一步。

第七阶段的 proactive_service 已经能：

  1. 评估规则
  2. 检查窗口
  3. 检查频率
  4. 创建 ProactiveMessage
  5. 写 ActiveLog

第八阶段只给它加一层：

创建成功后，尝试实时推送

注意：

这里只是“尝试”推送。

推送失败不能影响主链路成功。

15.1 替换 app/services/proactive_service.py

完整代码如下：

from __future__ import annotations

from datetime import UTC, datetime
from typing import Any

from sqlalchemy import func, select
from sqlalchemy.orm import Session

from app.models.active_log import ActiveLog
from app.models.proactive_message import ProactiveMessage
from app.models.trigger_rule import TriggerRule
from app.services.active_log_service import active_log_service
from app.services.proactive_message_service import proactive_message_service
from app.services.proactive_window_service import proactive_window_service
from app.services.trigger_rule_service import trigger_rule_service
from app.ws.manager import realtime_manager


class ProactiveService:
    def execute_rule_for_user(
        self,
        db: Session,
        user_id: int,
        rule_id: int,
    ) -> dict[str, Any]:
        rule = trigger_rule_service.get_or_raise(db, rule_id)
        request_payload = {
            "rule_id": rule.id,
            "trigger_type": rule.trigger_type,
            "condition_json": rule.condition_json,
            "executed_at": datetime.now(UTC).isoformat(),
        }

        try:
            evaluation = trigger_rule_service.evaluate_for_user(db, user_id, rule)

            if not evaluation["triggered"]:
                log = active_log_service.create_log(
                    db=db,
                    user_id=user_id,
                    trigger_rule_id=rule.id,
                    action_type="generate_proactive_message",
                    status="skipped",
                    request_payload=request_payload,
                    response_payload=evaluation,
                )
                return {
                    "log_id": log.id,
                    "rule_id": rule.id,
                    "triggered": False,
                    "message_created": False,
                    "status": "skipped",
                    "reason": evaluation["reason"],
                    "proactive_message": None,
                }

            window = proactive_window_service.get_or_create_for_user(db, user_id)
            allowed, window_reason = proactive_window_service.allow_trigger_now(window)

            if not allowed:
                response_payload = {
                    **evaluation,
                    "window_reason": window_reason,
                    "quiet_hours_start": window.quiet_hours_start,
                    "quiet_hours_end": window.quiet_hours_end,
                }
                log = active_log_service.create_log(
                    db=db,
                    user_id=user_id,
                    trigger_rule_id=rule.id,
                    action_type="generate_proactive_message",
                    status="blocked",
                    request_payload=request_payload,
                    response_payload=response_payload,
                )
                return {
                    "log_id": log.id,
                    "rule_id": rule.id,
                    "triggered": True,
                    "message_created": False,
                    "status": "blocked",
                    "reason": window_reason,
                    "proactive_message": None,
                }

            today_count = self.count_today_created(db, user_id)
            if today_count >= window.max_trigger_per_day:
                response_payload = {
                    **evaluation,
                    "today_count": today_count,
                    "max_trigger_per_day": window.max_trigger_per_day,
                }
                log = active_log_service.create_log(
                    db=db,
                    user_id=user_id,
                    trigger_rule_id=rule.id,
                    action_type="generate_proactive_message",
                    status="rate_limited",
                    request_payload=request_payload,
                    response_payload=response_payload,
                )
                return {
                    "log_id": log.id,
                    "rule_id": rule.id,
                    "triggered": True,
                    "message_created": False,
                    "status": "rate_limited",
                    "reason": "max trigger per day exceeded",
                    "proactive_message": None,
                }

            title, content = self.build_message_content(rule, evaluation)
            proactive_message = proactive_message_service.create_message(
                db=db,
                user_id=user_id,
                trigger_rule=rule,
                title=title,
                content=content,
            )

            delivered_connection_count = self._push_created_message(
                user_id=user_id,
                proactive_message=proactive_message,
            )

            response_payload = {
                **evaluation,
                "proactive_message_id": proactive_message.id,
                "title": proactive_message.title,
                "status": proactive_message.status,
                "realtime_delivered_connection_count": delivered_connection_count,
            }
            log = active_log_service.create_log(
                db=db,
                user_id=user_id,
                trigger_rule_id=rule.id,
                action_type="generate_proactive_message",
                status="created",
                request_payload=request_payload,
                response_payload=response_payload,
            )
            return {
                "log_id": log.id,
                "rule_id": rule.id,
                "triggered": True,
                "message_created": True,
                "status": "created",
                "reason": evaluation["reason"],
                "proactive_message": proactive_message,
            }
        except Exception as exc:
            active_log_service.create_log(
                db=db,
                user_id=user_id,
                trigger_rule_id=rule.id,
                action_type="generate_proactive_message",
                status="failed",
                request_payload=request_payload,
                response_payload={"error": str(exc)},
            )
            raise

    def count_today_created(
        self,
        db: Session,
        user_id: int,
    ) -> int:
        day_start = datetime.now(UTC).replace(
            hour=0,
            minute=0,
            second=0,
            microsecond=0,
        )

        stmt = (
            select(func.count())
            .select_from(ActiveLog)
            .where(
                ActiveLog.user_id == user_id,
                ActiveLog.action_type == "generate_proactive_message",
                ActiveLog.status == "created",
                ActiveLog.created_at >= day_start,
            )
        )
        return db.scalar(stmt) or 0

    def build_message_content(
        self,
        rule: TriggerRule,
        evaluation: dict[str, Any],
    ) -> tuple[str, str]:
        if rule.trigger_type == "days_without_record":
            return self._build_days_without_record_message(evaluation)

        if rule.trigger_type == "recent_chat_keyword":
            return self._build_recent_chat_keyword_message(evaluation)

        return (
            "主动关怀提醒",
            "系统检测到你可能需要一次健康关注，你可以打开应用查看建议。",
        )

    def _build_days_without_record_message(
        self,
        evaluation: dict[str, Any],
    ) -> tuple[str, str]:
        record_type = str(evaluation.get("record_type", "健康数据"))
        days_without_record = int(evaluation.get("days_without_record", 1))
        record_label = self._record_type_label(record_type)

        title = "健康记录提醒"
        content = (
            f"我注意到你已经 {days_without_record} 天没有记录{record_label}了。"
            "如果今天方便，可以补记一条；如果最近状态有波动，也建议尽快测一次。"
        )
        return title, content

    def _build_recent_chat_keyword_message(
        self,
        evaluation: dict[str, Any],
    ) -> tuple[str, str]:
        keywords = evaluation.get("matched_keywords") or []
        keyword_text = "、".join(str(item) for item in keywords) or "当前困扰"

        title = "健康关怀提示"
        content = (
            f"我注意到你最近的表达里出现了“{keyword_text}”相关内容。"
            "如果你愿意，可以继续说说现在最困扰你的点，我会先帮你整理成几个可执行的小建议。"
        )
        return title, content

    def _record_type_label(self, record_type: str) -> str:
        mapping = {
            "blood_pressure": "血压",
            "blood_glucose": "血糖",
            "sleep": "睡眠",
            "medication": "用药",
            "mood": "情绪",
            "checkin": "健康打卡",
        }
        return mapping.get(record_type, record_type)

    def _push_created_message(
        self,
        user_id: int,
        proactive_message: ProactiveMessage,
    ) -> int:
        payload = {
            "event": "proactive_message_created",
            "data": {
                "id": proactive_message.id,
                "user_id": proactive_message.user_id,
                "trigger_rule_id": proactive_message.trigger_rule_id,
                "trigger_type": proactive_message.trigger_type,
                "title": proactive_message.title,
                "content": proactive_message.content,
                "status": proactive_message.status,
                "created_at": proactive_message.created_at.isoformat(),
                "displayed_at": (
                    proactive_message.displayed_at.isoformat()
                    if proactive_message.displayed_at
                    else None
                ),
            },
        }
        return realtime_manager.send_json_to_user_sync(user_id, payload)


proactive_service = ProactiveService()

15.2 这里最关键的设计点

1. 先落库，再推送

顺序一定要是：

  1. 创建 ProactiveMessage
  2. 尝试实时推送

不能反过来。

因为你真正的主事实应该是：

主动消息已经被系统创建出来了

实时推送只是“更快送达”的附加能力。

2. 推送结果只作为附加信息记录

你会看到：

realtime_delivered_connection_count

只是被写进了日志的 response_payload

它不是成功与否的主判断标准。

3. WebSocket 推送失败不会让主动服务失败

这点必须坚持。

否则第八阶段一加实时推送，系统稳定性反而会下降。

16. 写 Scheduler

这一层负责的事情非常明确：

按固定周期自动巡检用户和规则

16.1 第八阶段为什么建议先用 interval，不先用 cron

因为第八阶段的目标首先是：

证明自动巡检链路是通的

所以第一版最适合先用：

interval

比如每 1 分钟或者每 30 分钟扫一次。

这样你调试时最方便。

等你链路稳定了，再把任务从 interval 改成：

  1. 每天 8 点
  2. 每小时
  3. 每晚 23 点归档

这种 cron 风格，会更顺。

16.2 新建 app/core/scheduler.py

完整代码如下：

from __future__ import annotations

import logging

from apscheduler.schedulers.background import BackgroundScheduler
from sqlalchemy import select

from app.core.config import settings
from app.db.session import SessionLocal
from app.models.trigger_rule import TriggerRule
from app.models.user import User
from app.services.proactive_service import proactive_service

logger = logging.getLogger(__name__)

scheduler = BackgroundScheduler(timezone=settings.SCHEDULER_TIMEZONE)


def proactive_scan_job() -> None:
    logger.info("proactive scan job started")

    db = SessionLocal()
    try:
        users = list(
            db.scalars(
                select(User)
                .where(User.is_active.is_(True))
                .order_by(User.id.asc())
            ).all()
        )
        rules = list(
            db.scalars(
                select(TriggerRule)
                .where(TriggerRule.enabled.is_(True))
                .order_by(
                    TriggerRule.priority.asc(),
                    TriggerRule.id.asc(),
                )
            ).all()
        )

        if not users:
            logger.info("proactive scan skipped: no active users")
            return

        if not rules:
            logger.info("proactive scan skipped: no enabled trigger rules")
            return

        for user in users:
            for rule in rules:
                try:
                    result = proactive_service.execute_rule_for_user(
                        db=db,
                        user_id=user.id,
                        rule_id=rule.id,
                    )

                    if result["status"] in {"created", "blocked", "rate_limited"}:
                        break
                except Exception:
                    logger.exception(
                        "proactive scan failed for user_id=%s rule_id=%s",
                        user.id,
                        rule.id,
                    )
                    continue
    finally:
        db.close()

    logger.info("proactive scan job finished")


def start_scheduler() -> None:
    if not settings.SCHEDULER_ENABLED:
        logger.info("scheduler is disabled by config")
        return

    if scheduler.get_job("proactive_scan_job") is None:
        scheduler.add_job(
            proactive_scan_job,
            trigger="interval",
            minutes=settings.SCHEDULER_SCAN_INTERVAL_MINUTES,
            id="proactive_scan_job",
            replace_existing=True,
            max_instances=1,
            coalesce=True,
        )

    if not scheduler.running:
        scheduler.start()
        logger.info(
            "scheduler started, proactive_scan_job interval=%s minutes",
            settings.SCHEDULER_SCAN_INTERVAL_MINUTES,
        )


def shutdown_scheduler() -> None:
    if scheduler.running:
        scheduler.shutdown(wait=False)
        logger.info("scheduler stopped")

16.3 这份调度器代码的关键点

1. 为什么任务里直接查 User 和 TriggerRule

因为 Scheduler 本质是一个装配和巡检层。

它当然可以调用 service，但这里查：

  1. 活跃用户列表
  2. 已启用规则列表

属于非常基础的装配查询，放在这里是合理的。

真正的主动执行仍然完全复用：

proactive_service.execute_rule_for_user()

2. 为什么每个用户在出现 created / blocked / rate_limited 后就 break

这是一个非常实用的小优化。

如果你不 break，会发生什么？

同一个用户在一次扫描里，后面的规则可能继续被执行，然后产生很多重复的：

  1. blocked
  2. rate_limited

日志。

这样会让第八阶段演示和排查都变得很乱。

所以当前最小版的策略是：

对每个用户按优先级从高到低扫规则，一旦已经：

  1. 成功创建主动消息
  2. 被窗口阻塞
  3. 被频率限制

就结束这个用户本轮扫描。

3. 为什么这里用 BackgroundScheduler

因为你当前主动服务是同步风格、数据库访问也是同步风格。

BackgroundScheduler 在第八阶段最省心：

  1. 好接入
  2. 容易理解
  3. 和当前项目风格匹配

17. 用 lifespan 把 Scheduler 和 WebSocket 运行时整合到主程序

第八阶段最值得你吸收的工程点之一，就是：

用 FastAPI 的 lifespan 做应用启动和关闭装配。

17.1 为什么这里不用旧式 startup / shutdown 事件

因为 lifespan 更适合把：

  1. 绑定事件循环
  2. 启动调度器
  3. 应用关闭时回收资源

这些动作统一放到一个地方。

你后面讲工程结构时也更清楚。

17.2 替换 app/main.py

完整代码如下：

import asyncio
from contextlib import asynccontextmanager

from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

from app.api.router import api_router
from app.core.config import settings
from app.core.exception import register_exception_handlers
from app.core.scheduler import shutdown_scheduler, start_scheduler
from app.ws.manager import realtime_manager


@asynccontextmanager
async def lifespan(app: FastAPI):
    realtime_manager.bind_loop(asyncio.get_running_loop())
    start_scheduler()

    try:
        yield
    finally:
        shutdown_scheduler()
        realtime_manager.reset()


def create_application() -> FastAPI:
    app = FastAPI(
        title=settings.PROJECT_NAME,
        debug=settings.DEBUG,
        version="0.1.0",
        lifespan=lifespan,
    )

    app.add_middleware(
        CORSMiddleware,
        allow_origins=settings.BACKEND_CORS_ORIGINS or ["*"],
        allow_credentials=True,
        allow_methods=["*"],
        allow_headers=["*"],
    )

    app.include_router(api_router, prefix=settings.API_V1_PREFIX)
    register_exception_handlers(app)

    return app


app = create_application()

17.3 这里最关键的两个动作是什么

1. realtime_manager.bind_loop(asyncio.get_running_loop())

这是为了让同步代码也能安全触发异步 WebSocket 推送。

如果不绑定事件循环，前面写的：

send_json_to_user_sync()

就不知道该把协程扔到哪个 loop 上执行。

2. start_scheduler() / shutdown_scheduler()

这让调度器和应用生命周期真正绑定起来：

  1. 应用启动，调度器启动
  2. 应用关闭，调度器关闭

这就是第八阶段里 main.py 的核心职责。

18. 第八阶段最终的文件改动顺序建议

你真正动手时，建议严格按这个顺序来：

  1. requirements.txt 追加 APScheduler
  2. app/core/config.py 增加配置项
  3. app/ws/manager.py 新建连接管理器
  4. app/schemas/realtime.py 新建实时测试 schema
  5. app/api/routes/realtime.py 新建实时路由
  6. app/api/router.py 挂上实时路由
  7. app/services/proactive_service.py 升级成支持实时推送版
  8. app/core/scheduler.py 新建调度器
  9. app/main.py 切换成 lifespan 装配版

原因：

  1. 没有 manager，你后面实时路由和主动服务都没法写
  2. 没有实时路由，你很难先独立验证 WebSocket
  3. 没有 proactive_service 升级，Scheduler 只会自动落库，不能联动实时推送
  4. 没有 main.py 生命周期整合，Scheduler 就没有正确的启动时机

19. 第八阶段最容易写错的地方

19.1 把 WebSocket 推送放到主事务前面

顺序一定不能反。

应该先：

  1. 落主动消息
  2. 写主动日志
  3. 再尝试推送

19.2 在 WebSocket 路由里长时间持有数据库会话

这一点前面已经讲过。

WebSocket 是长连接，不适合把请求级 Session 一直挂着不放。

19.3 调度器里重写业务逻辑

你如果在 scheduler.py 里又重写一份：

  1. 规则判断
  2. 主动窗口判断
  3. 主动消息创建

那说明层次已经乱了。

调度器只应该调：

proactive_service.execute_rule_for_user()

19.4 实时推送失败就把整个主动服务判失败

这是第八阶段很常见的误区。

WebSocket 推送失败只说明：

用户当前可能不在线，或者连接断了

不应该说明：

主动消息创建失败

19.5 一上来就做聊天流式输出

第八阶段当然可以往后做聊天流式输出，但不应该作为第一个目标。

因为你当前项目最有价值、最能展示主动性的实时场景其实是：

主动提醒实时弹出

先把这个做稳，比先做 SSE 或 token 流推送更有性价比。

20. WebSocket 怎么测

WebSocket 不能像普通 HTTP 接口那样直接靠 Swagger 页面点一下就完整演示。

所以你最好按下面顺序测。

20.1 先登录拿 JWT

还是走你当前已有的：

POST /api/v1/auth/login

拿到：

access_token

20.2 用浏览器控制台或者前端页面建立连接

最小示例：

const token = "这里换成你自己的 access_token";
const ws = new WebSocket(
  `ws://127.0.0.1:8000/api/v1/realtime/ws?token=${token}`
);

ws.onopen = () => {
  console.log("ws connected");
  ws.send("ping");
};

ws.onmessage = (event) => {
  console.log("ws message:", JSON.parse(event.data));
};

ws.onclose = (event) => {
  console.log("ws closed:", event.code, event.reason);
};

如果连接成功，你应该先收到一条：

connected

事件。

然后发送 ping 后，应该能收到：

pong

事件。

20.3 用 HTTP 测试推送接口验证实时通道

调用：

POST /api/v1/realtime/test-push/me

请求体示例：

{
  "content": "这是一条实时测试消息"
}

你应该看到两边同时成立：

  1. HTTP 返回里有 delivered_connection_count
  2. 浏览器控制台里收到 manual_test_push 事件

如果 HTTP 返回 delivered_connection_count = 0，通常意味着：

  1. WebSocket 还没连上
  2. token 不是同一个用户
  3. 连接已经断开

21. Scheduler 怎么测

Scheduler 的测试思路要和 WebSocket 分开。

21.1 先把扫描间隔改小

临时把 .env 里的：

SCHEDULER_SCAN_INTERVAL_MINUTES=1

这样更容易演示。

21.2 先确保第七阶段前置已经准备好

至少要有：

  1. 一条启用状态的 TriggerRule
  2. 一个允许主动的 ProactiveWindow
  3. 能命中的测试数据

比如：

  1. 最近 3 天没血压记录
  2. 最近聊天里含焦虑关键词

21.3 启动服务后观察 3 个地方

1. 终端日志

你应该看到调度器启动，以及 proactive scan job started / finished

2. GET /api/v1/active-logs

你应该能看到新的：

generate_proactive_message

日志。

3. GET /api/v1/proactive/messages

如果规则命中且允许创建主动消息，这里应该会新增记录。

4. 如果当前 WebSocket 在线

浏览器应该能直接收到：

proactive_message_created

事件。

这就说明第八阶段真正形成闭环了：

自动检查 -> 主动执行 -> 落库 -> 留痕 -> 实时到达

22. 第八阶段完成后，你现在真正具备了什么

如果你按这份教程完整补完，第八阶段结束后你已经具备：

  3. HTTP 测试推送接口
  4. proactive_service 创建主动消息后的实时联动
  5. APScheduler 自动巡检任务
  6. main.py 生命周期装配

更重要的是，你的系统已经从：

“后端能生成一条主动消息”

升级成：

“系统能自动发现时机、自动生成消息，并在用户在线时立刻送达”

这已经非常接近真实产品体验了。

23. 第八阶段和最终演示怎么衔接

路线 1：WebSocket 单独演示

  1. 你的业务不是假数据弹窗
