# Healthy System FastAPI 第三阶段详细教程

## 1. 这一阶段到底要解决什么

第三阶段只做一件核心事:

把 `Record` 做成一个完整、清晰、能直接在 Swagger 演示的业务 CRUD 模块。

这一阶段的目标不是一上来就把健康记录设计得很复杂，而是先把下面这条完整链路真正走通:

1. ORM 模型
2. Pydantic schema
3. service 业务层
4. router 接口层
5. CRUD
6. 数据校验
7. 分页
8. 条件筛选
9. 当前用户权限归属

你把第三阶段做扎实，后面的聊天上下文、主动提醒、健康分析，基本都只是复用这一套开发方式。

---

## 2. 第三阶段完成后的验收标准

做到下面这些，就算第三阶段合格:

1. `Record` 模块 CRUD 完整
2. 每条记录都归属当前登录用户
3. 列表支持分页
4. 列表支持按类型筛选
5. 列表支持按时间范围筛选
6. 用户不能查看、修改、删除别人的记录
7. `/docs` 里可以直接完成演示

---

## 3. 为什么第三阶段先做 `Record`

因为它是最适合练完整业务闭环的模块。

`Record` 比 `Conversation`、`Message`、`Trigger` 都更独立，依赖更少，特别适合你现在这个阶段。

它会逼着你把下面这些能力真正练会:

1. 建表
2. 写 schema
3. 写 service
4. 写 route
5. 写列表接口
6. 写筛选条件
7. 做用户权限限制

如果第三阶段做扎实，后面大部分业务模块都可以照这个模板继续开发。

---

## 4. 这一阶段先不做什么

这一阶段先不要扩展这些内容:

1. 不做图表统计
2. 不做 AI 解释
3. 不做复杂指标拆分
4. 不做多表联查
5. 不做 `value_json` 这种异构高级设计
6. 不做多 profile 管理

先把“当前用户自己的记录 CRUD”做出来，这才是第三阶段的正确目标。

## 5. 先说明你当前项目里的 `record` 草稿文件

你现在仓库里已经有这些文件:

1. `backend/app/db/models/record.py`
2. `backend/app/schemas/record.py`

但你前两阶段的正式结构已经变成了:

1. ORM 模型放在 `app/models/`
2. Alembic 通过 `app/db/base.py` 找模型
3. `app/models/__init__.py` 负责统一导出模型

所以第三阶段要统一成这套正式结构。

### 5.1 这一阶段的处理原则

1. 正式版 `Record` 放到 `app/models/record.py`
2. `app/schemas/record.py` 直接按第三阶段需求重写
3. `app/db/base.py` 和 `app/models/__init__.py` 要导入新的 `Record`
4. `app/db/models/record.py` 先不要管它，也不要继续 import 它

### 5.2 为什么不建议你现在去删旧草稿文件

因为你现在是学习阶段，删文件容易把自己绕乱。

最稳的做法是:

1. 旧草稿先留着
2. 只保证没有任何正式代码去 import 它
3. 第三阶段正式代码只认 `app/models/record.py`

这样最不容易出错。

## 6. 第三阶段推荐的数据设计

### 6.1 推荐最小模型

这一阶段建议直接用下面这套字段:

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | int | 主键 |
| `user_id` | int | 归属用户 |
| `record_type` | str | 记录类型 |
| `value` | str | 记录值 |
| `unit` | str \| null | 单位 |
| `record_time` | datetime | 业务记录时间 |
| `note` | text \| null | 备注 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

### 6.2 为什么这里先用 `user_id`

你第二阶段已经完成了:

1. `User`
2. `Auth`
3. `Profile`

而第三阶段的验收标准里强调的是:

每条记录归属当前用户

所以这一阶段最直接、最容易理解、最适合演示的做法就是:

让 `Record` 直接挂到 `user_id`

这样做的好处是:

1. 权限判断最简单
2. CRUD 查询最直接
3. 不依赖 profile 是否先创建
4. Swagger 演示最顺

以后如果你要做“一个账号管理多个老人 profile”，再把 `Record` 改成挂 `profile_id` 也不迟。

### 6.3 为什么这里先把 `value` 设计成字符串

因为你当前的记录类型不止一种:

1. 血压: `138/91`
2. 血糖: `6.8`
3. 睡眠: `7.5`
4. 心情: `happy`
5. 用药: `metformin`

如果你现在把 `value` 强行设计成 `float`，立刻就会把自己卡死。

第三阶段先统一成:

`value: str`

这是为了让结构简单、统一、好演示。

### 6.4 推荐的 `record_type`

建议第三阶段先限制成固定枚举，别让前端随便传字符串。

推荐值:

1. `checkin`
2. `blood_pressure`
3. `blood_glucose`
4. `sleep`
5. `medication`
6. `mood`

这样做好处是:

1. Swagger 更清楚
2. 校验更稳定
3. 后面筛选更方便

## 9. 开发顺序建议

这个顺序不要乱:

1. 先做模型 `model`
2. 再做结构 `schema`
3. 再做业务逻辑 `service`
4. 再做接口层 `router`
5. 再补总路由 `app/api/router.py`
6. 再补模型入口 `app/models/__init__.py`
7. 再补 Alembic 入口 `app/db/base.py`
8. 最后做 migration 和 Swagger 验收

原因很简单:

1. 没有模型就没有表
2. 没有 schema 就没有输入输出边界
3. 没有 service，router 会堆业务逻辑
4. 没有 base 导入，Alembic 会生成空 migration
5. 没有 Swagger 验收，你不知道链路是不是真的通了

## 11. 先建 `Record` ORM 模型

### 11.1 新建 `app/models/record.py`

如果这个文件还没有，就新建。  
如果你已经自己建过一个版本，也建议先按下面这份完整代码来。

完整代码如下:

```python
from __future__ import annotations

from datetime import UTC, datetime

from sqlalchemy import DateTime, ForeignKey, Integer, String, Text
from sqlalchemy.orm import Mapped, mapped_column

from app.db.session import Base


class Record(Base):
    __tablename__ = "records"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, index=True)
    user_id: Mapped[int] = mapped_column(
        ForeignKey("users.id", ondelete="CASCADE"),
        index=True,
        nullable=False,
    )
    record_type: Mapped[str] = mapped_column(String(30), index=True, nullable=False)
    value: Mapped[str] = mapped_column(String(100), nullable=False)
    unit: Mapped[str | None] = mapped_column(String(30), nullable=True)
    record_time: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        default=lambda: datetime.now(UTC),
        nullable=False,
        index=True,
    )
    note: Mapped[str | None] = mapped_column(Text, nullable=True)
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        default=lambda: datetime.now(UTC),
        nullable=False,
    )
    updated_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        default=lambda: datetime.now(UTC),
        onupdate=lambda: datetime.now(UTC),
        nullable=False,
    )
```

### 11.2 这份模型代码在做什么

1. `id` 是主键
2. `user_id` 表示记录归属哪个用户
3. `record_type` 表示记录类型
4. `value` 表示记录值
5. `unit` 表示单位
6. `record_time` 表示业务发生时间
7. `note` 表示备注
8. `created_at` 和 `updated_at` 是系统维护时间

### 11.3 这里为什么要区分 `record_time` 和 `created_at`

因为这两个时间不是一回事:

1. `record_time` 是“这件事什么时候发生”
2. `created_at` 是“这条数据什么时候写进数据库”

举例:

你可能晚上 11 点睡觉，第二天早上 8 点才补录昨天的睡眠记录。

这时:

1. `record_time` 应该是昨天晚上
2. `created_at` 是今天早上

所以健康记录列表排序和时间筛选，应该优先围绕 `record_time`。

### 11.4 这里为什么暂时不写 relationship

因为第三阶段的目标是先把 CRUD 跑通。

你当然可以继续给 `User` 增加:

```python
records = relationship(...)
```

但这不是第三阶段最核心的部分。  
为了让你少改一个文件，这一阶段先不加 relationship，也完全可以正常完成业务。

## 12. 把 `Record` 注册到模型入口

如果只写了 `app/models/record.py`，但没有把它导入到统一入口，Alembic 很可能看不到这个表。

所以这一步必须做。

### 12.1 修改 `app/models/__init__.py`

如果你当前这个文件已经存在，直接整文件替换成下面的内容。

完整代码如下:

```python
from app.models.profile import Profile
from app.models.record import Record
from app.models.user import User

__all__ = ["User", "Profile", "Record"]
```

### 12.2 这一步的作用

作用有两个:

1. 统一导出模型
2. 让项目里的其他地方可以从 `app.models` 拿到完整模型集合

### 12.3 修改 `app/db/base.py`

如果你当前这个文件已经存在，直接整文件替换成下面的内容。

完整代码如下:

```python
from app.db.session import Base
from app.models.profile import Profile  # noqa: F401
from app.models.record import Record  # noqa: F401
from app.models.user import User  # noqa: F401

__all__ = ["Base", "User", "Profile", "Record"]
```

### 12.4 这一步为什么必须做

因为 Alembic 自动生成 migration 时，要通过 `Base.metadata` 扫描模型。

如果 `Record` 没有被导入到这里，后果通常就是:

1. migration 文件是空的
2. `records` 表没有被创建
3. 你后面接口一运行就报表不存在

## 13. 写 `Record` 模块的 Schema

schema 的作用不是“再写一遍表字段”，而是明确:

1. 新增时前端允许传什么
2. 更新时前端允许传什么
3. 返回给前端的结构长什么样
4. 列表接口的分页和筛选参数长什么样

### 13.1 修改 `app/schemas/record.py`

你当前项目里已经有这个文件。  
第三阶段建议直接整文件替换，按下面这份来。

完整代码如下:

```python
from datetime import datetime
from enum import Enum

from pydantic import BaseModel, ConfigDict, Field


class RecordType(str, Enum):
    checkin = "checkin"
    blood_pressure = "blood_pressure"
    blood_glucose = "blood_glucose"
    sleep = "sleep"
    medication = "medication"
    mood = "mood"


class RecordBase(BaseModel):
    record_type: RecordType
    value: str = Field(..., min_length=1, max_length=100)
    unit: str | None = Field(default=None, max_length=30)
    note: str | None = Field(default=None, max_length=500)


class RecordCreate(RecordBase):
    record_time: datetime | None = None


class RecordUpdate(BaseModel):
    record_type: RecordType | None = None
    value: str | None = Field(default=None, min_length=1, max_length=100)
    unit: str | None = Field(default=None, max_length=30)
    note: str | None = Field(default=None, max_length=500)
    record_time: datetime | None = None


class RecordRead(RecordBase):
    id: int
    user_id: int
    record_time: datetime
    created_at: datetime
    updated_at: datetime

    model_config = ConfigDict(from_attributes=True)


class RecordQueryParams(BaseModel):
    page: int = Field(default=1, ge=1)
    page_size: int = Field(default=10, ge=1, le=50)
    record_type: RecordType | None = None
    start_time: datetime | None = None
    end_time: datetime | None = None


class RecordListData(BaseModel):
    items: list[RecordRead]
    total: int
    page: int
    page_size: int
    total_pages: int
```

### 13.2 每个 schema 的作用

#### `RecordType`

作用:

把 `record_type` 限制在固定范围内。

好处:

1. 前端不容易拼错
2. Swagger 下拉更清楚
3. 后续筛选更稳定

#### `RecordBase`

作用:

把新增和读取共用的基础字段抽出来。

#### `RecordCreate`

作用:

新增记录时使用。

这里让 `record_time` 可不传，是为了:

如果用户不传时间，就默认用当前时间。

#### `RecordUpdate`

作用:

更新记录时使用。

这里所有字段都允许不传，因为更新通常是局部更新。

#### `RecordRead`

作用:

返回给前端的读取结构。

为什么要单独写这个结构:

因为返回体里需要包含:

1. `id`
2. `user_id`
3. `created_at`
4. `updated_at`

这些字段在创建时不应该让前端自己传。

#### `RecordQueryParams`

作用:

把列表接口的分页和筛选参数统一收口。

#### `RecordListData`

作用:

定义分页列表最终返回的数据结构。

## 14. 写 `RecordService`

第三阶段从这里开始，真正进入业务逻辑层。

service 层主要负责:

1. 组装查询条件
2. 做当前用户归属判断
3. 做不存在时的业务异常
4. 做更新逻辑
5. 做删除逻辑

你现在项目第二阶段还没有强推 repository 层，所以第三阶段继续沿用当前风格:

`router -> service -> Session/ORM`

### 14.1 新建 `app/services/record_service.py`

完整代码如下:

```python
from datetime import UTC, datetime

from sqlalchemy import func, select
from sqlalchemy.orm import Session

from app.core.exception import BusinessException
from app.models.record import Record
from app.schemas.record import RecordCreate, RecordQueryParams, RecordUpdate


class RecordService:
    def create_for_user(
        self,
        db: Session,
        user_id: int,
        payload: RecordCreate,
    ) -> Record:
        record = Record(
            user_id=user_id,
            record_type=payload.record_type.value,
            value=payload.value,
            unit=payload.unit,
            record_time=payload.record_time or datetime.now(UTC),
            note=payload.note,
        )
        db.add(record)
        db.commit()
        db.refresh(record)
        return record

    def get_or_raise(
        self,
        db: Session,
        user_id: int,
        record_id: int,
    ) -> Record:
        stmt = select(Record).where(
            Record.id == record_id,
            Record.user_id == user_id,
        )
        record = db.scalar(stmt)
        if not record:
            raise BusinessException(
                code=40421,
                message="record not found",
                status_code=404,
            )
        return record

    def list_for_user(
        self,
        db: Session,
        user_id: int,
        query: RecordQueryParams,
    ) -> tuple[list[Record], int]:
        filters = [Record.user_id == user_id]

        if query.record_type:
            filters.append(Record.record_type == query.record_type.value)

        if query.start_time:
            filters.append(Record.record_time >= query.start_time)

        if query.end_time:
            filters.append(Record.record_time <= query.end_time)

        total = db.scalar(
            select(func.count()).select_from(Record).where(*filters)
        ) or 0

        stmt = (
            select(Record)
            .where(*filters)
            .order_by(Record.record_time.desc(), Record.id.desc())
            .offset((query.page - 1) * query.page_size)
            .limit(query.page_size)
        )
        items = list(db.scalars(stmt).all())
        return items, total

    def update_for_user(
        self,
        db: Session,
        user_id: int,
        record_id: int,
        payload: RecordUpdate,
    ) -> Record:
        record = self.get_or_raise(db, user_id, record_id)
        update_data = payload.model_dump(exclude_unset=True)

        if "record_type" in update_data:
            if update_data["record_type"] is None:
                raise BusinessException(
                    code=40021,
                    message="record_type cannot be null",
                    status_code=400,
                )
            update_data["record_type"] = update_data["record_type"].value

        if "value" in update_data and update_data["value"] is None:
            raise BusinessException(
                code=40022,
                message="value cannot be null",
                status_code=400,
            )

        if "record_time" in update_data and update_data["record_time"] is None:
            raise BusinessException(
                code=40023,
                message="record_time cannot be null",
                status_code=400,
            )

        for field, value in update_data.items():
            setattr(record, field, value)

        db.add(record)
        db.commit()
        db.refresh(record)
        return record

    def delete_for_user(
        self,
        db: Session,
        user_id: int,
        record_id: int,
    ) -> None:
        record = self.get_or_raise(db, user_id, record_id)
        db.delete(record)
        db.commit()


record_service = RecordService()
```

### 14.2 这份 service 代码最核心的地方

#### 1. 所有查询都必须带 `user_id`

这是第三阶段最重要的权限原则。

不管是详情、更新还是删除，你都不能只写:

```python
db.get(Record, record_id)
```

因为这样只是按主键查，完全没有限制当前用户归属。

正确思路必须是:

```python
where(Record.id == record_id, Record.user_id == user_id)
```

#### 2. `create_for_user` 里不允许前端传 `user_id`

为什么:

因为 `user_id` 必须来自当前登录用户，不能由前端随便指定。

也就是说:

1. 前端负责传记录内容
2. 后端负责把当前用户身份写进去

#### 3. 列表接口的筛选条件要动态拼

`list_for_user` 里的 `filters` 列表就是干这个的。

一开始先放:

```python
[Record.user_id == user_id]
```

然后根据用户有没有传:

1. `record_type`
2. `start_time`
3. `end_time`

再决定要不要继续往里追加条件。

#### 4. 列表排序为什么推荐 `record_time.desc()`

因为健康记录通常要看“最近发生了什么”，不是看“最近插入数据库的是什么”。

所以排序推荐:

```python
.order_by(Record.record_time.desc(), Record.id.desc())
```

#### 5. 更新逻辑为什么用 `exclude_unset=True`

因为更新接口一般只改一部分字段。

比如前端只想改 `note`，那就不应该把其他没传的字段一起覆盖掉。

## 15. 写 `Record` 路由

路由层要做的事情是:

1. 接收请求
2. 注入数据库会话
3. 注入当前登录用户
4. 调用 service
5. 把结果转成统一响应结构

### 15.1 新建 `app/api/routes/records.py`

完整代码如下:

```python
from datetime import datetime

from fastapi import APIRouter, Depends, Query, status
from sqlalchemy.orm import Session

from app.api.deps import get_current_active_user, get_db
from app.core.exception import BusinessException
from app.core.response import success_response
from app.models.user import User
from app.schemas.common import ApiResponse
from app.schemas.record import (
    RecordCreate,
    RecordListData,
    RecordQueryParams,
    RecordRead,
    RecordType,
    RecordUpdate,
)
from app.services.record_service import record_service

router = APIRouter(prefix="/records", tags=["records"])


def get_record_query_params(
    page: int = Query(default=1, ge=1),
    page_size: int = Query(default=10, ge=1, le=50),
    record_type: RecordType | None = Query(default=None),
    start_time: datetime | None = Query(
        default=None,
        description="ISO8601，例如 2026-04-01T00:00:00Z",
    ),
    end_time: datetime | None = Query(
        default=None,
        description="ISO8601，例如 2026-04-10T23:59:59Z",
    ),
) -> RecordQueryParams:
    if start_time and end_time and start_time > end_time:
        raise BusinessException(
            code=40024,
            message="start_time cannot be greater than end_time",
            status_code=400,
        )

    return RecordQueryParams(
        page=page,
        page_size=page_size,
        record_type=record_type,
        start_time=start_time,
        end_time=end_time,
    )


@router.post(
    "",
    response_model=ApiResponse[RecordRead],
    status_code=status.HTTP_201_CREATED,
)
def create_record(
    payload: RecordCreate,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_active_user),
):
    record = record_service.create_for_user(db, current_user.id, payload)
    return success_response(
        data=RecordRead.model_validate(record),
        message="record created",
    )


@router.get(
    "",
    response_model=ApiResponse[RecordListData],
)
def list_records(
    query: RecordQueryParams = Depends(get_record_query_params),
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_active_user),
):
    items, total = record_service.list_for_user(db, current_user.id, query)
    total_pages = (total + query.page_size - 1) // query.page_size if total else 0

    return success_response(
        data=RecordListData(
            items=[RecordRead.model_validate(item) for item in items],
            total=total,
            page=query.page,
            page_size=query.page_size,
            total_pages=total_pages,
        ),
        message="success",
    )


@router.get(
    "/{record_id}",
    response_model=ApiResponse[RecordRead],
)
def get_record_detail(
    record_id: int,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_active_user),
):
    record = record_service.get_or_raise(db, current_user.id, record_id)
    return success_response(
        data=RecordRead.model_validate(record),
        message="success",
    )


@router.put(
    "/{record_id}",
    response_model=ApiResponse[RecordRead],
)
def update_record(
    record_id: int,
    payload: RecordUpdate,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_active_user),
):
    record = record_service.update_for_user(
        db=db,
        user_id=current_user.id,
        record_id=record_id,
        payload=payload,
    )
    return success_response(
        data=RecordRead.model_validate(record),
        message="record updated",
    )


@router.delete(
    "/{record_id}",
    response_model=ApiResponse[None],
)
def delete_record(
    record_id: int,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_active_user),
):
    record_service.delete_for_user(db, current_user.id, record_id)
    return success_response(message="record deleted")
```

### 15.2 这份路由代码在做什么

#### 1. `router = APIRouter(prefix="/records", tags=["records"])`

作用:

1. 设置这一组接口的统一前缀
2. 设置 Swagger 里的分组名字

#### 2. `get_record_query_params`

这个函数专门负责把列表页的查询参数收口起来。

它主要做三件事:

1. 处理分页参数
2. 处理筛选参数
3. 校验 `start_time` 不能大于 `end_time`

为什么不把这些参数直接全塞到 `list_records` 里:

因为那样路由函数会越来越长，不利于阅读。

#### 3. `create_record`

这个接口负责创建记录。

关键点:

1. 请求体是 `RecordCreate`
2. 当前用户来自 `get_current_active_user`
3. `user_id` 不从前端拿，而是用 `current_user.id`

#### 4. `list_records`

这个接口负责:

1. 分页
2. 筛选
3. 返回分页结构

为什么这里返回的是 `RecordListData` 而不是 `list[RecordRead]`

因为分页列表除了 `items` 之外，你还需要:

1. `total`
2. `page`
3. `page_size`
4. `total_pages`

#### 5. `get_record_detail`

这个接口负责查看单条记录详情。

关键点:

必须走:

```python
record_service.get_or_raise(db, current_user.id, record_id)
```

这样才能保证:

1. 数据存在
2. 并且数据属于当前用户

#### 6. `update_record`

这个接口负责更新记录。

关键点和详情一样:

更新前必须先确认这条记录属于当前用户。

#### 7. `delete_record`

这个接口负责删除记录。

删除的前提也一样:

先按:

1. `record_id`
2. `user_id`

一起查到记录，才能删。

## 19. 如果你想知道 migration 大概应该长什么样

这一节不是让你手敲 migration。


## 22. 这一阶段最容易写错的地方

### 22.1 只按 `record_id` 查，不按 `user_id` 限制

这是最典型的越权 bug。

错误写法:

```python
record = db.get(Record, record_id)
```

正确思路:

```python
where(Record.id == record_id, Record.user_id == current_user.id)
```

### 22.2 让前端传 `user_id`

这是权限设计里非常危险的一种写法。

如果前端可以自己传:

```json
{
  "user_id": 2
}
```

那用户就有机会把记录伪造到别人名下。

所以第三阶段一定要记住:

`user_id` 必须来自当前登录用户，而不是请求体。

### 22.3 把 `value` 设计成数字

这样会直接限制:

1. 血压
2. 心情
3. 用药
4. 文字打卡

第三阶段先统一成字符串，后面要细化时再扩展。

### 22.4 用 `created_at` 代替 `record_time`

健康记录的业务时间应该看 `record_time`，不是 `created_at`。

### 22.5 忘了在 `app/db/base.py` 里导入 `Record`

这会导致 Alembic 看不到 `records` 表。

### 22.6 修改了 `app/db/models/record.py`，却没改 `app/models/record.py`

你当前项目的正式模型入口是 `app/models/`。

如果你把主要精力花在 `app/db/models/record.py` 上，最后很容易出现:

1. 代码写了
2. Alembic 没看到
3. 接口没生效
4. 你以为自己代码错了

实际上只是路径改错了。
