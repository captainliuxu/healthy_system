# Healthy System 前后端对接规范
版本：Phase 8

状态：按 2026-04-14 当前仓库实际代码生成

适用范围：
1. 用户注册
2. 用户登录
3. 获取当前用户
4. 获取/创建/更新个人档案 Profile
5. 健康记录 Record
6. 会话 Conversation
7. 消息 Message
8. 聊天主链路 Chat

说明：

- 本文是给前端直接接入用的，不讨论后端内部 ORM、Service、数据库细节
- 前端应以本文为准，不要再根据旧版 `frontend/src/api/chat.js` 的写法继续扩展

## 1. 先说结论

当前前端如果要接通后端，必须先改 5 件事：

1. 统一把接口前缀改正确
2. 统一处理 `code/message/data`
3. 登录后统一带 `Authorization: Bearer <token>`
4. 聊天必须先有 `conversation_id`，再发消息
5. 健康记录页不能再存 `localStorage`，要改成调用 `/records`

最关键的一点：

当前后端真正的聊天接口不是：

```text
GET /api/chat?msg=xxx
```

而是：

```text
POST /api/v1/chat/send
Authorization: Bearer <token>
Content-Type: application/json
```

请求体必须是：

```json
{
  "conversation_id": 1,
  "content": "今天有点头晕"
}
```

## 3. 前端必须先修正的代理配置

当前 `frontend/vite.config.js` 的代理写法会把 `/api` 重写掉，最终请求不到后端真实地址。

当前错误方向：

```js
rewrite: (path) => path.replace(/^\/api/, '')
```

如果前端请求：

```text
/api/chat/send
```

会被转发成：

```text
/chat/send
```

但后端真实地址是：

```text
/api/v1/chat/send
```

所以前端开发环境必须二选一：

### 方案 A：推荐

前端统一请求 `/api/...`

Vite 代理改成：

```js
server: {
  proxy: {
    '/api': {
      target: 'http://127.0.0.1:8000',
      changeOrigin: true,
      rewrite: (path) => path.replace(/^\/api/, '/api/v1'),
    },
  },
}
```

这样前端写：

```text
/api/chat/send
```

实际转发到：

```text
/api/v1/chat/send
```

### 方案 B

保留代理不 rewrite，前端直接请求 `/api/v1/...`

但这样前端每个接口都要写 `/api/v1`，不如方案 A 整洁。

本文后续示例全部按方案 A 编写。

## 4. 统一返回格式

### 4.1 普通业务接口

除登录外，其他接口统一返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `code` | number | 业务状态码，`0` 表示成功 |
| `message` | string | 提示信息 |
| `data` | any | 真正业务数据 |

### 4.2 登录接口

登录接口单独返回，不包 `code/message/data`：

```json
{
  "access_token": "xxxxx",
  "token_type": "bearer"
}
```

### 4.3 错误响应

错误时后端也尽量保持统一外壳：

```json
{
  "code": 40431,
  "message": "conversation not found",
  "data": null
}
```

前端统一处理顺序：

1. 先看 HTTP 状态码
2. 再看 `payload.code`
3. 成功时只读 `payload.data`
4. 失败时优先显示 `payload.message`

### 4.4 校验失败

参数校验失败是：

```json
{
  "code": 42200,
  "message": "request validation error",
  "data": [
    {
      "field": "body.content",
      "message": "String should have at least 1 character"
    }
  ]
}
```

前端要把 `data` 当作字段级错误数组处理。

## 5. 认证规则

除注册和登录外，其余接口都要带：

```http
Authorization: Bearer <access_token>
```

Token 失效时，后端一般返回：

```json
{
  "code": 40100,
  "message": "invalid or expired token",
  "data": null
}
```

前端统一动作：

1. 清空本地 token
2. 跳转登录页
3. 提示“登录已过期，请重新登录”

### 6.1 统一请求封装

`frontend/src/api/request.js`

```js
const API_BASE = '/api'

function getToken() {
  return localStorage.getItem('access_token') || ''
}

function buildHeaders(customHeaders = {}, skipAuth = false) {
  const headers = new Headers(customHeaders)
  const token = getToken()

  if (!skipAuth && token && !headers.has('Authorization')) {
    headers.set('Authorization', `Bearer ${token}`)
  }

  return headers
}

function normalizeBody(body, headers) {
  if (body == null) return undefined

  const isForm =
    body instanceof FormData || body instanceof URLSearchParams

  if (!isForm && !headers.has('Content-Type')) {
    headers.set('Content-Type', 'application/json')
  }

  return isForm ? body : JSON.stringify(body)
}

async function parsePayload(response) {
  const text = await response.text()
  if (!text) return null

  try {
    return JSON.parse(text)
  } catch {
    return { code: response.status * 100, message: text, data: null }
  }
}

export async function request(path, options = {}) {
  const {
    method = 'GET',
    body,
    headers: customHeaders,
    skipAuth = false,
    raw = false,
  } = options

  const headers = buildHeaders(customHeaders, skipAuth)
  const finalBody = normalizeBody(body, headers)

  const response = await fetch(`${API_BASE}${path}`, {
    method,
    headers,
    body: finalBody,
  })

  const payload = await parsePayload(response)

  if (!response.ok) {
    const error = new Error(payload?.message || 'request failed')
    error.status = response.status
    error.code = payload?.code
    error.data = payload?.data
    throw error
  }

  if (raw) {
    return payload
  }

  if (payload?.code !== 0) {
    const error = new Error(payload?.message || 'business error')
    error.status = response.status
    error.code = payload?.code
    error.data = payload?.data
    throw error
  }

  return payload.data
}
```

这个封装决定了后面的接口层写法会很干净：

- 登录接口用 `raw: true`
- 普通接口直接拿到 `payload.data`
- 401/422/404 都能统一抛错

```json
{
  "code": 0,
  "message": "register success",
  "data": {
    "id": 1,
    "username": "alice",
    "email": "alice@example.com",
    "phone": "13800138000",
    "is_active": true,
    "created_at": "2026-04-14T10:00:00Z",
    "updated_at": "2026-04-14T10:00:00Z"
  }
}
```

前端接口函数：

```js
import { request } from './request'

export function register(payload) {
  return request('/auth/register', {
    method: 'POST',
    body: payload,
    skipAuth: true,
  })
}
```

### 7.1.2 登录

```http
POST /api/auth/login
Content-Type: application/x-www-form-urlencoded
```

请求体不是 JSON，而是表单：

```text
username=alice
password=12345678
```

前端接口函数：

```js
import { request } from './request'

export function login(username, password) {
  const form = new URLSearchParams()
  form.append('username', username)
  form.append('password', password)

  return request('/auth/login', {
    method: 'POST',
    body: form,
    skipAuth: true,
    raw: true,
  })
}
```

登录成功后前端要做：

1. `localStorage.setItem('access_token', res.access_token)`
2. 后续所有受保护接口自动带 token

只能传下面这些值：


```json
{
  "record_type": "blood_pressure",
```

### 7.4.3 列表查询

```http
GET /api/records?page=1&page_size=10&record_type=blood_pressure
Authorization: Bearer <token>
```

查询参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `page` | number | 否 | 默认 `1` |

| `record_type` | string | 否 | 上面 6 个枚举之一 |
| `start_time` | string | 否 | ISO 8601 |
| `end_time` | string | 否 | ISO 8601 |

```json

  "page": 1,
  "page_size": 10,
  "total_pages": 0
}
```

### 7.4.4 详情 / 更新 / 删除

```http
GET    /api/records/{record_id}
PUT    /api/records/{record_id}
DELETE /api/records/{record_id}
```

前端文件建议：

```js
import { request } from './request'

export function createRecord(payload) {
  return request('/records', {
    method: 'POST',
    body: payload,
  })
}

export function listRecords(params = {}) {
  const search = new URLSearchParams()

  Object.entries(params).forEach(([key, value]) => {
    if (value !== undefined && value !== null && value !== '') {
      search.append(key, value)
    }
  })

  const suffix = search.toString() ? `?${search.toString()}` : ''
  return request(`/records${suffix}`)
}

export function getRecordDetail(recordId) {
  return request(`/records/${recordId}`)
}

export function updateRecord(recordId, payload) {
  return request(`/records/${recordId}`, {
    method: 'PUT',
    body: payload,
  })
}

export function deleteRecord(recordId) {
  return request(`/records/${recordId}`, {
    method: 'DELETE',
  })
}
```

### 7.4.5 Record 页怎么改

当前 `RecordView.vue` 有三个问题：

1. 存的是 `localStorage`，不是后端接口
2. 提交的数据结构和后端 `RecordCreate` 不一致
3. 当前 UI 里的 `systolic/diastolic/symptom` 不是后端字段

前端要这样映射：

#### 服药按钮

“已服药 / 未服药”不要传：

```json
{
  "medication": "已服药"
}
```

而要传：

```json
{
  "record_type": "medication",
  "value": "taken",
  "note": "用户点击已服药"
}
```

或者：

```json
{
  "record_type": "medication",
  "value": "missed",
  "note": "用户点击未服药"
}
```

#### 血压输入框

不要传：

```json
{
  "systolic": "128",
  "diastolic": "82"
}
```

而要拼成一个后端能接收的 `value`：

```json
{
  "record_type": "blood_pressure",
  "value": "128/82",
  "unit": "mmHg"
}
```

#### 身体状态按钮

后端没有 `symptom` 字段，建议前端先映射为：

```json
{
  "record_type": "checkin",
  "value": "头晕",
  "note": "今日身体状态"
}
```

如果以后要细化症状模型，再让后端单独扩字段，不要现在前端自己发明字段。

## 8. 聊天页应该怎么改

这是前端改动重点。

## 8.1 当前旧写法哪里错

旧版 `frontend/src/api/chat.js` 和 `frontend/src/views/ChatView.vue` 存在这些不匹配：

1. 用了 `GET`
2. 路径写成 `/api/chat`
3. 没带 token
4. 没有 `conversation_id`
5. 把返回值当成 `{ reply: '...' }` 直接读取
6. 没有先拉会话列表和历史消息

## 8.2 正确页面初始化流程

聊天页挂载时建议按下面顺序：

1. 检查本地是否有 `access_token`
2. 没有 token 就跳登录页
3. 调 `GET /api/conversations`
4. 如果有会话，选中最新一个会话
5. 如果没有会话，先 `POST /api/conversations` 创建一个
6. 拿到 `conversation_id` 后，调 `GET /api/conversations/{id}/messages`
7. 把消息列表渲染到页面

伪代码：

```js
onMounted(async () => {
  const listData = await listConversations()

  if (listData.items.length > 0) {
    activeConversationId.value = listData.items[0].id
  } else {
    const conversation = await createConversation({ title: '新的健康咨询' })
    activeConversationId.value = conversation.id
  }

  const messageData = await listMessages(activeConversationId.value)
  messages.value = messageData.items
})
```

## 8.3 正确发送流程

点击发送时：

1. 先校验输入非空
2. 本地先插入一条用户消息，做乐观更新也可以
3. 调 `POST /api/chat/send`
4. 成功后把 `data.reply` 追加到消息列表
5. 如需最稳妥，也可以发送成功后重新拉一次 `GET /messages`

伪代码：

```js
async function handleSend() {
  const content = inputText.value.trim()
  if (!content) return
  if (!activeConversationId.value) return

  messages.value.push({
    role: 'user',
    content,
  })

  inputText.value = ''

  try {
    const chatData = await sendChatMessage(activeConversationId.value, content)

    messages.value.push({
      role: 'assistant',
      content: chatData.reply,
    })
  } catch (error) {
    messages.value.push({
      role: 'assistant',
      content: error.message || '发送失败，请稍后再试',
    })
  }
}
```

## 8.4 聊天消息渲染字段也要改

当前页面里用的是：

```js
{ role: 'user', text: '...' }
```

但后端消息对象是：

```js
{
  id,
  conversation_id,
  role,
  content,
  message_type,
  created_at,
}
```

所以建议前端统一改成：

```js
{ role: 'user', content: '...' }
```

模板里不要再渲染 `msg.text`，要渲染 `msg.content`。

## 10. Record 页应该怎么改

如果保留当前页面布局，最小改法如下：

1. “已服药/未服药”点一次就创建一条 `medication` 记录
2. “保存今日记录”时如果填了血压，就再创建一条 `blood_pressure` 记录
3. 如果选了身体状态，就再创建一条 `checkin` 记录
4. 页面下半部分再加一个“最近记录列表”，用 `GET /api/records` 拉取

伪代码：

```js
async function saveRecord() {
  const tasks = []

  if (medication.value) {
    tasks.push(
      createRecord({
        record_type: 'medication',
        value: medication.value === '已服药' ? 'taken' : 'missed',
        note: '来自记录页服药按钮',
      }),
    )
  }

  if (systolic.value && diastolic.value) {
    tasks.push(
      createRecord({
        record_type: 'blood_pressure',
        value: `${systolic.value}/${diastolic.value}`,
        unit: 'mmHg',
      }),
    )
  }

  if (symptom.value) {
    tasks.push(
      createRecord({
        record_type: 'checkin',
        value: symptom.value,
        note: '来自记录页身体状态按钮',
      }),
    )
  }

  await Promise.all(tasks)
}
```

注意：

- 一个页面动作可以拆成多条 Record
- 不要硬凑成一个后端不存在的大对象

## 12. 给前端的最终落地顺序

建议前端按下面顺序改：

1. 先修 `vite.config.js` 代理
2. 新增 `request.js`
3. 先接 `auth.js`
4. 再接 `profile.js`
5. 再接 `record.js`
6. 再接 `conversation.js + message.js + chat.js`
7. 最后改 `ChatView.vue`

原因：

- 如果没有统一请求层，后面每个页面都会各写一套 fetch，后续很难维护
- 如果没有 token 体系，聊天页一定接不通
- 如果没有会话接口，聊天接口也无法调用
