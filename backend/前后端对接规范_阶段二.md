1. 用户注册
2. 用户登录
3. 获取当前用户信息
4. 修改当前用户信息

## 2. 统一规则

### 2.1 成功响应格式

除登录接口外，所有普通业务接口统一返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `code` | number | 业务状态码，`0` 表示成功 |
| `message` | string | 提示信息 |
| `data` | object/null | 具体业务数据 |

### 2.2 错误响应格式

错误时也尽量保持同一外壳：

```json
{
  "code": 40400,
  "message": "profile not found",
  "data": null
}
```

前端处理原则：

1. 先判断 HTTP 状态码
2. 再判断 `code`
3. 成功时读取 `data`
4. 失败时优先展示 `message`

### 2.3 登录接口是例外

登录接口为了兼容 OAuth2 Bearer 规范，不包 `code/message/data`，直接返回：

```json
{
  "access_token": "xxxxx",
  "token_type": "bearer"
}
```

### 2.4 时间格式

所有时间字段统一使用 ISO 8601 字符串，推荐 UTC，例如：

```text
2026-04-08T10:30:00Z
```

### 2.5 认证方式

除注册、登录外，其余接口都需要登录态。

请求头格式：

```http
Authorization: Bearer <access_token>
```

### 2.6 内容类型

1. 注册、更新、创建 Profile：`application/json`
2. 登录：`application/x-www-form-urlencoded`

## 3. 数据结构定义

### 3.1 User 对象

后端返回给前端的用户信息结构：

```json
{
  "id": 1,
  "username": "alice",
  "email": "alice@example.com",
  "phone": "13800138000",
  "is_active": true,
  "created_at": "2026-04-08T10:30:00Z",
  "updated_at": "2026-04-08T10:30:00Z"
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | number | 用户主键 |
| `username` | string | 用户名 |
| `email` | string/null | 邮箱 |
| `phone` | string/null | 手机号 |
| `is_active` | boolean | 是否启用 |
| `created_at` | string | 创建时间 |
| `updated_at` | string | 更新时间 |

注意：

- 前端永远收不到 `password_hash`
- 前端也不需要知道数据库里的密码字段实现

### 3.2 Profile 对象

后端返回给前端的 Profile 结构：

```json
{
  "id": 1,
  "user_id": 1,
  "name": "Alice",
  "age": 68,
  "gender": "female",
  "height": 160,
  "weight": 56,
  "chronic_history": "hypertension",
  "allergy_history": "penicillin",
  "emergency_contact": "Bob 13800138001",
  "remark": "need low-salt diet",
  "created_at": "2026-04-08T10:40:00Z",
  "updated_at": "2026-04-08T10:40:00Z"
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | number | Profile 主键 |
| `user_id` | number | 所属用户 ID |
| `name` | string | 姓名 |
| `age` | number/null | 年龄 |
| `gender` | string/null | 性别 |
| `height` | number/null | 身高，单位 cm |
| `weight` | number/null | 体重，单位 kg |
| `chronic_history` | string/null | 慢病史 |
| `allergy_history` | string/null | 过敏史 |
| `emergency_contact` | string/null | 紧急联系人 |
| `remark` | string/null | 备注 |
| `created_at` | string | 创建时间 |
| `updated_at` | string | 更新时间 |

注意：

- 前端不需要主动传 `user_id`
- `user_id` 由后端根据当前登录用户自动确定

## 5. 认证模块

## 5.1 用户注册

### 接口信息

```http
POST /api/v1/auth/register
Content-Type: application/json
```

### 前端需要提交的数据

```json
{
  "username": "alice",
  "email": "alice@example.com",
  "phone": "13800138000",
  "password": "12345678",
  "confirm_password": "12345678"
}
```

字段要求：

| 字段 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- |
| `username` | 是 | string | 3 到 50 个字符，唯一 |
| `email` | 否 | string/null | 邮箱，可为空 |
| `phone` | 否 | string/null | 手机号，可为空 |
| `password` | 是 | string | 8 到 32 位 |
| `confirm_password` | 是 | string | 必须和 `password` 一致 |

### 后端成功返回

```json
{
  "code": 0,
  "message": "register success",
  "data": {
    "id": 1,
    "username": "alice",
    "email": "alice@example.com",
    "phone": "13800138000",
    "is_active": true,
    "created_at": "2026-04-08T10:30:00Z",
    "updated_at": "2026-04-08T10:30:00Z"
  }
}
```

### 常见失败场景

1. 用户名已存在
2. 邮箱已存在
3. 手机号已存在
4. 两次密码不一致
5. 参数格式不合法

前端建议处理：

1. 表单提交前做基础校验
2. 后端返回错误时直接展示 `message`

---

## 5.2 用户登录

### 接口信息

```http
POST /api/v1/auth/login
Content-Type: application/x-www-form-urlencoded
```

### 前端需要提交的数据

注意：不是 JSON，是表单。

```text
username=alice
password=12345678
```

字段说明：

| 字段 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- |
| `username` | 是 | string | 用户名 |
| `password` | 是 | string | 密码 |

### 后端成功返回

```json
{
  "access_token": "xxxxx",
  "token_type": "bearer"
}
```

### 前端拿到后要做什么

1. 保存 `access_token`
2. 后续请求统一带上：

```http
Authorization: Bearer xxxxx
```

### 常见失败场景

1. 用户不存在
2. 密码错误
3. 用户被禁用

## 6. 用户模块

### 接口信息

```http
GET /api/v1/users/me
```

无请求体。

只需要带 Token。

### 后端成功返回

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 1,
    "username": "alice",
    "email": "alice@example.com",
    "phone": "13800138000",
    "is_active": true,
    "created_at": "2026-04-08T10:30:00Z",
    "updated_at": "2026-04-08T10:30:00Z"
  }
}
```

### 前端用途

常见用于：

1. 页面初始化获取当前登录用户
2. 顶部导航栏展示用户名
3. 判断当前登录态是否有效

## 6.2 修改当前用户信息

### 接口信息

```http
PUT /api/v1/users/me
Authorization: Bearer <access_token>
Content-Type: application/json
```

### 前端可提交的数据

可以部分更新，不需要全部传。

```json
{
  "username": "alice_new",
  "email": "alice_new@example.com",
  "phone": "13800138011"
}
```

字段说明：

| 字段 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- |
| `username` | 否 | string/null | 新用户名 |
| `email` | 否 | string/null | 新邮箱 |
| `phone` | 否 | string/null | 新手机号 |

注意：

- 不支持在这个接口改密码
- 不支持前端传 `id`
- 不支持前端改 `is_active`

### 后端成功返回

```json
{
  "code": 0,
  "message": "update success",
  "data": {
    "id": 1,
    "username": "alice_new",
    "email": "alice_new@example.com",
    "phone": "13800138011",
    "is_active": true,
    "created_at": "2026-04-08T10:30:00Z",
    "updated_at": "2026-04-08T11:00:00Z"
  }
}
```

### 常见失败场景

1. 新用户名已存在
2. 新邮箱已存在
3. 新手机号已存在
4. Token 无效

## 8. 常见错误码建议

以下是前端需要重点识别的业务错误码。

| code | 含义 | 前端建议 |
| --- | --- | --- |
| `0` | 成功 | 正常处理 |
| `40001` | 用户名已存在 | 提示更换用户名 |
| `40002` | 邮箱已存在 | 提示更换邮箱 |
| `40003` | 手机号已存在 | 提示更换手机号 |
| `40100` | Token 无效或已过期 | 清空登录态并跳转登录页 |
| `40101` | 用户名或密码错误 | 登录页提示错误 |
| `40300` | 用户不可用 | 提示联系管理员 |
| `40411` | Profile 不存在 | 引导创建 profile |
| `42200` | 请求参数校验失败 | 逐项提示表单错误 |
| `50000` | 服务器内部错误 | 统一 toast 提示 |

## 9. 前端联调顺序建议

前端建议按下面顺序接入：

1. 先做注册页
2. 再做登录页
3. 登录成功后保存 `access_token`
4. 调 `GET /users/me` 验证登录态
5. 进入个人中心页
6. 调 `GET /profiles/me`
7. 如果没有 profile，则引导去创建
8. 创建成功后进入 profile 展示页
9. 后续用 `PUT /profiles/me` 做编辑

## 10. 前端必须遵守的约定

### 10.1 不要自己拼用户 ID

以下接口都不需要前端传 `user_id`：

1. `/users/me`
2. `/profiles/me`

原因：

- 当前用户由 Token 决定
- 不能由前端自己指定“我要操作谁的数据”

### 10.2 统一从 `data` 里取业务数据

对于普通接口：

```json
{
  "code": 0,
  "message": "success",
  "data": { ... }
}
```

前端应统一从 `data` 取真正业务数据。

### 10.3 登录接口单独处理

登录接口直接取：

```json
{
  "access_token": "...",
  "token_type": "bearer"
}
```

不要按普通 `data` 包装去取。

### 10.4 Token 过期统一处理

如果接口返回：

- HTTP 401
- 或业务码 `40100`

前端应：

1. 清除本地 Token
2. 跳转登录页
3. 提示“登录已过期，请重新登录”

---

## 11. 给前端的简化总结

前端只需要记住这几件事：

1. 注册发 JSON
2. 登录发表单
3. 登录成功后保存 `access_token`
4. 其余受保护接口统一加 `Authorization: Bearer <token>`
5. 普通接口统一读 `code/message/data`
6. `/users/me` 获取当前用户
7. `/profiles/me` 获取或编辑当前用户档案
